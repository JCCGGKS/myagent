# 客服 Agent MVP 详细设计

## 1. 文档目标

本文档用于细化客服 Agent MVP 的实现设计，重点覆盖：

+ 多轮对话意图识别
+ 上下文管理
+ 记忆管理
+ 规划能力与设计模式
+ 工具调用机制

---

## 2. 设计总原则

客服 Agent 的核心链路应设计为：

`输入理解 -> 状态更新 -> 策略判断 -> 检索/工具执行 -> 回复生成 -> 记忆沉淀`

设计原则如下：

+ 多轮对话优先依赖结构化状态
+ 高风险场景默认收敛到人工
+ 检索和工具调用必须受控
+ 记忆先轻后重
+ 规划采用受控流程图

## 5. 记忆管理设计

### 5.1 设计目标

记忆管理用于解决两个问题：

+ 当前会话中，系统需要持续知道用户正在处理什么
+ 跨会话时，系统需要保留少量高价值信息

### 5.2 记忆分类

建议分三类：

+ 短期记忆
+ 长期用户记忆
+ 经验记忆

### 5.3 短期记忆

范围：

+ 当前会话对话历史
+ 当前状态对象
+ 最近工具调用结果
+ 当前检索结果摘要

存储建议：

+ `Redis`：缓存活跃会话短期状态
+ `PostgreSQL`：持久化会话日志和状态快照

作用：

+ 维持任务连贯性
+ 支撑多轮追问和工具调用
+
### 5.4 长期用户记忆

范围：

+ 用户偏好的沟通方式
+ 已验证过的基础信息
+ 具有跨会话价值的固定偏好

设计原则：

+ 只保留少量稳定字段
+ 不做复杂用户画像
+ 敏感信息必须最小化存储并受权限控制

MVP 阶段建议：

+ 可选，不作为第一优先级
+ 若做，先只保留非常有限的业务必要信息

### 5.5 经验记忆

范围：

+ 常见失败案例
+ 高价值追问模板
+ 常见误判样本
+ 知识库缺口

MVP 阶段建议：

+ 先不做自动学习
+ 先通过日志回放和评测结果，由人工整理沉淀为知识库、规则或测试集

### 5.6 记忆写入策略

不是所有信息都应写入记忆。建议采用以下原则：

+ 当前任务临时信息写短期记忆
+ 跨会话可复用信息才考虑写长期记忆
+ 失败样本先写日志，人工审核后再沉淀为经验资产

## 6. 规划能力与设计模式

### 6.1 设计目标

客服 Agent 的规划能力用于：

+ 根据当前意图决定进入哪个处理流程
+ 根据当前状态决定下一步应该追问、检索、查单还是转人工

### 6.2 推荐设计模式

MVP 阶段推荐采用 `Router + State Machine + Tool Executor`：

+ `Router`：识别用户当前意图并决定进入哪条流程
+ `State Machine`：根据状态决定当前阶段和下一步动作
+ `Tool Executor`：负责执行检索和业务调用

该模式本质上是一个 `workflow-first` 的受控 `agentic workflow`，适合客服场景中“用户输入开放、系统动作受控”的要求。

### 6.3 不推荐的模式

MVP 阶段不建议直接采用：

+ 重度 `Plan-and-Execute`
+ 复杂 `Multi-Agent`
+ 完全开放式 `ReAct`

原因：客服流程边界明确，MVP 阶段更强调稳定、可控、可审计。

### 6.4 适合客服的流程图设计

建议按业务流拆成子图：

```text
Router
  -> FAQ Flow
  -> Order Flow
  -> Logistics Flow
  -> Refund Flow
  -> Handoff Flow
```

各子图内部再细分：

+ `collecting_info`
+ `clarifying`
+ `retrieving`
+ `executing`
+ `confirming`
+ `done`
+
### 6.5 ReAct 的使用边界

`ReAct` 仅建议在低风险、需要有限证据推理的节点受控使用：

+ FAQ / 规则检索后的回答生成节点
+ 售后规则判断节点
+ 澄清问题生成节点

不建议用于：

+ 顶层 Router
+ 状态更新节点
+ 高风险工具执行节点
+ 转人工触发节点
+ 权限判断节点

### 6.6 规划层决策输出

建议规划层统一输出动作类型：

+ `answer_directly`
+ `retrieve_knowledge`
+ `query_business_tool`
+ `ask_clarification`
+ `handoff_human`
+
这样后续执行器可以稳定处理，而不是直接依赖自然语言描述。

## 7. 工具调用设计

### 7.1 设计目标

工具调用用于在受控条件下稳定完成查询和操作。

### 7.2 工具分类

建议按风险和用途分为四类：

+ 检索类工具
+ 查询类工具
+ 流程类工具
+ 高风险操作类工具

#### 检索类工具

+ FAQ 检索
+ 售后规则检索
+ 活动规则检索

#### 查询类工具

+ 查询订单状态
+ 查询物流状态
+ 查询售后进度
+
#### 流程类工具

+ 创建转人工请求
+ 生成会话摘要
+
#### 高风险操作类工具

+ 自动退款
+ 修改地址
+ 修改手机号
+ 变更支付信息

MVP 阶段建议：

+ 只开放检索类、查询类和低风险流程类工具
+ 高风险操作类默认不自动执行

### 7.3 工具定义原则

每个工具都应满足：

+ 单一职责
+ 输入结构化
+ 输出结构化
+ 参数可校验
+ 可记录、可回放、可审计

### 7.4 工具参数设计

建议使用 `Pydantic` 定义工具输入输出：

```python
from pydantic import BaseModel


class QueryOrderArgs(BaseModel):
    order_id: str
    user_id: str


class QueryOrderResult(BaseModel):
    order_id: str
    status: str
    delivery_status: str | None = None
```

这样可以保证：

+ 参数不全时不执行
+ 参数类型错误时立即暴露
+ 工具输出便于统一处理

### 7.5 Slot Schema 与 Tool Schema 的职责边界

本项目中两者分工明确：

+ `slot schema`：负责任务推进，决定当前主链路需要抽取哪些关键参数
+ `tool schema`：负责执行前校验，确保工具调用参数完整且类型正确

因此不采用“先判断调用哪个工具，再根据工具参数反推主 slot”的主方案。主流程采用：

`intent -> slot schema -> state update -> policy decision -> tool call(if needed)`

这样可以减少一次额外的 LLM 推理，并避免在工具调用前引入不必要的工具解析复杂度。

### 7.6 工具调用策略

建议采用以下策略：

+ 必要参数不齐全时不调用
+ 高风险工具必须先确认
+ 外部接口失败时做统一降级
+ 工具原始结果不直接回给用户，由 Agent 转译成客服回复

### 7.7 工具调用结果处理与上下文隔离

工具调用结果建议拆成三层：

+ `raw_result`：原始结构化响应，用于日志、调试和审计
+ `sanitized_result`：清洗后的结构化结果，用于状态更新和后续推理
+ `user_facing_summary`：给用户展示的自然语言摘要

### 7.8 状态对象驱动的上下文隔离

上下文隔离在工程上主要通过状态对象实现：

+ 原始工具结果保留在日志、数据库或专用结果层
+ 状态对象只保留当前任务推进所需的最小必要信息
+ 主推理链路优先读取状态对象，而不是直接读取原始工具响应

示例：

查询物流接口可能返回：

```json
{
  "order_id": "12345",
  "user_id": "u789",
  "delivery_status": "IN_TRANSIT",
  "warehouse_code": "WH-02",
  "internal_note": "high-risk customer",
  "carrier_tracking_no": "SF1234567890"
}
```

但写入状态对象的内容应收敛为：

```json
{
  "current_intent": "query_logistics",
  "slots": {
    "order_id": "12345"
  },
  "logistics_status": "in_transit",
  "latest_logistics_summary": "包裹运输中，预计明日送达"
}
```

只有进入状态对象或被明确选入上下文的数据，才会暴露给主推理链路。

### 7.9 推荐处理流程

建议每次工具调用后按以下流程处理：

1. 工具返回 `raw_result`
2. 结果清洗节点生成 `sanitized_result`
3. 状态更新节点只提取必要字段写入 `state`
4. 回复生成节点基于 `sanitized_result` 或 `state` 生成用户回复
5. `raw_result` 仅进入日志和审计，不直接进入长期上下文窗口

## 8. 统一执行链路设计

建议整个客服 Agent 按以下统一链路运行：

1. `Input Normalizer`
标准化输入消息、渠道信息、用户信息

2. `Intent Router`
识别意图、槽位、置信度、风险等级、是否切话题

3. `State Tracker`
更新当前任务状态

4. `Policy Layer`
决定下一步是直接回答、检索、调用工具、澄清还是转人工

5. `Executor`
执行 FAQ 检索、业务查询或转人工请求

6. `Response Generator`
生成客服风格回复

7. `Memory Writer`
将会话日志、状态快照、失败样本写入存储

## 9. MVP 阶段最重要的设计取舍

第一阶段最容易做过头的地方有三个：

+ 过度规划：把客服 Agent 做成开放式自治体
+ 过度记忆：一开始就构建复杂用户画像和自动学习系统
+ 过度并发：把多意图做成并行执行引擎

MVP 阶段应该优先保障：

+ 意图识别稳定
+ 状态对象清晰
+ 工具调用可控
+ 转人工顺畅
+ 检索链路可靠

---

## 10. 最终建议

客服 Agent MVP 的核心设计建议如下：

+ 意图识别采用“分层路由 + 显式状态跟踪”
+ 上下文管理采用“滑动窗口 + 状态摘要 + 任务相关知识”
+ 记忆管理采用“短期优先、长期从简、经验人工沉淀”
+ 规划能力采用“Router + State Machine + Tool Executor”
+ 工具调用采用“结构化参数 + 风险分级 + 审计可回放”

该设计适合 LangGraph 实现，并便于评测、审计和后续演进。
