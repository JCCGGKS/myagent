# 客服 Agent MVP 详细设计

## 1. 文档目标

本文档用于细化客服 Agent MVP 的实现设计，重点覆盖：

+ 多轮对话意图识别
+ 上下文管理
+ 记忆管理
+ 规划能力与设计模式
+ 工具调用机制

---

## 2. 设计总原则

客服 Agent 的核心链路应设计为：

`输入理解 -> 状态更新 -> 策略判断 -> 检索/工具执行 -> 回复生成 -> 记忆沉淀`

设计原则如下：

+ 多轮对话优先依赖结构化状态
+ 高风险场景默认收敛到人工
+ 检索和工具调用必须受控
+ 记忆先轻后重
+ 规划采用受控流程图

---

## 3. 多轮对话意图识别设计

### 3.1 设计目标

意图识别是客服 Agent 的路由入口，职责包括：

+ 判断用户当前在做什么
+ 提取推进任务所需的关键参数
+ 判断是否需要追问
+ 判断是否发生话题切换
+ 判断是否属于高风险场景

### 3.2 输出结构

建议统一输出如下结构：

```json
{
  "intent": "query_logistics",
  "slots": {
    "order_id": "12345"
  },
  "confidence": 0.91,
  "needs_clarification": false,
  "risk_level": "low",
  "is_intent_shift": false,
  "candidate_intents": [
    "query_logistics",
    "refund_consult"
  ]
}
```

字段说明：

+ `intent`：当前主意图
+ `slots`：当前轮提取出的关键参数
+ `confidence`：模型判断置信度
+ `needs_clarification`：是否必须继续澄清
+ `risk_level`：`low / medium / high`
+ `is_intent_shift`：是否从前一任务切换到新任务
+ `candidate_intents`：候选意图，便于后续分析和兜底

### 3.3 意图体系设计

MVP 阶段建议使用可执行动作命名，而不是业务概念命名：

+ `faq`
+ `query_order`
+ `query_logistics`
+ `refund_consult`
+ `return_consult`
+ `exchange_consult`
+ `invoice_consult`
+ `coupon_consult`
+ `handoff_human`
+ `unsupported`

设计原则：

+ 粒度不要过细
+ 一个意图应对应一条清晰的处理链路
+ 对无法判断的请求必须保留 `unsupported` 或 `unknown`

### 3.4 多轮识别策略与分层路由

客服场景推荐采用“显式状态跟踪 + 分层路由”：

推荐三层路由：

1. 规则层  
处理高频、刚性、可强匹配的意图，如“转人工”“查订单”“查物流”“投诉”“敏感禁用操作”

2. 轻量识别层  
处理 FAQ、固定问题、口语化但不复杂的表达，以及单轮可判断的常见业务意图

3. LLM 路由层  
处理多轮上下文依赖、省略、改口、歧义、多意图、风险判断和是否追问

推荐执行顺序：

`规则层 -> 轻量识别层 -> LLM 路由层`

### 3.5 轻量识别层设计

轻量识别层用于承接大部分单轮常见请求。

适合处理：

+ FAQ 类问题
+ 单轮可判断的查询意图
+ 常见口语表达和同义表达
+ 标准售后咨询入口

不适合处理：

+ 强上下文依赖
+ 用户改口
+ 多意图并存
+ 高风险判断
+ 复杂流程分支

建议输入：

+ 当前用户消息
+ 最近 `1~2` 轮简短上下文
+ 可选的 `current_intent`

建议输出：

```json
{
  "intent": "query_logistics",
  "confidence": 0.82,
  "slots": {},
  "needs_clarification": true,
  "route_source": "light_router"
}
```

### 3.6 轻量识别层实现方案

MVP 阶段建议采用 `embedding 相似度匹配`。

基本流程：

1. 对输入文本做标准化
2. 生成 query embedding
3. 在 `intent_examples` 中做相似度召回
4. 取 top-k 候选并汇总到 intent 级别
5. 结合阈值判断是否接受结果，否则升级到 LLM 路由层

建议至少关注三个分数：

+ top-1 分数
+ top-2 分数
+ top-1 与 top-2 的差值

推荐阈值思路：

+ 高置信且明显胜出：直接路由
+ 中等置信：追问或升级 LLM
+ 低置信：直接升级 LLM

### 3.7 轻量识别层样本与存储

轻量识别层存储“意图示例表达”而非单个 intent 名称。

例如：

+ `query_logistics -> 快递到哪了`
+ `query_logistics -> 物流什么时候到`
+ `query_order -> 我的订单发货了吗`

MVP 阶段建议存储在 `PostgreSQL + pgvector`。

建议按逻辑分表管理：

+ `intent_definitions`
+ `intent_examples`
+ `faq_examples`

其中：

+ `intent_definitions`：存意图名称、风险等级、必填槽位、说明
+ `intent_examples`：存示例表达和 embedding
+ `faq_examples`：存 FAQ 示例和 embedding

Redis 可用于缓存热门样本，但不建议作为 embedding 样本主存储。

### 3.8 多意图与意图切换处理

对一句话包含多个诉求的情况，不建议 MVP 阶段并行执行多个流程，建议：

+ 输出 `主意图 + 候选/子任务`
+ 优先处理高风险、高优先级任务
+ 如有必要，引导用户分开处理

对意图切换的判断建议基于：

+ 当前轮内容与 `current_intent` 的语义距离
+ 是否出现了新的动作目标
+ 是否能从旧任务直接继承参数

### 3.9 失败处理策略

+ `confidence` 低于阈值时不直接执行
+ 缺少必要槽位时优先追问
+ 连续多轮无法判断时转人工

## 4. 上下文管理设计

### 4.1 设计目标

上下文管理的目标是向模型提供“当前决策所需的最小充分信息”。

### 4.2 上下文组成

建议将上下文拆成四层：

+ 最近几轮原始对话
+ 当前结构化状态
+ 当前任务相关知识片段
+ 必要的长期记忆

### 4.3 推荐上下文输入结构

建议提供给 LLM 的上下文包括：

+ 最近 3 到 6 轮用户和系统对话
+ 当前会话状态对象
+ 当前检索结果或工具结果
+ 必要的业务规则
+
不建议直接提供：

+ 全量长对话历史
+ 无筛选的原始工具结果
+ 与当前任务无关的历史记忆

### 4.4 会话状态对象

建议维护如下状态对象：

```json
{
  "current_intent": "refund_consult",
  "stage": "collecting_info",
  "slots": {
    "order_id": "12345"
  },
  "missing_slots": [
    "refund_reason"
  ],
  "confirmed_slots": [
    "order_id"
  ],
  "topic_changed": false,
  "summary": "用户正在咨询订单12345的退款流程"
}
```

字段说明：

+ `current_intent`：当前主任务
+ `stage`：当前任务阶段
+ `slots`：当前已知参数
+ `missing_slots`：继续推进任务所需但尚缺失的信息
+ `confirmed_slots`：已确认无歧义的信息
+ `topic_changed`：是否发生话题切换
+ `summary`：对当前任务的简要摘要

### 4.5 上下文控制策略

建议结合以下机制：

+ 滑动窗口：保留最近若干轮原始对话
+ 摘要压缩：对较长历史生成任务摘要
+ 状态显式化：优先读状态对象，而不是让模型自己总结
+ 上下文隔离：工具原始响应与用户可见内容分离

### 4.6 摘要窗口策略

客服 Agent 建议采用“摘要窗口”管理长对话上下文。

摘要窗口的基本形式为：

+ `running_summary`：对较早历史对话的增量摘要
+ `recent_messages`：最近若干轮保留原文
+ `state`：当前任务的结构化状态对象

基本形式：

+ `running_summary`
+ `recent_messages`
+ `state`

### 4.7 何时触发摘要压缩

建议采用“硬阈值 + 软阈值”组合触发：

#### 软触发条件

+ 历史上下文 token 占可用预算约 `60%`
+ 历史轮次超过 `8` 轮
+ 当前任务状态较稳定
+ 早期消息中已经出现大量重复确认、寒暄或低价值内容

#### 硬触发条件

+ 历史上下文 token 占可用预算约 `80%`
+ 历史轮次超过 `12` 轮
+
#### 暂缓压缩条件

+ 刚发生 `intent shift`
+ 最近刚补齐关键槽位
+ 最近刚发生高风险确认
+ 最近刚返回关键工具结果，且尚未完成状态更新

### 4.8 摘要窗口的推荐默认值

MVP 阶段建议默认如下：

+ 最近原文窗口保留：`6` 轮
+ 软触发轮数：`8` 轮
+ 硬触发轮数：`12` 轮
+ 历史上下文软触发：可用上下文预算的 `60%`
+ 历史上下文硬触发：可用上下文预算的 `80%`

如果最近发生了以下事件，可临时扩大原文窗口到 `8` 轮：

+ 意图切换
+ 关键槽位改写
+ 高风险确认
+ 关键工具调用结果刚返回

### 4.9 摘要更新方式

建议采用“增量摘要”，而不是每次重写全量历史摘要。

推荐方式：

1. 永远保留最近 `N` 轮原始消息
2. 将“即将移出窗口的旧消息”合并进 `running_summary`
3. 新摘要替换旧摘要
4. 后续上下文使用：`running_summary + recent_messages + state`

不建议：

+ 每轮都重新总结全部历史
+ 每次固定删除一半消息
+ 摘要后只保留 1 到 2 轮原文

### 4.10 摘要内容要求

摘要应保留当前任务推进所需的关键事实：

+ 历史主意图和当前主意图
+ 已确认的关键槽位
+ 用户明确表达的诉求
+ 工具调用得到的重要结果
+ 已给出的规则说明或承诺
+ 当前仍待补充的信息
+ 转人工原因或风险标记

推荐摘要示例：

+ “用户先查询订单12345物流，后切换为退款咨询；订单号已确认，退款原因为尺码不合适；已告知退货规则，当前仍缺少图片凭证”

## 5. 记忆管理设计

### 5.1 设计目标

记忆管理用于解决两个问题：

+ 当前会话中，系统需要持续知道用户正在处理什么
+ 跨会话时，系统需要保留少量高价值信息

### 5.2 记忆分类

建议分三类：

+ 短期记忆
+ 长期用户记忆
+ 经验记忆

### 5.3 短期记忆

范围：

+ 当前会话对话历史
+ 当前状态对象
+ 最近工具调用结果
+ 当前检索结果摘要

存储建议：

+ `Redis`：缓存活跃会话短期状态
+ `PostgreSQL`：持久化会话日志和状态快照

作用：

+ 维持任务连贯性
+ 支撑多轮追问和工具调用
+
### 5.4 长期用户记忆

范围：

+ 用户偏好的沟通方式
+ 已验证过的基础信息
+ 具有跨会话价值的固定偏好

设计原则：

+ 只保留少量稳定字段
+ 不做复杂用户画像
+ 敏感信息必须最小化存储并受权限控制

MVP 阶段建议：

+ 可选，不作为第一优先级
+ 若做，先只保留非常有限的业务必要信息

### 5.5 经验记忆

范围：

+ 常见失败案例
+ 高价值追问模板
+ 常见误判样本
+ 知识库缺口

MVP 阶段建议：

+ 先不做自动学习
+ 先通过日志回放和评测结果，由人工整理沉淀为知识库、规则或测试集

### 5.6 记忆写入策略

不是所有信息都应写入记忆。建议采用以下原则：

+ 当前任务临时信息写短期记忆
+ 跨会话可复用信息才考虑写长期记忆
+ 失败样本先写日志，人工审核后再沉淀为经验资产

## 6. 规划能力与设计模式

### 6.1 设计目标

客服 Agent 的规划能力用于：

+ 根据当前意图决定进入哪个处理流程
+ 根据当前状态决定下一步应该追问、检索、查单还是转人工

### 6.2 推荐设计模式

MVP 阶段推荐采用 `Router + State Machine + Tool Executor`：

+ `Router`：识别用户当前意图并决定进入哪条流程
+ `State Machine`：根据状态决定当前阶段和下一步动作
+ `Tool Executor`：负责执行检索和业务调用

该模式本质上是一个 `workflow-first` 的受控 `agentic workflow`，适合客服场景中“用户输入开放、系统动作受控”的要求。

### 6.3 不推荐的模式

MVP 阶段不建议直接采用：

+ 重度 `Plan-and-Execute`
+ 复杂 `Multi-Agent`
+ 完全开放式 `ReAct`

原因：客服流程边界明确，MVP 阶段更强调稳定、可控、可审计。

### 6.4 适合客服的流程图设计

建议按业务流拆成子图：

```text
Router
  -> FAQ Flow
  -> Order Flow
  -> Logistics Flow
  -> Refund Flow
  -> Handoff Flow
```

各子图内部再细分：

+ `collecting_info`
+ `clarifying`
+ `retrieving`
+ `executing`
+ `confirming`
+ `done`
+
### 6.5 ReAct 的使用边界

`ReAct` 仅建议在低风险、需要有限证据推理的节点受控使用：

+ FAQ / 规则检索后的回答生成节点
+ 售后规则判断节点
+ 澄清问题生成节点

不建议用于：

+ 顶层 Router
+ 状态更新节点
+ 高风险工具执行节点
+ 转人工触发节点
+ 权限判断节点

### 6.6 规划层决策输出

建议规划层统一输出动作类型：

+ `answer_directly`
+ `retrieve_knowledge`
+ `query_business_tool`
+ `ask_clarification`
+ `handoff_human`
+
这样后续执行器可以稳定处理，而不是直接依赖自然语言描述。

## 7. 工具调用设计

### 7.1 设计目标

工具调用用于在受控条件下稳定完成查询和操作。

### 7.2 工具分类

建议按风险和用途分为四类：

+ 检索类工具
+ 查询类工具
+ 流程类工具
+ 高风险操作类工具

#### 检索类工具

+ FAQ 检索
+ 售后规则检索
+ 活动规则检索

#### 查询类工具

+ 查询订单状态
+ 查询物流状态
+ 查询售后进度
+
#### 流程类工具

+ 创建转人工请求
+ 生成会话摘要
+
#### 高风险操作类工具

+ 自动退款
+ 修改地址
+ 修改手机号
+ 变更支付信息

MVP 阶段建议：

+ 只开放检索类、查询类和低风险流程类工具
+ 高风险操作类默认不自动执行

### 7.3 工具定义原则

每个工具都应满足：

+ 单一职责
+ 输入结构化
+ 输出结构化
+ 参数可校验
+ 可记录、可回放、可审计

### 7.4 工具参数设计

建议使用 `Pydantic` 定义工具输入输出：

```python
from pydantic import BaseModel


class QueryOrderArgs(BaseModel):
    order_id: str
    user_id: str


class QueryOrderResult(BaseModel):
    order_id: str
    status: str
    delivery_status: str | None = None
```

这样可以保证：

+ 参数不全时不执行
+ 参数类型错误时立即暴露
+ 工具输出便于统一处理

### 7.5 工具调用策略

建议采用以下策略：

+ 必要参数不齐全时不调用
+ 高风险工具必须先确认
+ 外部接口失败时做统一降级
+ 工具原始结果不直接回给用户，由 Agent 转译成客服回复

### 7.6 工具调用结果处理与上下文隔离

工具调用结果建议拆成三层：

+ `raw_result`：原始结构化响应，用于日志、调试和审计
+ `sanitized_result`：清洗后的结构化结果，用于状态更新和后续推理
+ `user_facing_summary`：给用户展示的自然语言摘要

### 7.7 状态对象驱动的上下文隔离

上下文隔离在工程上主要通过状态对象实现：

+ 原始工具结果保留在日志、数据库或专用结果层
+ 状态对象只保留当前任务推进所需的最小必要信息
+ 主推理链路优先读取状态对象，而不是直接读取原始工具响应

示例：

查询物流接口可能返回：

```json
{
  "order_id": "12345",
  "user_id": "u789",
  "delivery_status": "IN_TRANSIT",
  "warehouse_code": "WH-02",
  "internal_note": "high-risk customer",
  "carrier_tracking_no": "SF1234567890"
}
```

但写入状态对象的内容应收敛为：

```json
{
  "current_intent": "query_logistics",
  "slots": {
    "order_id": "12345"
  },
  "logistics_status": "in_transit",
  "latest_logistics_summary": "包裹运输中，预计明日送达"
}
```

只有进入状态对象或被明确选入上下文的数据，才会暴露给主推理链路。

### 7.8 推荐处理流程

建议每次工具调用后按以下流程处理：

1. 工具返回 `raw_result`
2. 结果清洗节点生成 `sanitized_result`
3. 状态更新节点只提取必要字段写入 `state`
4. 回复生成节点基于 `sanitized_result` 或 `state` 生成用户回复
5. `raw_result` 仅进入日志和审计，不直接进入长期上下文窗口

## 8. 统一执行链路设计

建议整个客服 Agent 按以下统一链路运行：

1. `Input Normalizer`
标准化输入消息、渠道信息、用户信息

2. `Intent Router`
识别意图、槽位、置信度、风险等级、是否切话题

3. `State Tracker`
更新当前任务状态

4. `Policy Layer`
决定下一步是直接回答、检索、调用工具、澄清还是转人工

5. `Executor`
执行 FAQ 检索、业务查询或转人工请求

6. `Response Generator`
生成客服风格回复

7. `Memory Writer`
将会话日志、状态快照、失败样本写入存储

## 9. MVP 阶段最重要的设计取舍

第一阶段最容易做过头的地方有三个：

+ 过度规划：把客服 Agent 做成开放式自治体
+ 过度记忆：一开始就构建复杂用户画像和自动学习系统
+ 过度并发：把多意图做成并行执行引擎

MVP 阶段应该优先保障：

+ 意图识别稳定
+ 状态对象清晰
+ 工具调用可控
+ 转人工顺畅
+ 检索链路可靠

---

## 10. 最终建议

客服 Agent MVP 的核心设计建议如下：

+ 意图识别采用“分层路由 + 显式状态跟踪”
+ 上下文管理采用“滑动窗口 + 状态摘要 + 任务相关知识”
+ 记忆管理采用“短期优先、长期从简、经验人工沉淀”
+ 规划能力采用“Router + State Machine + Tool Executor”
+ 工具调用采用“结构化参数 + 风险分级 + 审计可回放”

该设计适合 LangGraph 实现，并便于评测、审计和后续演进。
