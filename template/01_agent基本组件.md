## 记忆--短期/长期

### 短期记忆
记录当前的对话历史、中间步骤的思考过程、工具调用的结果，用于维持任务的连贯性
+ llm的上下文窗口：最直接的短期记忆载体
+ 缓冲区：agent框架中通常使用不同类型的缓冲区管理对话历史
    + ConversationBufferMemory
    + ConversationBufferWindowMemory
    + ConversationSummaryBufferMemory
+ 暂存器(scratchpad)：用于记录ReAct框架中的"Thought-Action-Observation"轨迹，是Agent进行逐步推理的关键

### 长期记忆
存储跨越任务和时间维度的信息，比如过去的经验、知识、用户偏好等，使得agent不断学习和成长
+ 情景记忆(Episodic)
+ 程序记忆(Procedural)
+ 语义记忆(Semantic)

# 自主规划--Agent设计模式
agent设计模式的本质是将人类的思维模式和管理方式以结构化prompt的方式告诉llm进行规划，并调用工具执行，且不断迭代

## 模式
### 设计模式（架构层）
+ ReAct（观察-行动-感知）
+ Plan-and-Execute（规划-执行）
+ Reflection（反思）：核心在于迭代，而迭代的前提是能够记住之前的尝试和获得的反馈。因此短期记忆是必不可少的模块
+ Multi Agent

### 推理模式--赋予llm规划能力（llm思考层）
+ CoT(Chain of Thought)：在`prompt`增加{step by step/让我们一步步思考}
    + `Zero-shot CoT`：零示例
    + `Few-shot CoT`：在`prompt`增加实际的例子{bad case | good case}
    + 单一路径风险，一步错步步错
+ ToT(Tree of Thought)：1.生成多个候选思路2.评估思路可行性3.择中选优
    + 同时探索多条推理路径
+ GoT(Graph of Thought)：允许不同路径的中间结果合并、复用

### ReAct & CoT
CoT是一个连续思考的过程，不能被打断，思考结束执行下一步
潜在的问题：无法获取实时信息，不能及时错误纠正
引入ReAct，根据行动输出进一步观察，重新思考，继续行动
思考可以被打断，可以通过观察获取实时信息，及时错误纠正

## 上下文管理
上下文工程是在上下文窗口为下一步填充恰到好处信息的精妙艺术和科学
上下文包括：`Instructions(prompt)` `Knowledge(rag)` `Tools`
### 写入上下文
将其保存在上下文窗口之外，帮助Agent执行任务
+ 临时笔记板。可以是简单的工具调用，写入文件；也可以是会话期间持久的运行时状态对象中的字段；帮助Agent在给定会话内完成任务
+ 记忆。从多个会话中记住事情；从过去Agent反馈集合中定期合成记忆
### 读取上下文
将其拉入到上下文窗口，帮助Agent执行任务
+ 临时笔记板。通过调用工具读取；可以选择在每个步骤向Agent公开状态的哪些部分
+ 记忆。选择少量示例作为期望行为的示例(情景记忆)；指令引导行为(程序记忆)；实事作为任务相关上下文
+ 工具。工具多且描述重叠会混淆模型的判断，一种方案是对工具描述应用rag
+ 知识。

### 压缩上下文
保留执行任务所需的token
+ 上下文摘要。通过llm压缩
+ 上下文修剪。删除较旧的信息

### 隔离上下文
+ 使用agent隔离。拆分上下文到子agent帮助Agent执行任务，其中一组agent可以处理特定的子任务，每个agent都有特定的工具集、指令和自己的上下文窗口
+ 使用环境隔离。输出包含所需工具的代码，在沙箱中运行，然后将选定的上下文传回llm
+ 状态。状态对象可以用具有上下文可以写入字段的模式设计，模式的字段可以在agent每个轮次暴露给llm，且可以将其他信息隔离在其他字段

### langgraph与上下文管理
+ 上下文处理策略对照表

| 策略 | 核心目标 | 核心依赖 | 解决痛点 |
| ---- | ---- | ---- | ---- |
| 写入上下文 | 持久保存中间临时信息 | Checkpointer+Store 双层记忆 | 分步推理数据丢失、会话中断恢复 |
| 选择上下文 | 精准筛选送入 LLM 的内容 | 节点粒度状态读取 + 检索 | Prompt 冗余 Token、无关信息干扰 |
| 压缩上下文 | 降低上下文 Token 占用 | 自定义状态处理节点 | 上下文窗口溢出、超长会话截断 |
| 隔离上下文 | 分割不同类型数据，按需加载 | 自定义 State 字段结构 | 原始工具数据污染主推理上下文 |


### RAG is dead
+ 传统RAG vs 上下文工程视角(高级RAG)

|阶段|传统RAG|上下文工程方法|
|-----|--------|----------|

# 自进化Agent
> 自进化Agent是一种能够依据自身交互轨迹、任务反馈或环境信号，对上下文、记忆、技能、工具、工作流、代码或模型参数进行持续更新，并让这些更新影响未来任务表现的智能体

## 实践路线
+ 第一阶段：内建上下文闭环。让agent具备最基本的成长能力，记录用户偏好、总结失败经验、检索历史会话、把复杂任务沉淀为操作步骤
+ 第二阶段：把经验沉淀为skill。 当经验开始重复出现，应该从记忆层提升到技能层。技能比记忆更结构化，更容易迁移给其它agent
+ 第三阶段：引入共享层和群体智能。当多个用户、多个设备、多个agent都在积累经验时，需要一个共享层去重、验证、合并和分发
+ 第四阶段：谨慎尝试参数、代码或工作流自修改。

上下文/记忆自进化 -> 技能资产自进化 -> 群体经验自进化 -> 参数、代码或工作流自进化


## 自进化系统的关键问题
+ 评估器比生成器更重要。自进化的核心不是改动，而是改动没有让系统变坏
+ 可回滚是自进化的基础设施。
+ 共享经验需要治理。权限分层、PII脱敏、候选验证、版本审计
+ 技能供应链安全不能事后再补。

# Agent评估
agent评估至少关注三个层次：最终回复是否正确、执行路径是否合理、关键节点是否稳定。复杂Agent不能只看最终答案，因为答案偶尔正确不代表流程可靠，特别是涉及路由、工具调用、数据库写操作时，更需要过程级评估。

## 三类评估
+ Final response（最终回复评估）。面向端到端结果，输入通常是用户问题，输出是Agent最终回复。适合衡量用户真实感知，但只能看到“结果对不对”，不容易定位具体是哪一步出了问题。
+ Trajectory（轨迹评估）。检查Agent是否走了预期步骤，比如是否进入了正确子图、是否调用了正确工具、顺序是否基本合理。相比简单的通过/失败，更适合给复杂Agent“部分分”，即使最后答案错了，也能看出哪些步骤是对的。
+ Single step（单步评估）。把某个高风险节点单独拎出来测，例如意图路由、首个工具选择、参数提取、是否触发澄清提问。它对调试和迭代最有价值，因为问题定位最直接。

## 评估设计方法
+ 先让Agent“可测试”。如果Agent包含退款、删除、下单这类副作用操作，评估时要提供`test/mock`运行模式，避免真实修改外部系统。
+ 为不同评估目标准备数据集。端到端评估的数据集可包含`question -> 标准回复/标准轨迹`；单步评估的数据集则直接围绕某个节点设计，如`messages -> route`。
+ 明确目标函数输出。最终回复评估要求目标函数返回`response`；轨迹评估要求返回`trajectory`；单步评估则只返回某个节点结果，例如`route`。
+ 评估器要贴合任务。语义等价的最终回复适合用`LLM-as-a-judge`；轨迹评估适合比较“实际步骤序列”与“期望步骤序列”；单步评估常用精确匹配即可。

## 轨迹评估的意义
+ 复杂Agent的失败点往往不止一个，单纯看最终结果损失信息太大。
+ 轨迹评估本质是比较“实际路径”和“期望路径”，例如`refund_agent -> lookup -> refund`是否按预期发生。
+ 评分不一定是0/1，可以按完成了多少期望步骤来计算比例，更适合衡量复杂流程的稳定性。
+ 如果使用LangGraph，可以通过流式事件记录子图节点进入情况和工具调用情况，把这些事件还原成可比较的执行轨迹。

## 单步评估的意义
+ 端到端测试信号最真实，但调试成本高。
+ 单步评估适合压测最脆弱的决策点，例如“用户是在问商品信息还是申请退款”“当前是否应该继续追问身份信息”。
+ 当单步指标下降时，通常能直接定位到提示词、分类逻辑、结构化抽取或工具选择策略的问题。

## 实践建议
+ 先建端到端评估，确认Agent整体可用；再补轨迹评估，确认流程没有走偏；最后把最关键节点拆成单步评估，提升迭代效率。
+ 不同评估最好分开维护数据集，不要把所有目标混在同一个集合里，否则分析结论容易发散。
+ 对有副作用的工具，评估环境必须默认走mock分支，这是Agent可持续回归测试的基础设施。
+ 评估不是一次性动作，而是每次改Prompt、改工具、改路由、改记忆策略之后都要重复运行的回归机制。
