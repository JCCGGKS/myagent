目标：把用户的一句话（或一段对话）映射成 Agent 下一步该做什么。

可以把它理解成一个“路由器（router）”：输入自然语言，输出结构化的 意图（intent）+ 参数（slots）+ 置信度 + 是否需要追问，再交给后续的工具调用、工作流编排或子 Agent。

0. 先把“意图识别”在 Agent 场景里说清楚

传统 NLP 里的意图识别（Intent Classification）多用于客服/机器人：

  • 输入：用户一句话
  • 输出：意图标签（比如：查订单、退货、投诉…）

而在 Agent 场景，意图识别通常还要额外产出：

  1. 动作类型：回答/调用工具/发起检索/创建计划/执行工作流/升级人工…
  2. 参数：工具调用需要的字段，例如 city=杭州、date=明天、repo=xxx
  3. 多轮策略：是否要先澄清（缺槽位/歧义）
  4. 安全与权限：是否涉及敏感操作（删除、转账、改配置等）

所以一个更“Agent 化”的输出长这样：

{
	"intent": "book_flight",
	"action": "call_tool",
	"slots": {"from": "上海", "to": "北京", "date": "2026-01-20"},
	"confidence": 0.86,
	"needs_clarification": true,
	"clarify_questions": ["请问出发时间偏上午还是下午？"],
	"risk_level": "low"
}

补充：槽位（slot）到底是什么意思？

槽位（slot）可以理解成：为了把“自然语言的意图”变成“可执行动作”，你必须从用户话里抽取出来的参数位。

  • 在传统对话系统里：intent + slots 往往等价于“用户要做什么 + 做这件事需要哪些信息”。
  • 在 Agent/工具调用里：slots 往往就是工具函数的入参（arguments），或者工作流里某一步需要的关键字段。

一个直观类比：

  • **意图（intent）**像“函数名”
  • **槽位（slots）**像“函数参数”
例子 1：订机票（结构化参数）

用户：\

“明天从上海飞北京，尽量便宜点。”

  • intent：book_flight
  • slots 可能包括：
    • from=上海（出发地）
    • to=北京（目的地）
    • date=明天（日期）
    • preference=便宜（偏好）

如果少了关键槽位（比如缺 date 或缺 to），Agent 通常就必须追问：

  • “你想飞到哪个城市？”
  • “是明天上午还是下午？”
例子 2：文档助手

用户：\

“把 firstResearch.md 总结成 5 条要点。”

  • intent：summarize_doc
  • slots：source_type=file、source=firstResearch.md、count=5、style=bullet
槽位的几个常见设计点

  1. 必填槽位 vs 可选槽位：

    • 必填：缺了就无法执行（比如订机票缺目的地）
    • 可选：缺了也能执行，但效果可能差（比如总结风格/长度）

  2. 槽位值要尽量可校验/可规范化：

    • 日期（“明天/下周一”→ 规范化成 YYYY-MM-DD）
    • 金额（“两百以内”→ budget_max=200）
    • 文件（是否存在，路径是否允许访问）

  3. 槽位 ≠ 实体（entity）但经常重叠：

    • 实体抽取偏“识别文本里有什么”（人名、地点、组织…）
    • 槽位抽取偏“为了完成意图，需要哪些字段”（更任务导向）
    • 很多时候槽位值来自实体，但槽位会带上角色语义（比如 from 和 to 都是地点实体，但语义角色不同）。

1) 第一步：先定义你的“意图体系”（比模型更重要）

1.1 别一上来就做 200 个 intent

很多团队失败的原因不是模型不行，而是：意图体系乱、边界不清、缺少兜底。

建议从一个最小可用集合（MVP）开始：

  • 通用动作意图（几乎所有 Agent 都要）：
    • ask_question（需要回答/解释）
    • retrieve_info（需要查资料/检索/读文档）
    • execute_task（需要做事：写代码、生成报告、跑命令等）
    • follow_up（基于上下文继续）
    • chitchat（闲聊）
    • unsupported（超出能力范围）

再在 execute_task 下面细分“工具/工作流级意图”。

1.2 用“可执行动作”来命名 intent

好的 intent 命名具备：

  • 动词化：create_ticket、summarize_doc、run_sql、book_meeting
  • 可被直接路由到工具/流程：看到 intent 就知道下一步执行什么
  • 边界清晰：减少重叠

1.3 给每个 intent 写一个“契约（contract）”

每个 intent 至少写清楚四件事：

  • 触发条件：用户什么表达算这个 intent
  • 反例：哪些看似相似但不算
  • 需要的槽位（slots）及类型
  • 缺槽位时追问策略

示例：summarize_doc

  • 触发：总结/提炼/概括某个文档或链接
  • 反例：写新文章（不是总结）
  • slots：source_type(file/url/text)，source，style(bullet/paragraph)，length
  • 追问：缺少 source 时先问“你要总结哪个内容？”

2) 数据怎么做：标注规则与样本（从小到大）

2.1 先做“黄金集”（Gold Set）

不需要一开始就大规模标注。建议：

  • 每个 intent 先收集 30～100 条高质量样本
  • 覆盖口语、错别字、省略、指代、同义表达
  • 加上一批“困难样本”：
    • 多意图："帮我总结下这篇文章，然后给我列 5 个标题"
    • 歧义："把那个关了"（关什么？）
    • 上下文依赖："继续"、"就按刚才那个"

2.2 标注什么

标注不只 intent，还要标注 slots 和“是否需要澄清”。推荐字段：

{
	"text": "明天早上从上海飞北京，顺便订个靠窗位",
	"intent": "book_flight",
	"slots": {"from": "上海", "to": "北京", "date": "明天", "time": "早上", "seat": "window"},
	"needs_clarification": true,
	"clarify": "需要确认航空公司/预算/是否高铁可选",
	"language": "zh",
	"channel": "app"
}

2.3 常见坑

  • 只标意图不标槽位：上线后工具参数错误会爆炸
  • intent 太细导致数据稀疏
  • 反例不写清，引发“标签漂移”

3) 由浅入深的实现路线

下面是一条典型路线，从最容易落地到最强能力。

阶段 A：规则/关键字路由（最快上线）

适用：业务很窄、意图很少、容错要求不高。

做法：

  • 正则/关键词/词典
  • 简单优先级：更具体的规则优先
  • 必须配 fallback（兜底）

例子：

  • 句子包含“总结/概括/提炼” → summarize_doc
  • 包含“发邮件/邮件给/抄送” → send_email

优势：解释性强、成本低。 劣势：召回差，难处理口语和复杂表达。

阶段 B：传统/轻量分类器（可控、稳定）

适用：意图 10～100 个、希望稳定、成本低。

常见方案：

  • TF-IDF + Linear SVM / Logistic Regression
  • Sentence-BERT 向量 + KNN/浅层分类

特征：

  • 推理延迟小
  • 可离线训练
  • 但对上下文、多意图、工具参数抽取能力有限

阶段 C：LLM 做 intent + slots（现在主流）

适用：需要理解强、意图变化快、支持多语言、多轮。

核心思想：让 LLM 输出 严格结构化 JSON，你只做校验与路由。

C1. “单轮 Router Prompt” 模板

你可以用下面这种形式（关键是：给 schema、给可选 intent 列表、给少量示例）：

输入：用户消息 + 可选上下文摘要

输出：一个 JSON

示例（伪 prompt）：

  • 角色：Router
  • 任务：在给定 intent 列表中选择，并抽取 slots
  • 约束：只输出 JSON；不可编造 slots；缺信息要标记 needs_clarification=true
C2. 输出校验

无论模型多强，工程上都要：

  • JSON parse
  • schema validate
  • slots 类型校验（日期、金额、邮箱等）
  • confidence 低时降级策略（追问/回退到检索问答/人工）

阶段 D：多步路由（Router → Planner → Executor）

当任务变复杂你会发现：

  • 用户一句话里可能包含多个动作
  • 需要先计划再执行

这时通常拆成：

  1. Router：识别意图/风险/是否需要澄清
  2. Planner：把任务拆步骤（可能是工作流）
  3. Executor：调用工具、读写文件、迭代执行

举例：

用户："把这个 repo 的 README 重写一下，顺便加上 Windows 安装说明"

  • Router：update_docs（需要上下文：repo 路径、目标风格）
  • Planner：列 3～5 个步骤
  • Executor：读文件→写文件→跑 lint/test

4) 关键难点：多意图、上下文、与“澄清”

4.1 多意图（Multi-intent）怎么处理

两种常见策略：

策略 1：输出一个主意图 + 子任务列表（推荐用于 Agent）

{
	"intent": "composite",
	"tasks": [
		{"intent": "summarize_doc", "slots": {"source": "..."}},
		{"intent": "generate_titles", "slots": {"count": 5}}
	]
}

策略 2：强制只选一个

适用于流程非常严格的系统，但会牺牲体验。

4.2 上下文依赖（指代/省略）

用户经常说："按刚才那个"、"继续"、"就这么办"。

工程上你要准备一个 对话状态（state），至少包含：

  • 上一个意图
  • 上一个工具调用的参数
  • 关键实体（文件名、日期、城市、客户 id…）

Router 输入里要带一个“对话摘要”，而不是把全对话都塞进去（成本与噪声）。

4.3 什么时候追问（Clarification Policy）

这是 Agent 成败的关键之一。

建议你把“追问”当作一种显式动作：

  • 缺少必要槽位：无法安全执行
  • 风险操作：删除/付款/改权限
  • 代价大：可能耗时/耗钱/不可逆

例子：

  • 用户："把我昨天的文件删了"
  • Router 输出：delete_file，但 file_path 不明确 + 风险高
  • 应追问："你指的是哪个文件？我看到昨天你编辑了 A、B 两个文件。"

5) 评测怎么做：别只看 accuracy

意图识别在 Agent 里更像一个“决策系统”。推荐指标：

  1. Intent 准确率：Top-1 / Top-k
  2. 槽位准确率：
    • Slot F1（抽取对不对）
    • 值是否可用（日期解析是否成功）
  3. 澄清策略质量：
    • 该追问时有没有追问（Recall）
    • 不该追问时是否过度追问（Precision）
  4. 端到端成功率：最终任务是否完成（最重要）
  5. 安全误触发率：高风险动作误判成可执行的比例

评测集建议包含：

  • 常规样本
  • 困难样本（歧义、多意图、上下文、省略）
  • 攻击样本（提示注入/越权请求）

6) 上线工程：路由架构与降级

一个实用的线上路由链路（从便宜到昂贵）：

  1. 规则（fast path）命中 → 直接路由
  2. 轻量分类器（cheap model）
  3. LLM Router（strong model）
  4. 仍不确定 → 追问 / 转人工 / 兜底问答

并加上：

  • 置信度阈值：低于阈值不执行高风险工具
  • 白名单：只有某些 intent 允许直接执行
  • 日志与回放：保存 router 输入输出，便于持续迭代

7) 两个贯穿例子：从简单到复杂

例子 1：文档助手

假设你要做一个“研究笔记 Agent”，意图可能是：

  • summarize_doc
  • extract_outline
  • compare_docs
  • generate_todo
  • ask_question

用户说："把 firstResearch.md 总结成 5 条要点"

Router 期望输出：

{
	"intent": "summarize_doc",
	"action": "call_tool",
	"slots": {"source_type": "file", "source": "firstResearch.md", "style": "bullet", "count": 5},
	"confidence": 0.93,
	"needs_clarification": false
}

用户说："再短一点"（上下文依赖）

Router 期望输出：

{
	"intent": "summarize_doc",
	"action": "continue",
	"slots": {"length": "shorter"},
	"confidence": 0.84,
	"needs_clarification": false
}

例子 2：企业 IT Agent（含风险控制）

用户："把生产环境的日志级别调成 debug"

Router 输出（必须追问 + 风险高）：

{
	"intent": "update_config",
	"action": "need_approval",
	"slots": {"env": "prod", "key": "log_level", "value": "debug"},
	"confidence": 0.88,
	"needs_clarification": true,
	"clarify_questions": [
		"这是生产环境，打开 debug 可能影响性能与敏感信息暴露。要持续多久？",
		"需要走变更审批吗？"
	],
	"risk_level": "high"
}

8) 用 LangGraph 直观实现：一个最小的 Intent Router 图

下面是一个最小可运行的 demo，只包含 3 个文件：

  • intent_router_graph.py：最小图
  • requirements.txt：最小依赖
  • README.md：运行方式

8.1 这个例子对应了哪些概念？

在 intent_router_graph.py 里：

  • AgentState：图里传递的状态（相当于你的“对话状态 state”）
  • route_intent()：Router 节点（负责识别 intent + 抽取 slots）
  • add_conditional_edges(...)：条件路由（根据 intent 决定走哪个节点）
  • summarize_doc / compare_docs / ask_question：不同意图的处理节点（这里用占位输出，保持 demo 简单）

你可以把它理解成：

Router（意图识别）先把自然语言结构化 → Graph 再按 intent 分发到不同执行分支。

8.2 Router 输出的结构，和本文前面 JSON 有什么关系？

这个 demo 里 router 的输出被写回到 state：

  • intent: 例如 summarize_doc
  • slots: 例如 { "source": "firstResearch.md", "count": 5 }

这正对应前文的思想：

  • intent = “要做什么”
  • slots = “做这件事需要的参数”

只是 demo 为了少代码，暂时没加：confidence / needs_clarification / risk_level。

真实项目中你通常会把这些字段也放进 state，然后在 decide_next() 里加上：

  • 低置信度 → 走 ask_clarify
  • 高风险 → 走 need_approval

8.2.1 这些字段是“规定的吗”？我能任意加字段吗？

结论：不是规定的，你可以按业务需要自定义字段。

  • 在 LangGraph 里，state 本质上就是节点之间传递的一份结构化数据（通常是 dict）。
  • 你在 AgentState(TypedDict) 中声明的字段，更多是为了类型提示和团队协作的“约定”，LangGraph 并不会强制只能用这些字段。

因此，除了 intent/slots，你可以按需求添加各种“路由相关字段”，例如：

  • confidence: 置信度（0~1）
  • needs_clarification: 是否需要追问
  • clarify_questions: 追问问题列表
  • risk_level: 风险等级（low/medium/high）
  • tool_name/tool_args: 如果你直接在 Router 阶段就决定要调用哪个工具
  • missing_slots: 缺失的关键槽位
  • requires_approval: 是否需要审批

经验：字段当然可以多，但建议以“能驱动决策”为准，否则字段会变成无用日志。

8.2.2 如果我自定义字段，那 LLM 怎么“理解并正确填这些字段”？

LLM 不会自动知道你的字段含义。要让它稳定输出，你需要把这些字段当作一份 路由协议（routing schema） 来设计，并在 prompt 里“讲清楚 + 给例子 + 做校验”。

(1) 在 prompt 里给出明确的 Schema（字段、含义、取值范围、约束）

比如你希望 LLM 输出下面这些字段（示例）：

{
	"intent": "summarize_doc | compare_docs | ask_question | unsupported",
	"slots": {"...": "..."},
	"confidence": 0.0,
	"needs_clarification": false,
	"clarify_questions": [],
	"risk_level": "low"
}

你需要在 Router prompt 里明确说清：

  • intent 必须从给定列表中选（禁止自由发挥）
  • slots 只能从用户输入/上下文中抽取，不要编造
  • confidence 范围是 0~1（并简单约定打分标准：越确定越接近 1）
  • needs_clarification=true 的触发条件（例如缺关键槽位/有歧义/风险高）
  • risk_level 的枚举与判定标准（例如涉及删除/改配置/付费 → high）
(2) 给 2～4 个“对齐示例”（few-shot）

示例非常重要，它会让模型学会你的字段应该如何填、何时为空、何时需要追问。

(3) 输出必须可校验（schema validate），失败就重试或降级

真实系统里通常会做：

  • JSON 解析
  • schema 校验（字段是否齐全、类型是否正确、枚举是否合法）
  • 置信度/风险阈值判断（不满足就不执行、转追问/人工）

经验：“任意字段”是可以的，但要让它变成“可依赖的接口”，就得有 schema + 校验。

8.4 如何把它变成“真实的意图识别”？

保持图结构不变，只替换一个点：把 route_intent() 从“关键词规则”换成“LLM Router 输出 JSON”。

换句话说：

  • LangGraph 负责流程编排与路由
  • LLM/分类器负责识别 intent + 抽取 slots

这就是很多 Agent 项目最常见、也最稳的落地方式。

"""一个尽量精简的 LangGraph 示例：
- router 节点负责“意图识别”（此处用极简规则模拟；真实项目可替换为 LLM 结构化输出）
- 根据 intent 路由到不同的处理节点

运行：python intent_router_graph.py
"""

from __future__ import annotations

from typing import Any, Dict, Literal, Optional, TypedDict

from langgraph.graph import END, StateGraph


# --- 1) 定义 State（图中节点之间传递的结构化状态）
class AgentState(TypedDict, total=False):
    user_text: str
    intent: str
    slots: Dict[str, Any]
    answer: str


# --- 2) Router：这里用极简规则模拟“意图识别 + 槽位抽取”
#     真实项目中：替换成 LLM Router（输出 JSON: intent/slots/confidence/needs_clarification...）

def route_intent(state: AgentState) -> AgentState:
    """Router 节点：根据用户输入识别 intent，并抽取 slots 写回 state。"""
    text = state["user_text"].strip()

    # 极简 demo：识别 3 个 intent
    if "总结" in text or "概括" in text:
        # 槽位抽取也做得“能用就行”：演示 slot 的概念
        slots: Dict[str, Any] = {"source": _guess_filename(text), "count": _guess_count(text) or 5}
        return {"intent": "summarize_doc", "slots": slots}

    if "对比" in text or "比较" in text:
        a, b = _guess_two_filenames(text)
        return {"intent": "compare_docs", "slots": {"a": a, "b": b}}

    return {"intent": "ask_question", "slots": {}}


def _guess_filename(text: str) -> Optional[str]:
    """从文本里粗略猜一个 .md 文件名（仅为 demo 简化；真实场景应更可靠）。"""
    # 非严格：只为 demo 服务。真实场景建议用 LLM 或更可靠的解析。
    for token in text.replace("，", " ").replace(",", " ").split():
        if token.endswith(".md"):
            return token
    return None


def _guess_two_filenames(text: str) -> tuple[Optional[str], Optional[str]]:
    """从文本里粗略猜两个 .md 文件名（用于 compare_docs 的 slots）。"""
    files = []
    for token in text.replace("，", " ").replace(",", " ").split():
        if token.endswith(".md"):
            files.append(token)
    if len(files) >= 2:
        return files[0], files[1]
    if len(files) == 1:
        return files[0], None
    return None, None


def _guess_count(text: str) -> Optional[int]:
    """从文本里粗略提取数字（如“5条/5 点”），用于 count 槽位。"""
    # 非严格：识别“X条”“X 点”这类
    digits = "".join([c for c in text if c.isdigit()])
    return int(digits) if digits else None


# --- 3) 各 intent 对应的处理节点（这里用占位输出）

def summarize_doc(state: AgentState) -> AgentState:
    """summarize_doc 节点：根据 slots 生成“总结任务”的占位输出（示例用）。"""
    slots = state.get("slots", {})
    src = slots.get("source") or "<未指定文件>"
    count = slots.get("count") or 5
    return {"answer": f"[summarize_doc] 将总结 {src}，输出 {count} 条要点（此处省略具体总结实现）"}


def compare_docs(state: AgentState) -> AgentState:
    """compare_docs 节点：根据 slots 生成“对比任务”的占位输出（示例用）。"""
    slots = state.get("slots", {})
    a = slots.get("a") or "<A 未指定>"
    b = slots.get("b") or "<B 未指定>"
    return {"answer": f"[compare_docs] 将对比 {a} vs {b}（此处省略具体对比实现）"}


def ask_question(state: AgentState) -> AgentState:
    """ask_question 节点：兜底分支，把输入当作普通问答处理（示例用）。"""
    q = state.get("user_text", "")
    return {"answer": f"[ask_question] 我会把它当作问答来处理：{q}"}


# --- 4) 条件路由：根据 router 识别的 intent，决定走哪个节点

def decide_next(state: AgentState) -> Literal["summarize_doc", "compare_docs", "ask_question"]:
    """条件路由函数：读取 state.intent，决定 router 之后要走的下一个节点名。"""
    intent = state.get("intent")
    if intent == "summarize_doc":
        return "summarize_doc"
    if intent == "compare_docs":
        return "compare_docs"
    return "ask_question"


def build_graph():
    """构建并编译 LangGraph：注册节点、设置入口、配置条件边与结束边。"""
    # 这个图的结构（文本示意图）：
    #
    #                +--------------------+
    #                |       router       |
    #                |  route_intent()    |
    #                +----------+---------+
    #                           |
    #                 (conditional edges)
    # intent=summarize_doc / compare_docs / ask_question
    #              _________|______________
    #             /          |              \
    #            v           v               v
    #  +----------------+ +----------------+ +----------------+
    #  |  summarize_doc | |  compare_docs  | |  ask_question  |
    #  +-------+--------+ +-------+--------+ +-------+--------+
    #          |                  |                  |
    #          v                  v                  v
    #         END                END                END
    #
    g = StateGraph(AgentState)

    g.add_node("router", route_intent)
    g.add_node("summarize_doc", summarize_doc)
    g.add_node("compare_docs", compare_docs)
    g.add_node("ask_question", ask_question)

    g.set_entry_point("router")

    g.add_conditional_edges(
        "router",
        decide_next,
        {
            "summarize_doc": "summarize_doc",
            "compare_docs": "compare_docs",
            "ask_question": "ask_question",
        },
    )

    g.add_edge("summarize_doc", END)
    g.add_edge("compare_docs", END)
    g.add_edge("ask_question", END)

    return g.compile()


if __name__ == "__main__":
    app = build_graph()

    examples = [
        "把 firstResearch.md 总结成 5 条要点",
        "对比 firstResearch.md 和 supprot1.md 的核心观点",
        "温度设为 0 一定可复现吗？",
    ]

    for text in examples:
        out = app.invoke({"user_text": text})
        print("\n== user ==")
        print(text)
        print("== state(out) ==")
        print(out)
