# 轨迹评估优化方案

基于 [`trajectory_eval_report.md`](../../../eval/trajectory/trajectory_eval_report.md)（1000 条，规则评估）+ 答案评估的发现给出。
本方案动「决策 / 工具选择」与「意图路由」两环，**不重构图结构**（路径已 98% 以上正确）。

文档结构：
- **一、优化原理与方案**——问题根因 + 具体优化手段（为什么改、改什么）；
- **二、优化效果（对比基线）**——优化前后各项指标、按类别水位、残余失败；
- **三、总结**——核心优化手段 + 后续优化方向。

---

## 一、优化原理与方案

### 1.1 问题根因

1. **工具描述才是 function calling 的真实决策输入，而当前描述有歧义/过宽**。
   工具信息通过 `tools=` 参数（结构化 schema）注入 LLM，并不在系统提示词里罗列
   （见 `build_agent_system_prompt` 注释「可调用工具通过 tools= API 参数下发，不在此罗列」）；
   LLM 选哪个函数主要看各工具的 `description` + `parameters`。当前 `query_order` /
   `query_logistics` 描述重叠、`create_handoff` 过宽，导致选错。**修描述即可，
   无需在系统提示词里再重复一份路由表**（那会与 schema 形成双源、且提示词本就不罗列工具）。
2. **`query_order` 与 `query_logistics` 描述重叠**：都吃 `order_id`，都含"发货"语义。
   "发货了吗"类问题二义（已发货=订单状态 vs 运输中=物流），易选错。
3. **`create_handoff` 描述过宽**：退款类（缺订单号或不确定）频繁误触发转人工。
4. **工具选择走 LLM function calling，非确定性**：temperature 0.3 带来方差
   （答案评估已观察到同 case 浮动）。
5. **多轮跟进承接不稳**：`slot_followup` 在裸订单号样本上未稳定生效，意图先错，
   工具随之错（跨单点评估的连带问题）。
6. **退款子意图消歧缺失（意图路由层主瓶颈）**：
   `consult_policy`（咨询政策）/ `request_refund`（实际办理）/ `no_reason_return`（无理由退货）
   在 `intent_rules.yml` 与 `IntentRouterService` 中靠关键词区分，对「七天无理由怎么退 /
   退款规则 / 退款政策 / 退货退款 / 这单不想要了退了吧」这类**语义重叠**样本容易误判。
   工具选对，但**意图标签错** → 总准确率被拖累。这是意图路由层问题，非 schema/RAG。
7. **金标与评估脚本自身的两处缺陷**（不影响 agent，但会让结论误读，已修）：
   - 报告「按类别」表格分母为全局 1000（`run_eval.py` 的 `pct()` 用全局 `total`），
     应除以类别自身 `b["total"]`；否则每格都偏小、且"总准确率==意图"的假象掩盖真实类别内水位。
   - 金标 `REFUND_CONSULT` 的 `expected_tool=None`：咨询类不强校验工具，于是 agent 执行
     `request_refund` 也能过 `action`，只扣 `intent`。若要真正测"咨询应走知识检索"，
     应把 `expected_tool` 设为 `knowledge`（rag_retrieve）。

涉及文件：
- `app/business/prompts/system.py` → `build_agent_system_prompt`
- `app/business/tools/tool_executor.py` → `TOOL_SCHEMAS` 描述（P0 已改）
- `app/business/tools/rag_tool.py` → `RagRetrieveTool.description`（P0 已改）
- `app/business/agent/agent_node.py` → 决策温度 / 硬护栏（P3）
- `app/config/llm.py` → `LLMConfig.generation_kwargs`
- `app/business/intent/routing.py` → `slot_followup` 分支（多轮，P3）
- `app/business/intent/schema.py` + `config/intent_rules.yml` → **退款子意图消歧（P1）**
- `eval/trajectory/run_eval.py` / `eval/answer/run_eval.py`（`pct` 分母）→ 评估自身修复（已修）

### 1.2 优化方案（按收益 / 风险排序）

#### P0 — 强化工具元信息描述（最高收益，低风险，主修复手段）
工具选择由 `tools=` schema 驱动，因此**只改 `TOOL_SCHEMAS` 的 `description` / `parameters`**
即可，不动系统提示词（提示词继续只传意图/槽位上下文，不重复罗列工具路由）。

1. **消歧 `query_order` / `query_logistics`**：
   - `query_order`：聚焦**订单本身**——是否下单 / 是否已发货 / 待付款 / 商品 / 金额。
     描述去掉"发货情况"歧义，改"是否已发货"。
   - `query_logistics`：聚焦**已发货后的运输过程**——快递到哪了 / 运输中 / 派送中 / 是否签收。
   - 在两者描述里互相划界：**"已发货"属订单状态 → `query_order`；"运输中/派送中"属物流 → `query_logistics`**。
2. **收紧 `create_handoff` 触发条件**：
   描述明确"**仅当**用户明确要求人工 / 投诉情绪升级 / 多次澄清失败才调用；
   **不因缺订单号或信息不全而转人工**——先用业务工具，由工具提示用户补全"。
   降低退款类误转人工。
3. **分流 `request_refund` / `rag_retrieve`（after_sale_refund）**：
   - `request_refund`：强调"用户要**实际办理**退款/退货/换货/维修（而非仅咨询政策）时调用"；
   - `rag_retrieve`：增加"退换货政策、七天无理由等知识咨询"触发语，使纯咨询走知识检索而非转人工。

> 注：系统提示词（`build_agent_system_prompt`）保持现状即可——它已把
> `当前意图` / `已填槽位` 透传给 LLM，工具选择交给 schema 描述驱动。若描述优化后
> 个别样本仍选错，再考虑在调度节点补**一句**轻量约束（如"按当前意图选择对应工具"），
> 但不必重复整套路由表（避免与 schema 双源维护）。

#### P1 — 退款子意图消歧（意图路由，原主瓶颈）
工具选择已正确（P0），但**意图子标签**是残余缺口的主因：退款家族 399 条仅约 70–75%
子意图准确。根因在 `intent_rules.yml` / `IntentRouterService` 对 `consult_policy` /
`request_refund` / `no_reason_return` 的区分靠关键词，对语义重叠样本易误判。

落地方向（意图路由层，非 schema）：
1. **新增 `consult_policy` 咨询优先规则**：在通用 `consult_policy` 规则之前插入一条
   高优先级规则（first-match-wins），关键词覆盖「规则 / 政策 / 怎么退 / 怎么退款 / 退货流程 /
   退款条件 / 能不能退 / 退款吗 / 七天无理由怎么退」等探索性表述，将咨询类明确归 `consult_policy`，
   避免被「我要退款 / 帮我退款」等动作词经 `action_sub_intent` 误覆盖为 `request_refund`。
2. **收紧 `no_reason_return` 关键词**：由 `[无理由, 七天无理由, 不想要了想退]` 改为
   `[无理由退货, 无理由退]`，避免「七天无理由怎么退」这类咨询被抢去 `request_refund`。
3. **`consult_policy` 动作触发词 → `request_refund`**：追加 `退货退款 / 这单不想要了退了吧 /
   不想要了退了吧 / 我要退货 / 申请退货` 等强动作表述，经 `action_sub_intent` 归为实际办理。
4. **金标对齐（评估侧）**：把 `REFUND_CONSULT` 的 `expected_tool` 设为 `knowledge`
   （`rag_retrieve`），让"咨询应走知识检索"成为可测约束，避免 agent 执行退款也能过 `action`。

#### P1b — 投诉 / 多轮小修（高性价比，并入 P1 一起做）
- **投诉误路由**：`complaint`（如「越想越气」）被当闲聊走 `answer_directly`。
  在 `complaint.service_complaint` 关键词补 `气死我了 / 越想越气 / 态度` 等情绪词，修复误路由。
- **多轮上下文串味**：`multiturn_followup` 动作偏低（`order_query` / `logistics` 选错）。
  强化 `slot_followup`：order_id 命中 + `previous_sub_intent ∈ 跟进集` → 强制对应工具，
  避免被 LLM 兜底覆盖（详见 P3 多轮）。

#### P2 — RAG 检索质量（次要，等 P1 路由正确后再做）
P1 把咨询正确路由到 `rag_retrieve` 后，检索质量才成为关键。当前裁判扣分"缺政策事实点"
部分源于检索弱/空。落地方向（`app/business/rag/` + 知识库）：
1. **知识库覆盖**：确认退款/退换货政策 FAQ 已入库（`POST /knowledge/upload`），且
   chunk 切分（`chunk_size` 800 / `overlap` 100）未把「七天无理由」「原路退回」切碎；
2. **检索召回**：`retrieval_strategy` 当前 `bm25`、`min_score_threshold` 5.0。对短政策问句，
   评估 `semantic` / `hybrid` + rerank（`rag_config.rerank.enabled`）；
3. **回复合成**：`response_generator` 在 `rag_retrieve` 空/弱结果时回退到明确话术而非含糊带过。

#### P3 — 决策温度确定性
为 `agent_node` 的工具决策步**单独固定 `temperature=0`**（或 0.1），与回复生成温度解耦。
实现：在 `LLMConfig` 增加 `agent_temperature` 字段，`agent_node` 构造时覆盖
`generation_kwargs["temperature"]`。收益：消除同 case 浮动（答案评估已观察到同 case 浮动）。

#### P3 —（可选 / 实验）意图 → 工具的硬护栏
在 `agent_node` 中：当 `current_main_intent ∈ {order_query, logistics}` 且 `slots`
有 `order_id` 时，**直接调用对应工具、跳过 LLM function calling**。保证最高频且最易混淆
的两类 100% 正确。风险：降低 ReAct 灵活性，建议作为护栏 / fallback 而非默认，
先以 A/B 对照评估验证再决定是否常驻。

#### P3 — 多轮跟进意图承接（意图路由侧，跨评估）
`multiturn_followup` 动作偏低（工具选错）。排查 `slot_followup`：
- 确认轨迹评估 `run_target` 的 `update_state`（写入 `previous_sub_intent`）在
  checkpointer 读取时序上正确生效；
- 或在 `IntentRouterService` 强化：order_id 命中 + `previous_sub_intent ∈ 跟进集`
  → **强制** `slot_followup`，避免被 LLM 兜底覆盖。

### 2.2 关键修正与口径说明

- **`action` 是真实工具选择指标**：金标指定工具时（`order_query` / `logistics` / `refund_request`）
  会校验 `tool_kind`，所以 `action 95.80%` 是**真实的工具选择修复**，不是空指标。
- **残余缺口的定位**：基线→P0 的总准确率缺口**不是工具选择**，而是**意图子标签**——
  失败样本几乎都是「路径正确、工具选对，仅 `未过维度：意图`」。P1/P1b 正是补这一步。
- **评估报告分母已修正**：「按类别」表格原误用全局 1000 分母，现改为类别自身 `b["total"]`，
  以下 2.4 的类别水位为修正后的真实值。
- **答案侧二轮回落说明**：二轮优化后答案 `key_facts` / 裁判较 P0 轮回落（69.30%→59.80%、
  75.30%→72.10%），根因是 `agent_node` 在 `consult_policy` 下误选 `order_query` 而非
  `rag_retrieve`（如「单号A1002，退款政策」实际走 order_query），导致退款政策事实点
  （七天/原路退回）缺失——属 RAG 工具选择问题，非意图层（意图/路径均正确）。详见 2.6。

### 2.6 残余失败分析（约 1.8%，全部为「动作/工具」维度，路径与意图已正确）

- `multiturn_followup` 56%：金标不自洽（同句 `A1002 怎么样了` 在 ans_0076 期望非物流、

## 三、总结

### 3.1 核心优化手段

1. **强化工具元信息描述（P0，主修复）**：只改 `TOOL_SCHEMAS` 的 `description` / `parameters`
   （`query_order`/`query_logistics` 消歧、`create_handoff` 收紧、`request_refund`/`rag_retrieve` 分流），
   不动系统提示词。直接消除订单↔物流互调、退款误转人工，动作准确率 66.50% → 95.80%。
2. **退款子意图消歧（P1）**：在 `intent_rules.yml` 新增 `consult_policy` 咨询优先规则（first-match-wins，
   覆盖规则/政策/怎么退/流程等探索性表述）+ 收紧 `no_reason_return` 关键词 + 动作触发词归 `request_refund`。
   退款家族 `refund_consult`/`refund_request` 由约 70–75% → **100%**。
3. **投诉情绪关键词补全（P1b）**：`complaint.service_complaint` 补 `气死我了/越想越气/态度`，
   修复投诉被当闲聊误路由，`complaint` → **100%**。
4. **评估报告分母修复**：`run_eval.py`（trajectory + answer）「按类别」表格改用类别内分母，
   消除"总准确率==意图"假象，指标口径正确。

四项叠加：轨迹总准确率 61.10% → **98.20%**，意图 88.20% → **99.90%**。

### 3.2 后续优化方向

- **P2 — RAG 检索质量**：咨询正确路由到 `rag_retrieve` 后，检索弱/空导致 `key_facts` 回落。
  推进知识库覆盖、检索召回（semantic/hybrid + rerank）、回复合成回退，带动 `key_facts` 与裁判分。
- **P3 — 意图→工具硬护栏**：`consult_policy` / `order_query` / `logistics` 等高频易混意图
  强制绑定对应工具，跳过 LLM function calling，使答案质量稳定、不再受单次工具选择波动影响。
- **P3 — 多轮跟进承接**：强化 `slot_followup` 强制对应工具，兜住 `multiturn_followup`（当前 56%）；
  同时建议修订金标不自洽样本（同句不同期望工具）。
- **P3 — 决策温度固定**：`agent_node` 工具决策步单独 `temperature=0`，去方差。
