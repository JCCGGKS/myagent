# 06 工具调用（健壮性 / 风险工具 / 异步化）

聚焦 `ToolExecutor`（`app/business/tools/tool_executor.py`）及其驱动的 domain 服务（`app/business/tools/domain.py`）的健壮性治理：失败隔离（R6）、失败重试（退避）、异步化、退款二次确认（R1）、幂等（R2）。多工具编排本身属 `app/business/tools/tool编排.md` 范畴，本文不重复，只讲"单次 / 批量工具执行如何不翻车"。

## 背景与风险盘点（R1–R7）

`ToolExecutor.run` 接收 LLM 返回的一批 `tool_calls`，逐个执行并回灌 `tool` 消息。执行链此前是**裸调用**：handler 一抛异常就上抛、整批中断；退款/转人工这类动钱/拉真人的动作没有任何护栏。按"风险工具专门处理"原则逐条盘点，得到 R1–R7 共 7 项缺口：

| 编号 | 缺口 | 本轮状态 |
| --- | --- | --- |
| R1 | 退款无二次确认，模型一次调用就动钱 | ✅ 已闭环 |
| R2 | 退款/转人工无幂等，重试/重复调用会重复建单 | ✅ 已闭环 |
| R3 | 多工具并行执行（轮内并发调度） | ⏸️ 扩展优化（非核心缺口，R1+R2 已兜底） |
| R4 | 无意图作用域工具集，始终下发全量工具 | ⏸️ 暂缓（已决策） |
| R5 | 无统一参数 schema 校验层 | ✅ 已闭环 |
| R6 | 无失败隔离，单工具异常中断整批 | ✅ 已闭环 |
| R7 | 强审计：风险动作可追溯、可对账 | ✅ 已覆盖（统一 `event_log` 记录每个 `tool_result` 事件含风险动作，带 `trace_id` + `[tid]` 日志；独立 `ToolCall`/`HandoffRecord` 审计表为可选增强） |

下文 §1–§6 是本轮已落地的六项改造（R6 / 重试 / 异步 / R1 / R2 / R5）。

## §1 失败隔离（R6）

**问题**：`run()` 原是 `for tc in tool_calls: result = self._execute_one(...)` 的裸循环，handler 抛异常会直接上抛到 `agent_node` → 整批工具结果丢失，下游 `response_generator` 无 `tool_result` 可用，用户收到空答或 500。

**优化**（`tool_executor.py:233` `_execute_with_retry`）：把单个工具执行包进 try/except，handler 抛异常不再上抛，而是转成 error 型 `ToolExecutionResult` 回灌：

```python
# handler 主动返回 error = 业务/校验错误，不重试
if isinstance(result, ToolExecutionResult) and result.kind == "error":
    return result
# 重试耗尽仍失败 → 转 error 型 ToolExecutionResult（失败隔离），
# 让 LLM 收到 observation 自行决定下一步，而非中止整批。
logger.error("[tool] %s execution failed after %d attempt(s): %r", ...)
return ToolExecutionResult(kind="error", user_facing_summary=f"工具 {name} 执行出错，请稍后重试或联系人工客服。")
```

批内其余工具照常执行，指标（`observe_tool`）照常记录——一个工具挂不掉整批。

## §2 失败重试（退避，副作用工具不重试）

**问题**：原 `ToolExecutor` 完全没有重试；瞬时故障（DB 抖动、网络闪断）直接转 error，用户被迫重发。但退款/转人工这类**副作用**工具一旦重试，可能重复触发不可逆动作。

**优化**（`tool_executor.py:233` `_execute_with_retry` + 注册表 `side_effect` 标记）：

- 注册表为每个工具标注 `side_effect`（`tool_executor.py:146-152`）：`query_order`/`query_logistics`/`rag_retrieve` = `False`；`create_handoff`/`request_refund`/`modify_address`/`apply_invoice` = `True`。
- 重试语义分层：
  - **参数 JSON 非法 / handler 主动返回 error**（业务/校验错误）→ **不重试**，直接返回 error。
  - **handler 抛异常**（多为基础设施故障）→ 仅对**非副作用**工具按指数退避重试最多 `max_retries`（默认 2）次：`delay = min(retry_base_delay * 2**attempt, retry_max_delay)`（`:259`）。
  - **副作用工具（动钱/拉人）失败不自动重试**（`max_attempts = 1 if side_effect else self.max_retries + 1`，`:251`），避免"重试触发第二次退款"——R2 幂等补齐前这是更安全的默认。
  - 重试耗尽仍失败 → 转 error 型结果（与 R6 同源）。

`__init__` 新增 `max_retries=2 / retry_base_delay=0.1 / retry_max_delay=1.0`（`tool_executor.py:186-188`）。

## §3 异步化

**问题**：全链路要求 `async`（`async SSE → AsyncOpenAI → LangGraph astream → 原生 AsyncSession`），但 `ToolExecutor.run` 原是同步 `def`，在事件循环里阻塞线程，违背"全异步"架构约束。

**优化**：

- `run` / `_execute_with_retry` / `_execute_one` 全部改为 `async def`（`tool_executor.py:201,233,278`）。
- 退避等待用 `await asyncio.sleep(delay)`（`:264`）而非 `time.sleep`，让出事件循环。
- 业务处理器（domain 服务）目前是**同步**实现（mock / 同步 DAO），用 `await asyncio.to_thread(handler, args, state)`（`:287`）丢到线程池执行，避免阻塞主循环——这正是 `plans/full-async-plan.md` 的 M1 桥接思路。
- `agent_node.py:73` 对应改为 `tool_messages = await self.tool_executor.run(...)`。

> 说明：当前是"异步外壳 + 线程桥接同步 handler"的折中。真·延迟最优要等工具客户端（订单/物流/向量）全部原生 `awaitable`（httpx.AsyncClient / asyncpg），届时去掉 `to_thread` 即可，调用方无需改动。

## §4 退款二次确认（R1）

**问题**：`request_refund` 是动钱动作，原 `_request_refund`（`tool_executor.py:388`）只校验 `order_id` 非空就直接调 `refund_service.request_refund` 执行，模型一次误调用就真退款，没有任何确认闸门。

**优化**（`tool_executor.py:388` `_request_refund` + `:428` `_refund_confirmed`）：

1. 工具 schema 新增 `confirm: boolean` 参数，并在 description 写明两步协议——首次 `confirm` 默认 `false`，工具只返回确认提示不执行；用户确认后模型以 `confirm=true` 二次调用才真正办理。
2. 执行前判定 `confirmed = args.get("confirm") is True or self._refund_confirmed(state, order_id, refund_type)`：
   - **未确认** → 返回 `kind="confirmation"` 的 `ToolExecutionResult`（`state.py` 的 `ToolExecutionResult.kind` 新增 `"confirmation"` 字面量），**绝不调用 domain 服务**，并直接写 `state.reply` 保证确认提示直达用户（与 `create_handoff` 同思路，省一次多余 LLM 调用）。
   - **已确认** → 执行，并把痕迹 `refund:{order_id}:{refund_type}` 写入 `state.confirmed_slots` 供审计/跨轮去重。

协同：R1 拦下首次调用（零副作用）→ 用户确认触发 `confirm=true` → R2 保证重复确认也只生成一次受理单。

## §5 幂等（R2）

**问题**：`RefundService.request_refund` 与 `HandoffService.create_handoff` 原用 `self._counter += 1` 每次生成新单号，重复调用（重试、二次确认、并行触发）会**重复退款 / 重复建人工单**。

**优化**（`domain.py`）：

- `RefundService` 增加 `_by_key: dict[tuple[str, str], RefundResult]`（`:107`），以 `(order_id, refund_type)` 去重，命中即返回原 `refund_id`（`:111-112`）。
- `HandoffService` 增加 `_by_session: dict[str, HandoffResult]`（`:80`），以 `session_id` 去重，同会话重复转人工返回原 `ticket_id`（`:83-84`）。

真实后端此处应换成 `idempotency_key`（客户端幂等键）+ 数据库唯一约束；mock 层用内存字典等价演示同一语义。

## §6 统一参数 schema 校验（R5）

**问题**：此前每个 `_handler` 各自做 `if not order_id: return error` 之类的非空判断，没有统一层。缺点：① 校验逻辑散落、易遗漏（新增工具容易忘写）；② 类型错误（模型把 `order_id` 传成数字、把 `confirm` 传成字符串）要等 handler 内部逻辑跑飞才暴露；③ 出错信息不统一。

**优化**（`tool_executor.py` 新增 `_validate_tool_args`，在 `_execute_with_retry` 的 JSON 解析之后、调用 handler 之前拦截）：

- 直接复用 `TOOLS` 注册表里已有的 `parameters`（`required` / `properties.type` / `properties.enum`），**零新增 schema**——注册表本就是 function calling 的结构化定义，校验与工具定义同源，不会漂移。
- 校验项：① 必填字段存在且非空；② 显式传入参数的类型 / 枚举合法。
- 必填缺失时回退 `state.slots`（`_value(field) = args.get(f) or state.slots.get(f)`），与 handler 内 `_request_refund` 等 `args.get(...) or state.slots.get(...)` 的取值逻辑一致——多轮继承的订单号仍算齐全。
- 仅校验「显式传入的参数」的类型/枚举，`state.slots` 为可信内部状态不再重复校验。
- 校验失败 → 直接返回 `kind="error"` 的 `ToolExecutionResult`，**不进 handler、不重试**（属调用方错误，与 JSON 格式错误同处理）；handler 里的非空兜底判断保留为纵深防御，不删。

```python
# _execute_with_retry：JSON 解析后、handler 前
validation_err = self._validate_tool_args(name, args, state)
if validation_err:
    logger.warning("[tool] validation failed for %s: %s", name, validation_err)
    return ToolExecutionResult(kind="error", user_facing_summary=validation_err)
```

校验覆盖全部 7 个工具：query_order / query_logistics / request_refund / modify_address / apply_invoice 的 `order_id`（及 `new_address` / `invoice_title`）必填；request_refund 的 `refund_type` 枚举；rag_retrieve 的 `query` 必填；create_handoff 无必填。

## 预期收益

- **不翻车**：单工具异常不再中断整批（R6），瞬时故障自愈（重试），全异步不阻塞事件循环（§3）。
- **不动错钱**：退款必经二次确认（R1）+ 幂等（R2），误调用 / 重试 / 并行都不会重复退款或重复建人工单。
- **可观测**：每个工具结果都带 `kind`，配合 `[tid]` 日志与 `observe_tool`/`observe_handoff` 指标，故障可定位。
- **可演进**：`side_effect` 标记与 `confirm` 协议是统一护栏，未来加新风险工具只需在注册表标注，无需改执行框架。

## 风险工具防护（当前已生效）

- **风险工具防护**：风险工具 = 带**副作用**、不可逆或动真资源的工具——`request_refund`（动钱）/ `create_handoff`（拉真人）/ `modify_address` / `apply_invoice`，注册表统一标 `side_effect=True` 与非副作用工具（查订单/查物流/RAG 检索）区分。对风险工具施加统一护栏：① 二次确认（R1，`confirm` 闸门，首次调用零副作用）；② 幂等（R2，同键去重，重复调用只生成一单）；③ 统一参数校验（R5，缺必填/类型错/枚举非法直接拦）；④ 失败隔离且不自动重试（`side_effect` 工具 `max_attempts=1`，杜绝"重试触发第二次退款"）；⑤ 强审计（R7，`event_log` 记录每次风险动作带 `trace_id`）。该防护体系是**当前已实现的行为**，非待办，详见 §1–§6 已落地改造。

## 扩展优化项

- **R3 多工具并行执行**：`run()` 当前仍是串行循环，`tool编排.md §5.5` 已设计 `asyncio.gather` 并发方案。这是**性能扩展优化**而非正确性缺口——串行已能正确工作，且 R1+R2 保证即便并行时风险动作（退款/转人工）也不会重复触发；仅在出现明显延迟/吞吐瓶颈时再落地，落地时需为风险工具加"禁止并行触发"闸门（R1/R2 各自独立生效已能兜底）。
- **R4 意图作用域工具集**：`agent_node` 当前 `build_tool_schemas()` 下发全量工具，应只给当前意图授权可用工具，缩小模型误调风险工具面。

  **决策（扩展优化，非遗漏）**：当前工具数量少（7 个），按意图裁剪的收益有限，且 R1/R2/R5/R6 已对风险工具做足纵深防御，"误调"后果可控，故**本期不做 R4 拆分**。后续工具规模扩大、出现明显误调或延迟问题时，再引入 **MULTIAGENT 架构**作为扩展优化——按意图把工具与策略拆分到多个专职子 Agent（如订单 Agent / 退款 Agent / 物流 Agent），由 Orchestrator 路由分发，天然实现"意图作用域工具集"且职责更清晰；届时 R4 将随 MULTIAGENT 一并落地，而非在单 Agent 内做工具过滤。详见 `tool编排.md` §4.1。
- **R7 独立审计表（可选增强）**：统一 `event_log` 已记录每个请求的 `tool_result` 事件，风险动作（`request_refund` / `create_handoff` 等）连同 `trace_id` 一起进轨迹，配合 `[tid]` 日志与 `observe_tool`/`observe_handoff` 指标，已具备"可追溯、可对账"的强审计能力（见 §预期收益）。原 `ToolCall`/`HandoffRecord` 独立审计表属**可选增强**——若要更强对账（如按 `refund_id`/`ticket_id` 落独立表做财务级核对），可在 DAO 层重建。

## 额外问题解决

### 问题现象

用户说「退掉」（想退款但缺订单号），系统却回了一段**内部 monologue** 而不是正常的索要订单号：

```
用户想要退款，但缺少订单号信息。我需要向用户询问订单号才能继续处理退款请求。
当前上下文不足以完成操作，无需调用工具，直接结束本节点，由回复节点向用户索要订单号。
```

核心异常：面向用户的回复里出现了「本节点 / 回复节点 / 直接结束本节点」这类**系统内部架构术语与决策旁白**，用户看得莫名其妙。

### 根因（三处叠加）

1. **路由漏匹配**（`routing.py:221` 原 `_rule_matches`）：`matched = keyword_hit or emotion_hit`，**完全忽略 `action_keywords`**。「退掉」只写在退款规则的 `action_keywords` 里、不含任何基础关键词（退款/退货），规则不命中 → 退化到 LLM 兜底 → 兜底未设 `needs_clarification` → 路由到 `agent_process`（而非澄清节点）。
2. **调度节点把自言自语当回复**（`agent_node.py` 原代码 `state.reply = content`）：LLM 在调度节点输出的内部决策旁白被原样写进用户可见的 `state.reply`。
3. **提示词埋了内部术语**（`system.py` 原 agent 提示）：直白写「你是【调度节点】…直接结束本节点…由回复节点生成」，模型照着念，成了那句泄漏文本。

即问题在**调度/回复职责没隔离 + 路由漏匹配让本该澄清的请求错进调度节点**。

### 修复

1. **路由修复**（`routing.py:221`）：`matched = keyword_hit or emotion_hit or action_hit`，`action_keywords` 也能独立命中。「退掉 / 直接退 / 帮退一下 / 申请退 / 后悔了」等强动作词全部可命中退款规则。
2. **调度节点不再产出用户文本**（`agent_node.py`）：删除 `state.reply = content`。调度节点只做决策/工具调用，其 `content` 是内部旁白，**绝不**作为 `state.reply` 输出；用户回复统一由 `response_generator` 基于上下文生成（即便本轮无需工具，也由下游产出干净话术）。
3. **提示词去术语**（`system.py` `build_agent_system_prompt`）：去掉「调度节点 / 本节点 / 回复节点」等术语，明确「只做工具调用决策，绝不向用户输出任何文字、不输出分析推理，最终回复由系统后续环节统一生成」。
