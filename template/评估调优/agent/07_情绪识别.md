# 07 情绪识别（最新版）

## 设计定位（只保留两点）

1. **识别**：规则 + LLM 双路识别用户情绪（`neutral` / `positive` / `negative`）。
2. **回复语气**：在回复 / 澄清提示词中按情绪调整语气；`negative` 时先共情安抚再回答。

情绪**不再参与路由决策**——即不再因负向情绪直接路由到投诉 / 转人工（原 `emotion_escalation` 仅为标签，从未真正转人工，属冗余第三应用，已移除）。

## 1. 情绪识别（规则 + LLM 双路）

- 数据模型：`EmotionState(primary, confidence)`，`EmotionLabel = Literal["neutral","positive","negative"]`；`IntentResult` 与 `ConversationState` 均带 `emotion` 字段。
- 规则路（`routing.py` `_detect_emotion`）：匹配 `config/intent_rules.yml` 的 `emotion_keywords`；带**否定词消歧**（`_emotion_hit` + `_NEGATION_PREFIXES`，如「不生气」「不太满意」不误判为负面）。
- LLM 路（`llm_fallback.py`）：意图分类时一并输出 `emotion`；提示词 `build_llm_intent_user_prompt` 要求输出。
- 合并（`_merge_emotion`）：规则非 neutral 优先（确定性强）；规则 neutral 取 LLM（补无关键词盲区）；都 neutral → neutral。
- 负面记忆回退（`routing.py`）：本轮合并后仍为 neutral 但上一轮为 negative → 沿用 negative（轻微衰减 `confidence`），避免多轮愤怒断档。

## 2. 回复语气（提示词驱动，非硬编码）

情绪只作为「语气信号」透传给生成节点，不改路由：

- `prompts/system.py` `_render_base_context`：仅当 `emotion.primary != "neutral"` 时透传「用户情绪：negative」到提示词（neutral 不携带信号）。
- 回复提示词（`build_response_system_prompt`）新增【情绪与语气】段：
  - `negative`（用户不满 / 着急）→ 开头先简短共情安抚，再基于工具结果回答，语气温和耐心；
  - `positive`（感谢 / 满意）→ 自然呼应其正面情绪；
  - 未提供情绪字段（中性）→ 保持专业友好的常规语气。
- 澄清提示词（`build_clarification_system_prompt`）：当 `negative` 时追加「先简短共情安抚，再自然地提出追问，语气温和耐心」。

## 已移除（旧版 → 新版）

- 确定性安抚前缀：`response.py` `_apply_empathy_prefix` 与 `config/response_prompts.yml` 的 `empathy_prefix.negative` 已删除。安抚改为由 LLM 在提示词引导下生成，更自然、不冗余、不会与事实答案黏连。
- 情绪驱动路由：`intent_rules.yml` 投诉规则 `emotion: negative`、`handoff_reason_emotion: emotion_escalation`，以及 `routing.py` `_rule_matches` 的「路由情绪匹配器 `emotion_hit`」已删除。负向情绪不再直接命中意图；纯情绪投诉（如「你们这服务我真服了」）仍由 LLM 兜底识别为 `complaint`。
  - 注意：`routing.py` 中仍存在的 `_emotion_hit` 是 `_detect_emotion`（规则路情绪识别）内的**否定词消歧辅助函数**，与已删除的「路由情绪匹配器」同名但职责不同——前者只判断「关键词是否未被否定前缀抵消」，后者曾把 `emotion` 字段当作意图命中条件。二者不要混淆。

## 配置

- `config/intent_rules.yml`：`emotion_keywords`（negative / positive 词表）。
- `config/response_prompts.yml`：`response_prompts`（回复示例，供 LLM 注入提示词）。

- `tests/test_dialog_services.py` `TestEmotionSoothing`：校验回复 / 澄清提示词含「情绪」与「安抚」指令（替换原硬编码前缀用例）。

## 前端

- `frontend/src/types/chat.ts`：`emotion?: EmotionState` 已进会话状态类型，状态面板可展示情绪。

## 端到端效果示例

用户输入「气死，查一下订单」（未给单号）：

- `route`：命中 `order_query`，因本轮未显式给单号 → `needs_clarification = True`（见 06_澄清转人工 的继承订单澄清规则），`emotion = negative`。
- 澄清节点：提示词含「用户情绪：negative」+「先共情安抚再追问」→ LLM 产出如「很抱歉让你着急了，请提供订单号，我来帮你查询订单状态。」
- 若已带单号（如「气死，查一下 A1001」）：直接查工具，最终回复提示词引导下先安抚再答。
