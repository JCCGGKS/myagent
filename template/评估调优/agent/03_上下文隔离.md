# 03 上下文隔离调优

客服 Agent 的提示词构造按「节点职责」隔离状态字段：每个助手（调度/澄清/最终回复）只拿到对它执行有帮助的字段，其余执行侧元数据一律不透传。核心收益是 **token 更省、噪声更低、新增字段默认隔离不泄露**。

## 问题现象

- 提示词曾直接读 `state` 的任意字段内联拼装，没有「可见字段」边界；状态对象上的 `session_id` / `user_id` / `channel` / `action_history` / `running_summary` / `recent_messages` / `pending_intents` / 各类计数器 / `reply` 等执行侧字段会随提示词一起喂给 LLM。
- 三个对话节点（agent / clarification / response）拿到**同一份**上下文，但各自需要的字段不同：调度节点不需要情绪，回复节点不需要「当前动作 / 缺失槽位」。
- 工具信息一度被**重复**注入 prompt：既在 `tools=` API 参数里有结构化 schema，又在系统提示里写了一份 name:description 文本目录。

## 根因

1. 提示词构造直接读 `state` 任意字段（`_build_base_system_prompt` 内联挑选），没有显式的「提示词可见字段」边界；新增字段会默认泄露。
2. 三个对话节点共用一份白名单，未按节点职责区分。
3. `build_agent_system_prompt` 曾把工具文本目录写进 prompt，与 function calling 的 `tools=` 参数重复。

## 优化措施

### 1. 按节点隔离的状态字段白名单

`app/business/prompts/system.py` 定义三份**独立**白名单，每个助手只取对它执行有帮助的字段：

- `AGENT_FIELDS`：`当前意图 / 阶段 / 已填槽位 / 缺失槽位 / 当前动作`
  —— 调度节点决策调工具用，**不含情绪**（它不生成面向用户文本，不需要定语气）。
- `CLARIFICATION_FIELDS`：`当前意图 / 阶段 / 已填槽位 / 缺失槽位 / 当前动作 / 情绪`
  —— 澄清节点生成追问用，借情绪定追问语气。
- `RESPONSE_FIELDS`：`当前意图 / 阶段 / 已填槽位 / 已确认槽位 / 情绪 / 是否需要澄清`
  —— 最终回复节点组织友好回复用，**不含当前动作 / 缺失槽位**（回复是收尾，动作已完成、信息已齐）。

配套机制：

- `build_prompt_context(state, fields)`：按白名单抽取**非空**字段，得到只属于该节点的「提示词可见」切片；白名单之外的字段一律不进入。
- `_render_base_context(ctx)`：只渲染切片里实际存在的字段（意图/阶段/槽位/情绪等），切片没有的字段自然不出现。

### 2. 工具信息只经 `tools=` 参数下发（不进 prompt）

- 回退 prompt 内的文本工具目录；可调用工具的 JSON Schema 由 `app/business/agent/graph.py:147` 装配 `AgentNodeService` 时 `tools=build_tool_schemas()` 注册，经 `agent_node._call_llm`（`agent_node.py:109` `tools=self.tools`）→ `app/utils/llm.py:77` `tools=tools if tools else None` 下发——这是 function calling 的结构化通道。
- `build_agent_system_prompt` 只负责告诉模型「需要信息就调工具，够了就结束」，prompt 不重复罗列工具。
- 配套精简 `app/business/tools/tool_calling.md`：删除决策清单 / ReAct 完整模板 / 对照速记等重复小节，突出「工具注入说明」一节。

### 3. 意图识别提示词同样「借用状态对象」隔离

- `app/business/prompts/intent.py` 的 `build_llm_intent_user_prompt` 改收 `state`，仅取 `state.current_sub_intent` 作「上一轮子意图」多轮上下文；`LLMIntentFallbackService.classify` / `IntentRouterService._route_with_llm_fallback` 相应改收 `state`。同样不整份透传状态。

### 4. 自审优化（消除重复 / 噪声）

- 修正澄清提示词**重复输出「缺失槽位」**：公共渲染层 `_render_base_context` 已渲染该字段，builder 又追加了一次，现已删除重复块。
- `当前动作` 渲染收口到 `_render_base_context`，去重 agent / clarification 两个 builder。
- **中性情绪不进入提示词**：`emotion.primary == "neutral"` 不携带信号，渲染处加条件跳过。
- 空值判定提为模块常量 `_EMPTY_VALUES = (None, "", {}, [])`。

### 5. 上下文隔离 vs. 对话上下文，边界分清

- `running_summary` / `recent_messages` 是「对话上下文」（用户/助手的真实对话内容），由 `agent_node._build_messages` / `response._build_messages` 作为**独立消息**注入，与「状态字段隔离」是两回事。
- 前者是合法上下文、不算泄漏；后者是执行侧元数据、必须隔离。两者不混为一谈。

## 涉及文件

- `app/business/prompts/system.py` — 三份白名单 `AGENT_FIELDS` / `CLARIFICATION_FIELDS` / `RESPONSE_FIELDS` + `build_prompt_context` + `_render_base_context`
- `app/business/prompts/intent.py` — 意图提示词从 `state` 借用上下文（只取上一轮子意图）
- `app/business/prompts/README.md` — 同步说明「按节点隔离」原则
- `app/business/agent/agent_node.py` — `build_agent_system_prompt(state)` 不再传工具目录
- `app/business/tools/tool_calling.md` — 精简 + 「工具注入说明」一节

## 验证

- `pytest tests/` 通过（`test_dialog_services.py` / `test_routing_services.py` / `test_tool_executor.py`）。
- 功能断言（构造 `ConversationState` 调三节点 builder）：
  - 澄清提示词中「缺失槽位」**只出现一次**（重复已修）；
  - agent 提示词**不含**「用户情绪」（白名单无 emotion）；
  - 中性情绪的 response 提示词**不含**「用户情绪」（跳过 neutral）；
  - 各节点提示词均**不含** `session_id` / `user_id` / `channel` / `action_history` 等执行侧字段。

## 预期收益

- **token 成本**：每个节点只带自己需要的字段，prompt 更紧凑。
- **噪声降低**：执行侧元数据不干扰 LLM 的决策 / 生成。
- **可维护**：新增状态字段默认隔离，只有被加入对应 `*_FIELDS` 白名单才会透传给某个节点，不会无意泄露；白名单即「可见字段契约」，一眼可审。
