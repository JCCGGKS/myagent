# 05 澄清转人工兜底

聚焦「澄清轮次累积后强制转人工」的问题、为何兜底必须保留、以及如何收敛触发条件。意图路由 / 槽位继承 / 多意图等上游成因属后续讨论，本文不展开。

## 问题现象（来自 logs/01_test.md）

对话原文（用户 / 机器人）：

```
同时查询订单详情和物流          → 请提供订单号，我来帮你查询物流进度。
A1001                          → 订单 A1001 当前物流状态为 运输中，最近一条记录是 2026-07-03 10:00 派送中。
订单详情                       → 请问您想查询订单A1001的哪方面状态呢？是物流进度还是订单处理状态？
订单详情                       → 已为你转人工客服，服务单号 H1004。人工客服会基于当前会话上下文继续处理。
```

核心异常在最后一轮：用户再喊「订单详情」，机器人直接转人工 `H1004`。而 `order_query.query_status(A1001)` 在 `orders.json` 里**完全可答**（已发货 / 智能客服机器人 Pro / 1999），无任何理由升级。

其触发链条是：前面累计了多轮澄清（缺订单号的槽位澄清 + 一轮意图澄清），计数滚到阈值后，对一个槽位齐全、工具可用的普通查询做了最差兜底升级。即**「能答却转人工」**——这是本文要解决的问题。

> 注：中间澄清轮次（如「物流进度还是订单处理状态」）的文案不在当前 `clarification_prompts.yml` 里，说明该 log 可能来自旧构建或手工编排；但「澄清计数累积 → 误转人工」这一机制在**当前代码中依然存在**。

## 根因

转人工阈值触发过粗（`HandoffClarificationPolicy.decide`，`routing.py:394-404`）：

- **会话全局计数、无衰减**：`intent_clarification_count` 与 `slot_clarification_count` 整个 session 累加，达到 3 就强转，历史包袱会滚雪球。
- **槽位澄清也计入转人工**：缺订单号是常态，本不该升级，却和「真·听不懂」混为一谈。
- **无自助护栏**：当当前意图槽位齐全且存在可用工具时（如已知 A1001 的订单详情），本应直接作答，却因历史计数被强制升级。

即问题在**触发条件太粗**，不在「有没有出口」。

> 说明：上文澄清轮次本身的部分成因（意图路由、槽位继承等）属后续优化范畴，本文不展开。

## 为什么一定要保留「转人工兜底」

评估过「直接删掉阈值转人工」，结论：**必须保留**。

- **防死循环安全网**：没有它，一个反复发无法解析消息的用户会永远卡在澄清循环里（既答不了、又转不了人工），是真实的 UX 陷阱。
- **对「真·听不懂 / 反复答非所问」的用户，转人工本身是对的处置**：系统确实无法理解诉求时，交人工是合理收口。
- **删除 = 主动放弃兜底**，风险高且无收益。

因此正确治法是**收窄触发**，而非移除出口：让兜底只在「真的卡住」时才生效。

## 优化措施（本轮落地）

仅改 `HandoffClarificationPolicy.decide`（`routing.py:367-407`），三处修改 + 删一个死字段：

1. **仅 `intent_clarification_count` 计入转人工**：槽位澄清（缺订单号）是常态，不该升级；阈值只认「真·听不懂」的 intent 澄清。
2. **解析成功即清零计数**：新增 `if not state.needs_clarification: 两个计数归零`。阈值从「会话累计」变为「连续澄清失败次数」，消除历史包袱滚雪球。
3. **自助护栏**：新增 `self_service = not state.missing_slots and current_main_intent in {order_query, logistics, after_sale_refund}`；仅当 `stuck and not self_service and intent_count >= threshold` 才强制转人工。槽位已齐、工具可用的自助意图（如已知 A1001 的订单详情）**绝不**强制升级。
4. **删除 `slot_clarification_count` 死字段**：该字段记的是「系统追问槽位次数」，而追问根因是**用户没给订单号**（用户侧行为），对「系统哪里要改」没有指向性，不属于系统侧可观测指标；改后它又不参与任何决策，属纯装饰字段，按「不留死代码」原则清掉（`state.py:67` 删定义、`routing.py` 删 `+=1` 与日志引用、`tests/test_customer_service_agent.py:53,76` 删 `=0`、顺手清 `stage.md:26` 与 `template/07_系统架构.md:463` 两处文档债）。

行为对照：

| 场景 | 优化前 | 优化后 |
| --- | --- | --- |
| Turn 4：A1001 订单详情、计数已高 | 强制转人工 | `self_service` 命中 → 直接答（agent_process） |
| 连续 3 次 unrecognize（真听不懂） | 转人工 | 转人工（不变，兜底保留） |
| 缺单号被追问 3 次后给单号 | 可能误转 | 给单号 → 清零计数 → 正常作答 |
| 显式「转人工」/ 投诉 handoff | 转人工 | 转人工（不变） |

## 预期收益

- **消除「能答却转人工」**：槽位齐全且工具可用的查询，即便历史澄清计数偏高也绝不强制升级（直接命中问题现象）。
- **兜底仍保活**：连续 3 次真·听不懂仍会转人工，防死循环安全网不丢。
- **计数语义更合理**：从「会话累计」改为「连续澄清失败」，正常多轮咨询不会误触阈值。
- **代码更干净**：删掉一个无决策价值、记用户侧行为的死字段。
