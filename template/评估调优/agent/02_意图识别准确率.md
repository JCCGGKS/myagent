# 02 意图识别准确率调优

客服意图识别准确率（规则+LLM）从 **75.3% → 96.5%**，超过 90% 目标；规则-only 同步从 55.9% → 72.6%。评估数据集与评估工具一并重建，使结论可复现。

文档结构：
- **一、优化原理与方案**——背景、问题根因、具体优化手段（为什么改、改什么）；
- **二、优化效果（对比基线）**——优化前后各项指标、按类别水位、残余失败、验证；
- **三、总结**——核心优化手段 + 后续优化方向。

---

## 一、优化原理与方案

### 1.1 背景

- 旧评估数据集（1000 条）引用了已废弃意图（如 `chitchat.greeting`、`logistics.delayed` 当作规则产出），与当前路由层（`app/schema/intent.py` 的权威 `SubIntentCode`）不一致，评估意义失真。
- 评估工具 `run_eval.py` 创建 `ConversationState` 后未注入 `previous_sub_intent`，多轮跟进（`slot_followup`）无法被测试。
- 评估入口 `eval/intent/run_eval.py` 直接调用 `IntentRouterService.route()`，与生产路径一致；对比模式为「规则-only vs 规则+LLM」。

### 1.2 问题现象与根因（优化前）

**问题现象（优化前）**

| 模式 | 准确率 | 命中 |
|---|---|---|
| 规则-only | 55.90% | 559/1000 |
| 规则+LLM | 75.30% | 753/1000（LLM 兜底命中 346 次） |

按样本类别拆解（规则+LLM）：

| 类别 | 样本数 | 规则-only | 规则+LLM |
|---|---|---|---|
| rule_hit | 463 | 100.00% | 98.70% |
| finer_subintent | 221 | 0.00% | 38.46% |
| colloquial | 220 | 0.00% | 53.18% |
| multiturn_followup | 68 | 100.00% | 100.00% |
| unrecognize | 28 | 100.00% | 92.86% |

**根因（247 个失败）**

1. **finer_subintent（136 例）**：LLM 主意图判对，但**塌缩到默认子意图**——`damage_refund` / `lost_package` / `delayed` / `modify_address` / `apply_invoice` / `no_reason_return` / `compensate` 全被打成 `consult_policy` / `not_received` / `query_status`。根因：规则层只产出各主意图的默认子意图，从不产出细子意图。
2. **colloquial（103 例）**：约 52 例「想退 / 后悔了 / 申请退」LLM 给 `request_refund`，金标却保守写 `consult_policy`（**金标歧义**）；约 19 例物流状态查询 LLM 给 `delayed`，金标 `not_received`。
3. **rule_hit（6 例）**：LLM 覆盖把正确规则结果改错——`麻烦配送`→`delayed`、`我想我的订单`→`unrecognize`。根因：`routing.py` 在 `confidence < 0.8` 时用 LLM 覆盖，而 `0.76 / 0.78` 这类高置信规则结果被反噬。
4. **unrecognize（2 例）**：`有人吗` 被 LLM 误判为 `handoff_service`。
5. **评估工具缺陷**：`run_eval.py` 未注入 `previous_sub_intent`，多轮跟进无法测。

### 1.3 优化方案

#### 1. 规则层细分子意图（确定性产出，零 LLM 成本）
`config/intent_rules.yml` 在通用规则前新增 8 条细分子意图规则，命中即返回，避免 LLM 塌缩：

- `order_query.modify_address`：`[改地址, 修改地址]`
- `order_query.apply_invoice`：`[开发票]`
- `logistics.lost_package`：`[丢件, 包裹丢了]`（`needs_order`）
- `logistics.delayed`：`[延迟, 配送太慢]`（`needs_order`）
- `after_sale_refund.no_reason_return`：`[无理由, 七天无理由, 不想要了想退]`
- `after_sale_refund.wrong_goods`：`[错货, 发错, 收到错货]`
- `after_sale_refund.damage_refund`：`[商品损坏, 损坏了, 收到的东西坏了, 坏了要退, 质量问题要退]`
- `complaint.compensate`：`[赔偿, 赔付]`

末尾新增**纯问候兜底规则**（必须置于最后，仅当其它规则均未命中时触发）：
- `unrecognize.unknown`：`[你好, 在吗, 有人吗, hello, 嗨, 在不在, 在么, 随便聊聊]`

`after_sale_refund` 的 `action_keywords` 增加 `后悔了` / `申请退`（「想退」因与前缀「我想」+「退款/退货」拼接产生 `我想退款` 误判，故不加）。

#### 2. 金标重标 + 测试集重建
`eval/intent/gen_cases.py`（新增生成器）：金标对齐 `app/schema/intent.py` 权威意图空间，覆盖五类样本（rule_hit / finer_subintent / colloquial / multiturn_followup / unrecognize）。口语「想退 / 后悔了 / 申请退」期望子意图定为 `request_refund`（与 LLM / 自然语义一致）。旧数据集备份为 `intent_single_step_cases.json.bak`。

#### 3. 收敛 LLM 覆盖阈值
`app/business/intent/routing.py`：覆盖判定 `confidence < 0.8` → `< 0.7`。依据见下方「覆盖阈值调优方法」——核心证据是规则命中的置信度是**离散跳变值**，唯一低于 0.8 的是 `0.76 / 0.78` 共 23 例，其中规则对 20、规则错 3；原 `0.8` 阈值把它们全部送 LLM 覆盖后 `rule_hit` 从 100% 掉到 98.7%（6 例被正确规则被改错），即该 band 覆盖**净亏**，故关闭对它的覆盖。

#### 4. LLM 提示词收紧
`app/business/prompts/intent.py`（`build_llm_intent_user_prompt`）：
- 物流状态查询无「丢件 / 延迟 / 太慢」词时默认 `not_received`；明确 `丢件→lost_package`、`延迟/太慢→delayed`。
- 退款售后按细节区分 `consult_policy / request_refund / damage_refund / no_reason_return / wrong_goods`，仅说「质量 / 坏了 / 有问题」未明确要退归 `consult_policy`。
- 补 few-shot 示例（改地址 / 开发票 / 丢件 / 慢 / 想退 / 损坏 / 无理由 / 错货 / 赔偿 / 有人吗）。
- 强化问候守卫（`有人吗 / 在吗` → `unrecognize.unknown`）。

#### 5. 修复评估工具（多轮可测）
`eval/intent/run_eval.py`：每个 case 按 `previous_sub_intent` 注入 `state.current_main_intent` / `current_sub_intent`，使 `slot_followup` 路由生效；对比报告新增按类别拆解表。

### 1.4 覆盖阈值调优方法

覆盖阈值（`routing.py` 中 `intent.confidence < T` 时用 LLM 结果覆盖规则结果）是一个**净收益开关**：调高 T 让更多规则结果送 LLM 覆盖（可能修对错的、也可能改对错的对）；调低 T 保留更多规则原结果。

**调法（据效果扫描）**

对每条**规则命中**的 case 记录三元组 `(规则置信度, 规则是否正确, 覆盖后 LLM 是否正确)`，扫描候选 T 算净正确数：

```
net(T) = Σ_{conf≥T} 规则正确数          # 不覆盖，用规则
       + Σ_{conf<T} 覆盖后正确数        # 覆盖，用 LLM（无效则退回规则）
```

取使 `net(T)` 最大的 T。即：**某置信带覆盖净正才开着，净负就关掉**。

**本次数据结论**

规则命中的置信度分布（实际只有这些值）：

```
{0.76:9, 0.78:14, 0.8:222, 0.82:4, 0.88:123, 0.9:251, 0.95:39, 0.99:6, 0.92:2}
```

- **< 0.7 的 band 为空** → 当前 T=0.7 实际**从不触发覆盖**（所有规则结果 conf≥0.76）。
- 唯一会被覆盖的 band 是 `0.76 / 0.78` 共 23 例：规则对 20、规则错 3。
- 原 T=0.8 时这 23 例被覆盖 → `rule_hit` 100% → 98.7%（6 例正确规则被 LLM 改错），即该 band 覆盖**净亏**（即便 LLM 可能救回那 3 个错的，也抵不过改错的 6 个）。

结论：覆盖在唯一 eligible 的 band 里净亏，应关闭对它的覆盖 → T 取 **≤ 0.76**（当前 0.7 落在最优区，`rule_hit` 回到 100%）。

**一键自动扫描（已落地）**

手动分桶太慢，已把上面的 `net(T)` 扫描固化成命令：

```bash
python3 eval/intent/run_eval.py --sweep-threshold
```

实现要点（见 `eval/intent/run_eval.py` 的 `run_sweep_threshold`）：

- 扫描候选 `T ∈ {0.6, 0.65, 0.7, 0.75, 0.8}`（可用参数 `candidates` 覆盖）。
- 采用 `net(T)` 分解、避免对每档阈值重复调 LLM：
  1. 一遍 `use_llm=False` 得到纯规则路径结果（`rule` / `slot_followup` / `fallback`）；
  2. 每个样本仅调一次 LLM `classify` 得到覆盖候选结果（进一步只对「无规则命中」和「规则命中且 conf < max_T」的样本调 LLM，conf≥max_T 的规则结果任意 T≤max_T 都不会被覆盖，无需 LLM 候选）；
  3. 按 T 解析：rule 命中且 conf≥T → 保留规则；conf<T → 取 LLM 覆盖；非 rule 路径（slot_followup / 无规则走 LLM 兜底）与 T 无关直接采用。
- 该分解与 `IntentRouterService.route(override_threshold=T)` 的真实输出逐样本一致。
- 输出每档净准确率表，并给出**最优带**（所有达到最高准确率的 T）与**推荐值**：优先沿用当前默认 `0.7`（若落在最优带内，避免无谓改动），否则取最优带中最小 T（T 越小、被 LLM 覆盖的规则命中越少，运行时 LLM 调用成本越低）。

最近一次扫描结果（示例，LLM 非确定性会有 ±0.x% 波动）：

| 阈值 T | 准确率 | 命中 |
|---|---|---|
| 0.60 | 96.30% | 963 |
| 0.65 | 96.30% | 963 |
| 0.70 | 96.30% | 963 |
| 0.75 | 96.30% | 963 |
| 0.80 | 96.20% | 962 |

最优带为 `0.60–0.75`（均 96.30%，唯一低于 0.8 的覆盖 band 净亏 1 例），推荐沿用 `0.7`——与「收敛覆盖阈值」措施结论一致。

**何时该再调**

改了 `intent_rules.yml`（新增 / 调整规则会让更多 case 变高置信）或改了提示词后，**重跑评估并自动扫描阈值**：

1. `python3 eval/intent/gen_cases.py --count 1000` + `python3 eval/intent/run_eval.py`（both modes）。
2. `python3 eval/intent/run_eval.py --sweep-threshold` 一键扫出最优带与推荐 T，无需手工分桶。
3. 若推荐 T 与当前默认不同，按输出提示写入 `routing.py` 的 `override_threshold` 默认值或生产代码显式传入。

**更稳的长期方案（可选）**

当前置信度是离散跳变值，阈值只能在 0.76/0.78 一个点起作用，较脆。两条更干净的路：

- **按信号强弱而非阈值决定覆盖**：规则命中了显式关键词（`keyword_hit`，高精度）就永不覆盖；只有靠情绪 / 弱信号命中的才允许 LLM 覆盖。比单一阈值更可解释。
- **让 `confidence_without_order` 真正低于覆盖线**：把 order / logistics 无订单时的置信度设到 `0.6~0.65`，使「无订单的不确定规则」确实进入覆盖 band，同时把 T 固定在 0.7——覆盖只发生在「规则自己也不确定」时，语义更清晰。

### 1.5 涉及文件

- `config/intent_rules.yml` — 8 条细分子意图规则 + 纯问候兜底规则 + after_sale action_keywords
- `app/business/intent/routing.py` — 覆盖阈值 `0.8 → 0.7`；`from_env(use_llm, override_threshold)` 与 `__init__` 参数化 `override_threshold`（默认 0.7），`route()` 改用 `self.override_threshold` 判定覆盖
- `app/business/prompts/intent.py` — 子意图消歧 + few-shot + 问候守卫
- `eval/intent/gen_cases.py` — 新增测试集生成器（对齐权威意图空间）
- `eval/intent/run_eval.py` — 注入 `previous_sub_intent` 支持多轮 + 按类别对比表 + `--sweep-threshold` 一键扫描最优覆盖阈值
- `eval/intent/intent_single_step_cases.json` — 重新生成（约 1000 条，旧文件备份 `.bak`）

## 二、优化效果（对比基线）

### 2.1 总体指标对比

| 模式 | 优化前 | 优化后 |
|---|---|---|
| 规则-only | 55.90% | **72.60%** |
| 规则+LLM | 75.30% | **96.50%** |

### 2.2 按类别拆解（规则+LLM）

| 类别 | 样本数 | 优化前 | 优化后 |
|---|---|---|---|
| rule_hit | 463 | 98.70% | **100.00%** |
| finer_subintent | 221 | 38.46% | **93.67%** |
| colloquial | 220 | 53.18% | **90.45%** |
| multiturn_followup | 68 | 100.00% | **100.00%** |
| unrecognize | 28 | 92.86% | **100.00%** |

→ 细子意图塌缩（`finer_subintent` 38%→94%）、口语歧义（`colloquial` 53%→90%）是主要拉升来源；`rule_hit` 回到 100%（LLM 不再反噬正确规则）。

### 2.3 残余问题分析（35 例，约 3.5%）

基本是**真实意图边界歧义**，非模型错误：

- 14 例「我的东西发出来没有」→ LLM 判 `order_query`（已发货 / 未发货语义），金标写 `logistics`——LLM 更贴近「发货」语义。
- 14 例「物流更新很慢」→ LLM 判 `not_received`，金标 `delayed`——「很慢」未被提示词纳入 `delayed` 触发词。
- 其余零星 1 例。

### 2.4 验证

- `pytest tests/`：**53 passed**（全部通过，无回归）。路由测试 10 passed（注意：`RoutingServicesTestCase` 类名无 `Test` 前缀，需 `-o python_classes="*TestCase"` 才能被 pytest 默认规则收集，历史遗留）。
- 评估流程重跑：
  - `python3 eval/intent/gen_cases.py --count 1000` 重新生成测试集
  - `python3 eval/intent/run_eval.py` 完整对比（规则-only + 规则+LLM + 对比报告）

### 2.5 预期收益

- 规则+LLM 准确率 **75.3% → 96.5%**，超过 90% 目标。
- 规则-only **55.9% → 72.6%**：细分子意图规则确定性产出，不依赖 LLM，也降低线上 LLM 调用量与覆盖改错风险。
- 评估数据集与工具可复现：后续改规则 / 提示词后，跑 `gen_cases.py` + `run_eval.py` 即可回归对比。

## 三、总结

### 3.1 核心优化手段

本次优化围绕一个核心原则：**能确定解决的放规则层（零 LLM 成本、可复现），LLM 只补规则够不到的口语/歧义，且不要让 LLM 反向破坏已正确的规则结果**。四杠杆：

1. **规则层细分（主抓手）**：把 `modify_address / apply_invoice / lost_package / delayed / no_reason_return / wrong_goods / damage_refund / compensate` 等细子意图下沉为确定性规则，单点准确率从塌缩态（38%）拉到 94%，且不增加 LLM 负担。
2. **金标对齐权威意图空间**：新建 `gen_cases.py` 让测试集与 `app/schema/intent.py` 一致，并修正「想退/后悔了」类口语的金标歧义（→ `request_refund`），使评估反映真实能力而非陈旧标签。
3. **收敛 LLM 覆盖阈值**：`0.8 → 0.7`，关掉对高置信规则结果的覆盖，消除「LLM 改错正确规则」的反噬（rule_hit 回到 100%）。
4. **LLM 提示词收紧**：明确子意图消歧边界 + few-shot 示例 + 问候守卫，把口语/省略与细子意图的识别从 53% 拉到 90%。

### 3.2 后续优化方向

- **提示词微调（可选，不影响主目标）**：`delayed` 触发词补「很慢 / 慢」；统一「发出来 / 发货」类表达的归属（建议归 `order_query`）。当前 96.5% 已是稳健结果。
- **覆盖判定更稳的长期方案（可选）**：
  - 按信号强弱而非阈值决定覆盖：规则命中显式关键词（`keyword_hit`，高精度）就永不覆盖；只有靠情绪 / 弱信号命中的才允许 LLM 覆盖，比单一阈值更可解释。
  - 让 `confidence_without_order` 真正低于覆盖线：把 order / logistics 无订单时的置信度设到 `0.6~0.65`，使「无订单的不确定规则」确实进入覆盖 band，同时把 T 固定在 0.7——覆盖只发生在「规则自己也不确定」时，语义更清晰。
- **评估回归常态化**：改 `intent_rules.yml` 或提示词后，跑 `gen_cases.py` + `run_eval.py` 并 `--sweep-threshold` 一键扫描最优覆盖阈值，确保优化不退化。
