# 客服 Agent MVP 系统架构

## 1. 文档目标

本文档用于描述客服 Agent MVP 的系统架构设计，将前面的需求分析和技术选型落到具体模块、调用链路、数据流和部署边界上。

本文档重点回答以下问题：

+ 系统由哪些核心模块组成
+ 每个模块负责什么职责
+ 请求是如何在模块之间流转的
+ 哪些能力在 MVP 阶段必须具备
+ 后续系统如何平滑演进

---

## 2. 架构设计原则

MVP 阶段架构遵循以下原则：

+ 单体优先，避免过早微服务化
+ 主流程可控，避免过度自治
+ 一套主库优先，减少多组件同步成本
+ 检索能力够用优先，后续再独立扩容
+ 高风险操作默认收敛到人工
+ 全链路可观测，便于调试、回放和评估

---

## 3. 总体架构

### 3.1 逻辑分层

系统可以拆成以下五层：

1. 接入层
2. Agent 编排层
3. 检索与知识层
4. 业务集成层
5. 数据与观测层

### 3.2 总体调用关系

```text
用户/渠道
   |
   v
接入层(FastAPI)
   |
   v
Agent编排层(LangGraph)
   |                \
   |                 \
   v                  v
检索与知识层         业务集成层
(PG FTS+pgvector)    (订单/物流/售后/人工客服)
   \                  /
    \                /
     v              v
      数据与观测层(PostgreSQL/Redis/Langfuse)
```

---

## 4. 核心模块设计

### 4.1 接入层

推荐技术：

+ `FastAPI`

职责：

+ 提供对话接口
+ 接收来自 Web、App、IM、客服工作台的用户请求
+ 做鉴权、参数校验、限流、幂等控制
+ 将请求转交给 Agent 编排层
+ 返回标准化响应

MVP 阶段建议接口：

+ `POST /chat`
+ `POST /handoff`
+ `GET /session/{session_id}`
+ `GET /health`

边界说明：

+ 接入层不负责业务决策
+ 不直接调用知识库和外部系统
+ 只做协议转换、鉴权、基础校验和转发

---

### 4.2 Agent 编排层

推荐技术：

+ `LangGraph`
+ `Pydantic`

职责：

+ 维护客服 Agent 的核心流程
+ 执行意图识别、状态跟踪、澄清、路由、工具调用、转人工决策
+ 组织多轮对话中的上下文和状态
+ 对输出做结构化校验

建议的核心节点：

+ `input_normalizer`
+ `intent_router`
+ `state_tracker`
+ `faq_retriever`
+ `business_tool_executor`
+ `clarification_handler`
+ `handoff_handler`
+ `response_generator`
+
建议的状态字段：

+ `session_id`
+ `user_id`
+ `current_intent`
+ `stage`
+ `slots`
+ `missing_slots`
+ `confirmed_slots`
+ `risk_level`
+ `needs_clarification`
+ `topic_changed`
+ `summary`

边界说明：

+ 编排层是系统核心，但不承担底层存储和外部系统职责
+ 所有工具调用必须通过受控接口封装
+ 高风险动作不应在本阶段自动执行

---

### 4.3 检索与知识层

推荐技术：

+ `PostgreSQL Full Text Search`
+ `pgvector`
+
职责：

+ 存储 FAQ、售后规则、活动规则、商品知识、帮助文档等知识内容
+ 对用户问题执行混合检索
+ 返回候选知识片段供 Agent 回答
+ 为后续 rerank 和检索评估提供基础数据

推荐检索链路：

1. 对 query 进行标准化
2. 执行全文检索召回
3. 执行向量检索召回
4. 做结果融合，如 `RRF` 或加权分数融合
5. 必要时增加 rerank
6. 将 Top-K 结果返回给编排层

为什么这样设计：

+ 客服场景不仅依赖语义相似，还非常依赖关键词命中
+ 规则条款、商品型号、售后术语、活动名称都不适合只做纯向量检索
+
边界说明：

+ 检索层只负责“找内容”，不负责最终回答
+ 最终回答仍由 Agent 编排层决定
+
---

### 4.4 业务集成层

职责：

+ 对接订单系统
+ 对接物流系统
+ 对接售后/工单系统
+ 对接人工客服工作台
+ 对接用户账户系统

推荐实现方式：

+ 对每个外部系统封装统一的 service / tool adapter
+ 所有工具调用参数由 `Pydantic` 校验
+ 对外部接口错误统一处理并转换为 Agent 可理解的结果

MVP 阶段建议开放的能力：

+ 查询订单状态
+ 查询物流状态
+ 查询售后进度
+ 创建转人工请求
+
不建议在 MVP 开放的能力：

+ 自动退款
+ 修改敏感资料
+ 支付、提现、权限变更

边界说明：

+ 业务集成层不参与意图判断
+ 它只负责把受控请求转成外部系统调用
+
---

### 4.5 数据与观测层

推荐技术：

+ `PostgreSQL`
+ `Redis`
+ `Langfuse` 或 `LangSmith`

职责：

+ 存储会话记录、状态快照、审计日志、知识元数据
+ 缓存热点数据和短期会话态
+ 记录 prompt、模型调用、工具调用链、耗时、异常、最终回复
+ 支撑后续评测、问题回放和策略迭代

建议存储的数据类型：

+ 对话消息表
+ 会话状态表
+ 工具调用日志表
+ 转人工记录表
+ 检索结果日志表
+ 评测结果表
+
边界说明：

+ Redis 只做缓存和短期状态，不做长期审计存储
+ PostgreSQL 是 MVP 阶段的主事实来源

## 5. 关键调用链路

### 5.1 FAQ 问答链路

1. 用户问题进入接入层
2. 编排层识别为 `faq` 或知识查询意图
3. 检索层执行混合检索
4. 编排层根据检索结果生成回答
5. 若置信度不足，则追问或转人工

### 5.2 订单/物流查询链路

1. 用户提出查询需求
2. 编排层识别意图并抽取订单号等参数
3. 参数不足则进入澄清节点
4. 参数齐全后调用业务集成层
5. 返回查询结果并生成自然语言回复

### 5.3 售后咨询引导链路

1. 用户表达退款/退货/换货诉求
2. 编排层维护任务状态并逐步收集资料
3. 检索层提供售后规则依据
4. 若仅为咨询，则给出引导
5. 若需要升级处理，则触发转人工

### 5.4 转人工链路

1. 用户主动要求人工，或系统判定需要人工介入
2. 编排层生成摘要和结构化上下文
3. 业务集成层调用人工客服系统
4. 将会话转给人工，并保留完整上下文

## 6. MVP 关键数据流

### 6.1 输入数据

+ 用户消息
+ 会话标识
+ 用户标识
+ 渠道标识
+ 可能的附件或结构化参数

### 6.2 中间数据

+ 意图识别结果
+ 槽位提取结果
+ 会话状态对象
+ 检索召回结果
+ 工具调用结果
+ 转人工原因

### 6.3 输出数据

+ Agent 最终回复
+ 转人工请求
+ 审计日志
+ Trace 与评测事件

## 7. 部署架构建议

### 7.1 MVP 部署方式

建议先采用 `Docker Compose` 单机或轻量环境部署：

+ `api-service`
+ `postgres`
+ `redis`
+ `langfuse`（可选）

优势：

+ 环境简单
+ 本地开发、测试、演示一致
+ 方便快速验证端到端能力

### 7.2 不建议的过早设计

+ 一开始就拆微服务
+ 一开始就引入独立向量库集群
+ 一开始就引入 Elasticsearch 集群
+ 一开始就做复杂多 Agent 架构

原因：

+ 当前主要矛盾是业务闭环和产品可用性，不是基础设施扩展性

## 8. 安全与风险控制架构

### 8.1 风险控制原则

+ 高风险动作必须确认或转人工
+ 低置信度不直接执行
+ 外部接口错误不可静默吞掉
+ 敏感信息必须脱敏和审计

### 8.2 架构中的控制点

+ 接入层：鉴权、限流、基本参数校验
+ 编排层：意图风险分级、澄清、确认、转人工判断
+ 业务集成层：权限校验、接口兜底、错误封装
+ 数据层：日志审计、敏感字段脱敏

---

## 9. 后续演进方向

### 9.1 检索层演进

+ 当前：`PostgreSQL FTS + pgvector`
+ 后续：当数据量、QPS 或召回复杂度上来后，再评估 `Qdrant` / `Milvus` / `Elasticsearch`

### 9.2 服务层演进

+ 当前：单体服务
+ 后续：可按 `API层 / Agent层 / 检索层 / 业务集成层` 拆分

### 9.3 能力层演进

+ 当前：FAQ、查单、查物流、售后引导、转人工
+ 后续：图片理解、多渠道接入、客服 Copilot、主动服务

---

## 10. 最终建议

MVP 阶段建议采用以下总体架构：

+ 接入层：`FastAPI`
+ 编排层：`LangGraph + Pydantic`
+ 检索层：`PostgreSQL FTS + pgvector`
+ 集成层：订单/物流/工单/人工客服 adapter
+ 数据层：`PostgreSQL + Redis`
+ 观测层：`Langfuse` 或 `LangSmith`

该架构的核心优势在于：

+ 结构清晰
+ 组件数量少
+ 便于快速落地
+ 有足够的可控性和可观测性
+ 能平滑演进到更复杂的客服 Agent 系统
