# 客服 Agent MVP 知识库设计

## 1. 文档目标

本文档用于描述客服 Agent MVP 的知识库设计，重点覆盖：

+ 文档来源与文档类型
+ 切块策略
+ 向量存储
+ 双路召回
+ 融合与重排
+ 检索触发与边界

---

## 2. 设计目标

知识库的目标不是“尽可能多存文档”，而是为客服 Agent 提供可控、可检索、可更新的业务知识支撑。

主要服务于以下场景：

+ FAQ 问答
+ 售后规则说明
+ 发票、优惠、活动规则说明
+ 商品和服务说明
+ 需要规则依据的咨询型回复

知识库不用于替代实时业务系统，不负责订单、物流、售后进度等实时数据查询。

---

## 3. 文档来源与文档类型

MVP 阶段建议优先纳入以下文档类型：

+ FAQ 文档
+ 售后政策文档
+ 发票规则文档
+ 优惠活动规则文档
+ 商品说明文档
+ 帮助中心文档

建议按类型管理：

+ `faq`
+ `policy`
+ `product`
+ `help`

设计原则：

+ 只纳入会直接影响客服答复的知识
+ 规则型文档优先于泛化介绍型文档
+ 每篇文档必须有明确版本和启用状态

---

## 4. 切块方案

知识库切块的目标是：

+ 保留足够语义完整性
+ 控制检索粒度
+ 便于后续 RAG 回答和规则引用

MVP 阶段建议采用“文档结构优先 + 必要时递归细分”。

### 4.1 推荐切块原则

+ FAQ 文档：按问答对切块
+ 规则文档：按标题、条款或段落切块
+ 商品说明：按模块切块，如规格、售后、使用说明
+ 帮助文档：按标题和子标题切块

### 4.2 切块要求

+ 单块应聚焦单一主题
+ 单块尽量不要跨多个无关规则
+ 切块后应保留来源信息，如文档标题、章节标题、条款编号
+ 同一文档应支持版本替换与批量重建

### 4.3 元数据建议

每个块建议至少包含：

+ `document_id`
+ `doc_type`
+ `chunk_no`
+ `title`
+ `section`
+ `version`
+ `tags`
+
这些字段用于过滤、排序、审计和后续解释。

---

## 5. 向量存储

MVP 阶段建议使用 `PostgreSQL + pgvector` 存储知识块向量。

原因：

+ 与项目主库一致
+ 方便统一维护知识元数据
+ 数据量较小时运维成本低
+ 可与 PostgreSQL Full Text Search 组合实现混合检索

知识块存储建议：

+ 原始文档落在 `knowledge_documents`
+ 切块内容落在 `knowledge_chunks`
+ `embedding` 存于 `knowledge_chunks.embedding`
+ 全文检索字段存于 `knowledge_chunks.tsv`

---

## 6. 双路召回

本项目知识检索采用“双路召回”：

+ 全文检索召回
+ 向量检索召回

### 6.1 全文检索召回

适合处理：

+ 商品型号
+ 规则条款
+ 活动名称
+ 发票术语
+ 用户原话中的明确关键词

### 6.2 向量检索召回

适合处理：

+ 口语化表达
+ 同义问法
+ FAQ 相似问题
+ 语义接近但关键词不完全一致的问题

### 6.3 为什么采用双路

客服场景不是纯语义问答，很多问题对关键词命中非常敏感。

因此不建议只做纯向量召回，推荐同时保留：

+ 关键词召回的精确性
+ 向量召回的泛化能力

---

## 7. 融合与重排

双路召回后，需要对结果进行融合与筛选。

### 7.1 融合

MVP 阶段建议采用简单可控的融合方式：

+ `RRF`
+ 加权分数融合

目标：

+ 保留关键词命中的高精度结果
+ 引入语义召回的补充结果
+
### 7.2 重排

在以下场景建议增加 `rerank`：

+ 规则文档较多
+ FAQ 候选较多
+ 多条候选内容语义接近
+
MVP 阶段可选，不作为第一优先级。

### 7.3 结果筛选

建议对最终候选做以下控制：

+ 限制返回 Top-K
+ 去重
+ 剔除低相关性结果
+ 优先保留高可信来源文档

---

## 8. 检索触发策略

知识库检索不应作为默认动作，而应由 `policy layer` 受控触发。

适合触发检索的场景：

+ `faq`
+ `refund_service`
+ `refund_service`
+ `refund_service`
+ `refund_service`
+ `refund_service`
+
不应触发知识库检索的场景：

+ `order_service`
+ `logistics_service`
+ `handoff_human`
+ 当前仅缺少关键参数时
+
判断边界：

+ 规则/说明/政策问题 -> 走知识检索
+ 实时业务状态问题 -> 走工具调用
+ 当前只差参数 -> 先做 `slot` 澄清

---

## 9. 进一步优化方向

MVP 之后可按优先级考虑以下优化：

+ `rerank`
+ 分类型索引与召回策略
+ 多模态知识块
+ 更细的文档可信度权重
+ 基于失败样本补充 FAQ 示例

不建议在 MVP 阶段过早引入：

+ 复杂多阶段检索编排
+ 独立向量库集群
+ 多模态知识库主流程
+
---

## 10. 最终建议

本项目知识库设计建议正式定版如下：

+ 文档按 `faq / policy / product / help` 分类
+ 切块采用“结构优先，必要时递归细分”
+ 存储采用 `PostgreSQL + pgvector`
+ 检索采用“全文检索 + 向量检索”双路召回
+ 融合优先用简单可控的 `RRF` 或加权融合
+ 检索仅在知识型问题上触发，不替代工具查询

该方案适合客服 Agent 的 FAQ、规则说明和咨询型答复场景，并便于后续与数据表设计对接。
