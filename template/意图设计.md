# 结论先给：基础可用，但**业务分层不清晰、边界重叠、缺少前置兜底、缺少投诉情绪分支**，线上客服容易出现意图误判、路由错乱，尤其 Multi-Agent 调度场景会踩坑。
## 一、现有6类意图拆解优缺点
现有分类：`order` / `logistics` / `refund` / `handoff` / `chitchat` / `unsupported`
### 优点
1. 覆盖电商客服核心主干业务；
2. 粒度中等，不会过于细碎增加分类模型负担；
3. 人工转接、闲聊、无效咨询兜底齐全。

### 致命缺陷（线上高频踩坑点）
1. **业务边界高度重叠，极易识别混淆**
    - 用户：“我订单丢件，我要退款”
      同时包含 logistics + refund，单意图只能选一个，路由逻辑难写；
    - 用户：“商品破损，物流显示签收，申请退货退款”
      order+logistics+refund 三合一，单分类输出会丢次要诉求。
2. **缺失高权重客服分支：投诉/情绪安抚**
    生气、辱骂、不满、赔付诉求没有单独意图，全部挤入 refund/handoff，投诉工单无法自动分流投诉专员；
3. **order 定义模糊，范围太大**
    order 到底是查订单、改地址、改规格、开票？所有订单基础操作揉在一起，后续子Agent无法精细化调度；
4. **handoff 被动触发，缺少主动前置识别**
    handoff 仅代表用户要求转人工，但大量复杂纠纷、重复咨询模型自知解决不了，无法主动触发转接；
5. **unsupported 定义模糊**
    是不在业务范围？还是提问模糊无法识别？两类场景处理逻辑完全不同，混在一起难以运营统计。

## 二、优化后标准分层意图体系（电商客服生产通用，适配LangGraph多Agent路由）
### 一级意图（分类模型输出，互斥）
1. `order_query` 订单基础查询（查订单、改收货地址、修改规格、开具发票）
2. `logistics` 物流问题（丢件、延迟、未收到货、驿站、派送异常）
3. `aftersale_refund` 售后退款/退货换货（破损、质量问题、七天无理由）
4. `complaint` 投诉维权（不满、赔付、辱骂、服务纠纷、重复投诉）
5. `transfer_human` 用户主动要求转人工
6. `chitchat` 闲聊问候、无关日常对话
7. `unrecognize` 语句模糊、语义不清（不知道用户想问啥）
8. `unsupported_biz` 超出平台业务范围咨询（如加盟、招聘、线下门店）

### 配套二级子意图（可选，LLM二次拆解，解决多诉求重叠）
一级分类定路由大分支，进入对应Agent后再提取子诉求，完美解决一句话多业务问题。
示例：
- 一级：aftersale_refund
  二级：damage_refund / no_reason_return / wrong_goods
- 一级：logistics
  二级：lost_package / delayed / not_received

## 三、各类意图标准处理逻辑（Multi-Agent调度参考）
1. `order_query` → 订单查询Agent（调用订单API，返回订单信息）
2. `logistics` → 物流工具Agent（查快递轨迹、拦截派送）
3. `aftersale_refund` → 售后处理Agent（校验售后条件、发起退款工单）
4. `complaint` → 情绪安抚+投诉工单Agent（优先安抚，自动生成投诉单升级）
5. `transfer_human` → 直接转人工，同步完整会话上下文
6. `chitchat` → 闲聊回复Agent，不调用任何业务工具
7. `unrecognize` → 引导用户清晰描述问题，不触发工具
8. `unsupported_biz` → 统一标准话术告知不支持该业务

## 四、两种落地方案选择
### 方案1：仅保留一级互斥意图（推荐中小单Agent系统）
直接使用优化后的8类，微调prompt，让模型**优先识别主诉求**，次要诉求附带提取放入metadata传给下游Agent，规避多意图冲突。
适合：单Agent客服、访问量不大、业务简单平台。

### 方案2：一级粗分 + 二级细分子意图（大型Multi-Agent平台）
1. 路由层LLM输出一级意图，分流到对应业务子Agent；
2. 业务子Agent内部再调用LLM拆解详细子诉求，并行调用多个工具；
适合：你当前场景（LangGraph多智能体、工单、退款、物流多模块分离）。

## 五、原6分类改造最简过渡方案（不想大幅改动代码）
如果存量代码大量依赖原有6个枚举，不想改数据库、路由逻辑，只做规则增强：
1. 新增隐式标签 `complaint` 存入会话metadata，不改动主意图枚举；
2. 区分 `unsupported` 内部逻辑：增加关键词规则区分模糊提问/非业务咨询；
3. 在refund、logistics节点内部增加子诉求识别，处理一句话多业务场景；
4. prompt增加约束：多诉求时以用户**最终目的**作为主意图。

## 六、关键Prompt约束示例（解决多意图混淆）
> 你需要识别用户对话的**核心诉求（最终目的）**，只能输出单一意图；
> 若一句话同时包含物流、退款，用户最终想要退款则输出refund；
> 若用户情绪激烈、索要赔付，标记附带投诉标签；
> 枚举：order/logistics/refund/handoff/chitchat/unsupported

## 七、总结
1. 原始6个枚举**能跑，但不适合规模化生产**，投诉、多诉求、边界模糊三大硬伤会持续带来线上问题；
2. 长期迭代、多Agent架构建议升级为8类一级意图，搭配二级细分；
3. 存量系统不想重构可仅补充投诉元数据、增强prompt规则做过渡优化。
