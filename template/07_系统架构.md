# 客服 Agent MVP 系统架构

## 1. 文档目标

本文档用于描述客服 Agent MVP 的系统架构，重点覆盖：

+ 核心模块划分
+ LangGraph 中的节点与边设计
+ 主链路与子链路流转
+ 状态、检索、工具和转人工在架构中的位置
+ 部署与演进边界

---

## 2. 架构总览

MVP 阶段采用 `FastAPI + LangGraph + PostgreSQL/pgvector + Redis`。

总体分为五层：

1. 接入层
2. Agent 编排层
3. 知识检索层
4. 业务工具层
5. 数据与观测层

总体关系如下：

```text
用户/渠道
   |
   v
接入层(FastAPI)
   |
   v
Agent编排层(LangGraph)
   |               \
   |                \
   v                 v
知识检索层           业务工具层
(PG FTS+pgvector)    (订单/物流/售后/转人工)
   \                 /
    \               /
     v             v
      数据与观测层(PostgreSQL/Redis/Trace)
```

---

## 3. 模块划分

### 3.1 接入层

职责：

+ 提供对话接口
+ 接收 Web、App、IM、客服工作台请求
+ 做鉴权、限流、幂等和基础参数校验
+ 将请求转交给 LangGraph

建议接口：

+ `POST /chat`
+ `POST /handoff`
+ `GET /health`

### 3.2 Agent 编排层

职责：

+ 识别意图
+ 更新 DST 状态
+ 决定下一步动作
+ 驱动检索、工具、澄清与转人工
+ 生成最终回复

### 3.3 知识检索层

职责：

+ 承接 FAQ、规则、政策、帮助文档检索
+ 提供全文检索 + 向量检索双路召回
+ 返回可用于回答生成的知识片段

### 3.4 业务工具层

职责：

+ 查询订单状态
+ 查询物流状态
+ 查询售后进度
+ 创建转人工请求

### 3.5 数据与观测层

职责：

+ 存储会话、状态、工具日志、知识元数据
+ 缓存活跃会话和短期状态
+ 记录 trace、失败回放、评测数据

## 4. LangGraph 设计

### 4.1 设计原则

LangGraph 采用 `workflow-first` 的受控 `agentic workflow`。

核心原则：

+ 图负责流程编排
+ 状态对象负责跨节点传递信息
+ 节点只做单一职责动作
+ 条件边只根据结构化决策结果跳转
+ 高风险动作不允许开放式探索

### 4.2 主状态对象

图中的共享状态建议至少包含：

+ `session_id`
+ `user_id`
+ `channel`
+ `messages`
+ `summary`
+ `current_intent`
+ `sub_intent`
+ `stage`
+ `slots`
+ `missing_slots`
+ `confirmed_slots`
+ `candidate_intents`
+ `risk_level`
+ `needs_clarification`
+ `current_action`
+ `latest_action_result`
+ `handoff_reason`

### 4.3 主图节点

建议主图至少包含以下节点：

+ `input_normalizer`
+ `intent_router`
+ `state_tracker`
+ `policy_layer`
+ `clarification_node`
+ `knowledge_retriever`
+ `business_tool_executor`
+ `response_generator`
+ `handoff_node`
+ `context_compressor`
+ `message_writer`

各节点职责如下：

#### `input_normalizer`

+ 标准化输入消息
+ 附加渠道、用户、会话元信息
+ 加载当前活跃状态和最近上下文

#### `intent_router`

+ 执行规则层与 LLM 路由
+ 输出 `main_intent / sub_intent / candidate_intents / confidence / risk_level / emotion`
+ 判断是否发生 `main_intent shift`
+ **情绪识别**：与意图识别合并为同一次 LLM 调用，不单独设节点。
  规则层先识别明显情绪关键词（投诉、催促等），
  LLM 层补充复杂语境下的情绪判断，结果写入 `state.emotion`。

#### `state_tracker`

+ 根据 `main_intent` 读取 `slot schema`
+ 提取本轮 slot
+ 合并到状态对象
+ 计算 `missing_slots`
+ 处理继承、覆盖与状态冻结

#### `policy_layer`

+ 决定下一步动作
+ 输出以下受控动作之一：
  + `answer_directly`
  + `retrieve_knowledge`
  + `query_business_tool`
  + `ask_intent_clarification`
  + `ask_slot_clarification`
  + `handoff_human`

**决策伪代码**：

```
if handoff_reason is present:
    return handoff_human

if main_intent == "unsupported":
    return handoff_human   # 或 answer_directly（兜底话术）

if missing_slots is not empty:
    return ask_slot_clarification

if main_intent in {"faq"}:
    return retrieve_knowledge   # FAQ 走知识检索

if main_intent in {"order_service", "logistics_service", "refund_service"}:
    if all required_slots present:
        return query_business_tool
    else:
        return ask_slot_clarification

if confidence < CONFIDENCE_THRESHOLD:
    return ask_intent_clarification

return answer_directly   # 兜底：直接生成回复
```

> 注：`policy_layer` 是纯结构化决策节点，不含 LLM 调用。
> 决策结果写入 `state.current_action`，由条件边驱动下游节点。

#### `clarification_node`

+ 根据 `policy_layer` 的结果生成澄清问题
+ 区分意图澄清与 slot 澄清

#### `knowledge_retriever`

+ 仅在知识型问题时触发
+ 执行双路召回
+ 返回知识片段给后续回答节点

#### `business_tool_executor`

+ 执行订单、物流、售后、转人工工具
+ 产出 `raw_result / sanitized_result / user_facing_summary`

#### `response_generator`

+ 根据状态、知识片段或工具结果生成最终回复
+ 必要时可在低风险节点内受控使用 ReAct

#### `handoff_node`

+ 触发人工切换
+ 生成 handoff 摘要和状态快照

#### `context_compressor`

+ 判断当前轮是否需要触发上下文压缩
+ 维护 `running_summary`
+ 裁剪最近原文窗口
+ 在写入持久层前更新会话摘要

#### `message_writer`

+ 将状态、消息、工具调用、失败原因持久化
+ 更新会话摘要和活动状态

## 5. LangGraph 边设计

### 5.1 主链路

主链路建议如下：

```text
START
  -> context_compressor   ← 每轮开始前压缩长上下文
  -> input_normalizer
  -> intent_router
  -> state_tracker
  -> policy_layer
  -> (conditional edge)
```

> **说明**：`context_compressor` 放在每轮推理之前，
> 确保送入 `intent_router` 的上下文窗口不超过预算。
> 压缩后的 `running_summary` 写入状态对象，供后续节点读取。

### 5.2 `policy_layer` 条件边

`policy_layer` 是主分发点，建议基于结构化动作做条件边：

+ `answer_directly` -> `response_generator`
+ `retrieve_knowledge` -> `knowledge_retriever`
+ `query_business_tool` -> `business_tool_executor`
+ `ask_intent_clarification` -> `clarification_node`
+ `ask_slot_clarification` -> `clarification_node`
+ `handoff_human` -> `handoff_node`

### 5.3 检索链路边

```text
policy_layer
  -> knowledge_retriever
  -> response_generator
  -> message_writer
  -> END   ← 下一轮由 START 重新进入 context_compressor
```

说明：

+ 知识检索只负责返回候选片段
+ 不直接决定最终回复
+ `context_compressor` 不在本分支末尾重复执行，由下一轮 START 统一触发

### 5.4 工具链路边

```text
policy_layer
  -> business_tool_executor
  -> response_generator
  -> message_writer
  -> END   ← 下一轮由 START 重新进入 context_compressor
```

说明：

+ 工具节点返回结构化结果
+ 回复节点只读取清洗后的结果与状态对象
+ `context_compressor` 不在本分支末尾重复执行，由下一轮 START 统一触发

### 5.5 澄清链路边

```text
policy_layer
  -> clarification_node
  -> message_writer
  -> END   ← 下一轮由 START 重新进入 context_compressor
```

说明：

+ 意图澄清与 slot 澄清共用节点
+ 节点内根据 `current_action` 或 `clarification_type` 生成不同问题
+ `context_compressor` 不在本分支末尾重复执行，由下一轮 START 统一触发

### 5.6 转人工链路边

```text
policy_layer
  -> handoff_node
  -> message_writer
  -> END   ← 转人工后不再进入下一轮，无需触发 context_compressor
```

说明：

+ handoff 节点负责生成转人工摘要、快照与原因
+ message_writer 负责持久化 handoff 记录
+ 转人工为终态，无需压缩上下文

## 6. 子链路设计

主图可以进一步分成若干业务子链路：

+ `FAQ Flow`
+ `Order Flow`
+ `Logistics Flow`
+ `Refund Flow`
+ `Handoff Flow`

### 6.1 FAQ Flow

适用意图：

+ `faq`
+
默认动作：

+ `retrieve_knowledge`
+ `response_generator`

### 6.2 Order / Logistics Flow

适用意图：

+ `order_service`
+ `logistics_service`

默认动作：

+ 若缺 slot -> `ask_slot_clarification`
+ 否则 -> `query_business_tool`

### 6.3 Refund Flow

适用意图：

+ `refund_service`
+ `refund_service`
+ `refund_service`

默认动作：

+ 优先补齐 `required_slots`
+ 规则说明走 `retrieve_knowledge`
+ 状态查询或提交流程走 `query_business_tool`
+ 高风险或无法推进时走 `handoff_human`

### 6.4 Handoff Flow

适用意图：

+ `handoff_human`
+ 情绪升级
+ 多次澄清失败
+ 高风险或越权场景

默认动作：

+ `handoff_node`

## 7. 状态与边的关键规则

### 7.1 状态更新

+ 所有节点只更新自己负责的状态字段
+ `intent_router` 不直接写业务工具结果
+ `business_tool_executor` 不直接修改主链路决策

### 7.2 状态冻结

+ 只有 `main_intent shift` 才冻结旧状态
+ 旧状态标记为非活跃
+ 新状态通过 `parent_state_id` 关联旧状态

### 7.3 上下文控制

+ 主推理链路优先读取结构化状态
+ 原始工具结果不直接进入主上下文
+ 长对话通过摘要窗口压缩
+ 上下文压缩由 `context_compressor` 节点统一处理，不分散在其他节点中

### 7.4 错误处理

统一错误处理策略：

| 失败类型 | 检测方式 | 处理方式 |
|---------|---------|---------|
| LLM 调用失败 | 异常捕获 | 降级到规则层重试；仍失败则返回兜底话术 |
| 向量数据库超时 | 调用超时 | 降级为关键词检索；仍失败则返回"系统繁忙" |
| 业务工具调用失败 | `tool_calls.status == failed` | `policy_layer` 根据 `error_code` 决定：重试 / 解释性回复 / 转人工 |
| 意图识别低置信度 | `confidence < THRESHOLD` | 进入 `ask_intent_clarification`；连续 2 次仍低则转人工 |
| Slot 澄清失败 | `slot_clarification_count >= 2` | 直接转人工，不继续追问 |
| 状态对象异常 | 必填字段缺失 | 重置为 `unsupported` 意图，重新进入 `intent_router` |

+ 所有失败写入 `failure_cases` 表，供后续分析
+ 高风险操作（退款、改地址）失败必须转人工，不自动重试

### 7.5 WebSocket 流式输出事件映射

接入层 `WS /ws/chat` 会在执行过程中推送结构化事件，各事件与 LangGraph 节点的对应关系如下：

| 事件类型 | 触发节点 | 触发时机 |
|---------|---------|---------|
| `status` | `input_normalizer` | 每轮开始，标记 `processing` |
| `intent` | `intent_router` | 意图识别完成后推送 `main_intent / sub_intent / confidence` |
| `state` | `state_tracker` | 状态更新完成后推送当前状态快照 |
| `trace` | 各节点 | 每节点执行完毕后推送节点名与耗时 |
| `tool_result` | `business_tool_executor` | 工具调用完成后推送 `user_facing_summary` |
| `final` | `response_generator` | 最终回复生成后推送 |
| `handoff` | `handoff_node` | 转人工触发后推送 `handoff_reason` 与摘要 |

> 注：所有事件通过 WebSocket 推送到前端，
> 前端根据 `type` 字段区分展示（状态面板、消息流、轨迹时间线等）。

## 8. 部署建议

MVP 阶段建议采用单体部署：

+ `api-service`
+ `postgres`
+ `redis`
+ `langfuse` 或其他 trace 服务（可选）

不建议过早引入：

+ 微服务拆分
+ 独立向量库集群
+ 多 Agent 集群
+ 复杂任务树编排

---

## 9. 最终建议

本项目系统架构建议正式定版如下：

+ 接入层：`FastAPI`
+ 编排层：`LangGraph`
+ 知识检索层：`PostgreSQL FTS + pgvector`
+ 业务工具层：订单/物流/售后/转人工工具
+ 数据层：`PostgreSQL + Redis`

LangGraph 中应明确：

+ 节点只做单一职责动作
+ 条件边只基于结构化决策结果跳转
+ `policy_layer` 是主分发点
+ `message_writer` 是持久化收口点

该架构适合当前客服 Agent MVP，并可平滑演进到更完整的流程图与工具体系。
