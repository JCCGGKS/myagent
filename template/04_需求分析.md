# 客服 Agent 需求分析

## 1. 需求背景

传统客服系统主要依赖人工坐席或规则型机器人，通常存在以下问题：

+ 高频重复问题占用大量人工时间
+ 售前咨询响应慢，影响转化
+ 售后流程依赖人工经验，处理效率不稳定
+ 复杂场景缺少统一分流和转人工机制
+ 知识库、订单、物流、工单等系统彼此割裂

客服 Agent 的目标不是简单替代人工，而是在标准化、可审计、可收敛的场景中承担接待、理解、检索、查询、引导和分流工作，并与人工客服形成协同。

---

## 2. 项目目标

### 2.1 业务目标

+ 提升客服接待效率和首响速度
+ 降低重复性咨询带来的人工成本
+ 提升售前转化率和售后问题解决效率
+ 让客服回复质量更稳定、更可控

### 2.2 产品目标

+ 能稳定回答高频 FAQ
+ 能理解用户意图并路由到正确处理链路
+ 能在多轮对话中维护状态和上下文
+ 能接入业务系统完成查询类任务
+ 能在高风险或低置信度场景下安全转人工

### 2.3 非目标

+ 第一阶段不追求覆盖所有客服场景
+ 第一阶段不追求完全替代人工坐席
+ 第一阶段不开放高风险自动执行能力

---

## 3. 用户与业务场景

### 3.1 目标用户

+ 外部客户：咨询商品、订单、物流、售后问题
+ 内部客服：需要 Agent 辅助分流、检索、生成回复
+ 运营人员：维护知识库、话术、路由规则和效果评估

### 3.2 核心场景

建议优先聚焦以下高频场景：

+ 商品或服务 FAQ 问答
+ 订单状态查询
+ 物流进度查询
+ 退款/退货/换货咨询引导
+ 发票、优惠、支付、账户类常见问题
+ 转人工

### 3.3 MVP 范围建议

P0：

+ FAQ 问答
+ 订单查询
+ 物流查询
+ 转人工

P1：

+ 退款/退货咨询引导
+ 售后规则解释
+ 情绪识别和投诉升级分流

P2：

+ 图片理解类售后
+ 个性化推荐
+ 主动营销或外呼

---

## 4. 业务痛点拆解

### 4.1 售前场景

+ 常见问题重复率高，如价格、规格、发货时效、优惠活动
+ 用户等待时间长时容易流失
+ 不同客服话术不统一，影响品牌体验

### 4.2 售后场景

+ 用户描述不完整，需要多轮补充信息
+ 售后规则复杂，人工判断成本高
+ 查询订单、物流、退款状态需要切多个系统

### 4.3 服务协同场景

+ 机器人无法解决时缺少顺畅转人工机制
+ 人工接手时上下文丢失，需要用户重复描述
+ 失败案例难以沉淀为后续优化数据

---

## 5. Agent 角色定位

客服 Agent 在系统中的角色应定义为“可控的智能服务入口”，而不是无限能力的通用聊天机器人。

其职责包括：

+ 理解用户问题
+ 识别意图和关键参数
+ 回答知识性问题
+ 调用系统执行查询类动作
+ 补齐流程所需信息
+ 决定何时澄清、何时确认、何时转人工

其不应承担的职责包括：

+ 编造不确定答案
+ 未经确认直接执行高风险操作
+ 绕过权限限制读取敏感数据
+ 在越权场景下继续处理

---

## 6. 核心能力需求

### 6.1 对话接待能力

+ 支持用户以自然语言发起咨询
+ 输出符合品牌要求的客服语气
+ 支持问候、追问、补充说明、打断和换话题

### 6.2 意图识别能力

Agent 需要将用户输入转成可路由、可执行的结构化结果，包括：

+ 当前主意图
+ 关键参数
+ 识别置信度
+ 是否需要追问
+ 风险等级

第一阶段建议支持的意图包括（与 `config/intent_schemas.yml` 对齐）：

+ `faq` — FAQ 问答
+ `order_service` — 订单状态查询
+ `logistics_service` — 物流进度查询
+ `refund_service` — 退款/退货/换货咨询
+ `handoff_service` — 转人工
+ `chitchat` — 问候/感谢
+ `unsupported` — 无法处理

> 注：`refund_service` 通过子意图 `refund_service.consult_policy` / `refund_service.request_refund` 区分咨询与执行，
> 详见 `config/intent_rules.yml`。

### 6.3 多轮对话状态跟踪能力

客服对话不是单轮问答，需要维护当前任务状态。状态跟踪要解决的问题：

+ 用户首轮信息不完整，后续补充时要能合并
+ 用户改口时要能覆盖旧参数
+ 用户切换话题时要能中断当前任务并保留上下文
+ 执行前要能判断信息是否齐全

### 6.4 知识库问答能力

+ 支持 FAQ、规则、活动、售后政策等内容问答
+ 支持基于知识库检索后生成答案
+ 检索结果不确定时不强答，需澄清或转人工
+ 知识内容需支持持续更新和版本管理

### 6.5 业务系统调用能力

第一阶段建议只开放低风险查询类能力：

+ 查询订单状态
+ 查询物流状态
+ 查询退款/售后进度
+ 查询基础账户信息（需要权限控制）

### 6.6 流程引导能力

+ 在退款/退货等流程中主动收集缺失信息
+ 对执行条件做前置校验
+ 对高风险节点做二次确认
+ 不能自动执行时要明确告知并引导人工处理

### 6.7 转人工能力

以下场景应支持自动或半自动转人工：

+ 用户明确要求人工客服
+ Agent 连续低置信度
+ 高风险操作无法自动处理
+ 用户情绪激烈、投诉升级
+ 知识库和工具都无法解决

转人工时应附带必要的上下文信息，避免人工重复收集问题背景。

## 7. 功能流程需求

### 7.1 FAQ 流程

1. 用户发起问题
2. Agent 判断是否命中知识库
3. 命中则直接回答
4. 未命中则进入意图识别和后续路由
5. 仍无法判断则追问或转人工

### 7.2 订单/物流查询流程

1. 识别查询意图
2. 提取订单号、手机号等必要参数
3. 参数缺失时追问
4. 参数齐全后调用接口
5. 返回结果并支持继续追问

### 7.3 售后咨询流程

1. 识别退款/退货/换货意图
2. 收集订单号、原因、商品信息、凭证等内容
3. 校验是否符合售后规则
4. 能自动推进则继续引导
5. 需要人工审核则转人工

### 7.4 转人工流程

1. 判断是否满足转人工条件
2. 生成会话摘要和上下文信息
3. 将会话分配给人工坐席
4. 人工接手后无需用户重复描述

---

## 8. 能力边界与风险控制

### 8.1 允许 Agent 直接处理的事项

+ FAQ 回复
+ 订单/物流/售后进度查询
+ 售后规则解释
+ 收集资料和流程引导
+ 转人工和生成摘要

### 8.2 不允许 Agent 直接处理的事项

+ 未确认直接退款
+ 修改收货地址、手机号、支付信息
+ 处理付款、提现、权限变更等高风险动作
+ 在身份未校验时返回敏感信息

### 8.3 风险控制原则

+ 低置信度不执行
+ 高风险动作必须确认
+ 对外部接口错误有明确兜底话术
+ 全链路保留审计日志

---

## 9. 非功能需求

### 9.1 性能要求

+ 常规问答响应时间尽量控制在 3 秒内
+ 查询类请求在接口正常时尽量控制在 5 秒内

### 9.2 可用性要求

+ 外部接口失败时能优雅降级
+ 系统异常时有明确提示和兜底策略

### 9.3 安全与合规要求

+ 用户隐私信息需脱敏存储和展示
+ 敏感操作要有权限校验和操作审计
+ 防 prompt injection、防越权访问、防伪造身份

### 9.4 可观测性要求

+ 记录会话日志
+ 记录意图识别结果和置信度
+ 记录工具调用链路和失败原因
+ 支持问题回放与错误分析

### 9.5 可运营性要求

+ 知识库支持持续维护
+ 路由规则和阈值支持后台配置
+ 支持沉淀失败样本做后续优化

---

## 10. 系统对接需求

客服 Agent 若要真正落地，需要至少考虑以下系统对接：

+ 知识库系统
+ 订单系统
+ 物流系统
+ 售后/工单系统
+ 用户账户系统
+ 人工客服工作台

每个系统需要明确：

+ 是否已有标准 API
+ 可暴露给 Agent 的字段范围
+ 查询权限和执行权限如何区分
+ 是否要求操作留痕和审计

---

## 11. 评估与验收指标

### 11.1 业务指标

+ 自助解决率
+ 转人工率
+ 平均处理时长
+ 首次响应时长
+ 用户满意度
+ 投诉率

### 11.2 模型与流程指标

+ FAQ 命中率
+ 意图识别准确率
+ 槽位抽取准确率
+ 澄清策略准确率
+ 工具调用成功率
+ 高风险误执行率

### 11.3 验收建议

+ P0 场景要具备可用闭环
+ 关键流程要可回归测试
+ 效果要明显优于传统 FAQ 机器人

---

## 12. 第一阶段实施建议

第一阶段建议只做一个可控 MVP：

+ 渠道：先做文本渠道
+ 场景：FAQ、订单查询、物流查询、售后咨询引导、转人工
+ 能力：意图识别、多轮状态跟踪、知识库问答、低风险工具调用
+ 运维：会话日志、失败回放、基础评估

先不做：

+ 语音客服
+ 图片审核型复杂售后
+ 全自动退款执行
+ 主动营销和个性化推荐

---

## 13. 待确认问题

后续和业务方讨论时，需要优先确认以下问题：

+ 目标行业是什么
+ 第一阶段最重要的 3 个场景是什么
+ 是面向外部客户还是内部客服辅助
+ 哪些动作允许 Agent 直接执行
+ 当前已有哪几个业务系统和知识库
+ 是否需要多渠道接入
+ 是否要求完整审计日志
+ 项目的首要成功指标是什么
