# 客服 Agent MVP 技术选型

## 1. 选型目标

本阶段技术选型服务于客服 Agent MVP，优先满足：

+ 跑通核心闭环
+ 控制组件数量
+ 便于观测和审计
+ 支持后续演进

---

## 2. 总体结论

推荐的 MVP 技术栈如下：

+ 编程语言：`Python 3.11+`
+ Web 框架：`FastAPI`
+ Agent 编排：`LangGraph`
+ 数据模型校验：`Pydantic v2`
+ 主数据库：`PostgreSQL`
+ 向量检索：`pgvector`
+ 缓存/会话态：`Redis`
+ 大模型接入：`OpenAI SDK` 或兼容 OpenAI 协议的模型网关
+ 可观测与评测：`Langfuse` 或 `LangSmith`
+ 部署方式：`Docker Compose`

一句话概括：

> 第一阶段采用 `Python + FastAPI + LangGraph + PostgreSQL/pgvector + Redis` 的轻量方案完成客服 Agent MVP。

---

## 4. 选型理由

+ `Python`：LLM、Agent、RAG、评测生态最成熟，开发效率高
+ `FastAPI`：适合对话接口、后台 API 和 Webhook，和 `Pydantic` 配合顺畅
+ `LangGraph`：适合有状态流程，能表达意图路由、状态跟踪、澄清和转人工
+ `PostgreSQL + pgvector`：一套库同时承载业务数据、JSON 状态、全文检索和向量检索
+ `Redis`：适合缓存、短期状态、限流和幂等
+ `Langfuse / LangSmith`：用于 trace、回放和评测

---

## 5. 为什么暂时不选这些方案

### 5.1 为什么不直接选 Qdrant

+ 不能替代业务主库
+ 一开始就上双库会增加同步和运维复杂度
+ 当前阶段目标是先跑通客服流程，而不是独立检索层

参考URL：<https://qdrant.tech/documentation/search/hybrid-queries/>

### 5.2 为什么不直接选 Milvus

+ 体系更重
+ 更适合海量向量和独立检索层
+ 对当前客服 MVP 来说通常是过度设计

参考URL：<https://milvus.io/docs/hybrid_search_with_milvus.md>

### 5.3 为什么不直接选 Elasticsearch

+ 当前目标是构建客服 Agent，不是先建设完整搜索平台
+ `PostgreSQL FTS + pgvector` 已能满足第一阶段混合检索需求
+ 一开始引入 Elasticsearch 会增加索引同步和运维复杂度

参考URL：<https://www.elastic.co/elasticsearch>

### 5.4 为什么不直接选 Chroma / FAISS

+ `Chroma` 更适合原型和教学
+ `FAISS` 是向量检索库，不是完整数据库

## 6. 后续演进路线

### 6.1 第一阶段

+ `Python + FastAPI + LangGraph + PostgreSQL/pgvector + Redis`
+ 支持 FAQ、订单查询、物流查询、售后咨询引导、转人工

### 6.2 第二阶段

+ 引入更完善的评测体系
+ 优化混合检索、rerank、意图识别和状态跟踪
+ 根据数据规模考虑拆分独立检索层

### 6.3 第三阶段

+ 若检索层瓶颈明显，再评估 `Qdrant` / `Milvus` / `Elasticsearch`
+ 若多渠道和复杂流程增加，再拆分 Agent 服务、检索服务和业务集成层

---

## 7. 最终建议

MVP 阶段建议正式定版如下：

+ `Python 3.11+`
+ `FastAPI`
+ `LangGraph`
+ `Pydantic v2`
+ `PostgreSQL`
+ `pgvector`
+ `Redis`
+ `OpenAI SDK / 兼容网关`
+ `Langfuse` 或 `LangSmith`
+ `Docker Compose`

核心特点：

+ 架构简洁
+ 开发效率高
+ 足够支撑第一阶段落地
+ 后续可平滑演进
