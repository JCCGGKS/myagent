# 客服 Agent MVP 技术选型

## 1. 选型目标

本阶段技术选型服务于客服 Agent 的 MVP 落地，目标不是一次性搭建“终局架构”，而是优先满足以下要求：

+ 能尽快跑通 FAQ、订单查询、物流查询、转人工等核心闭环
+ 系统结构尽量简单，减少组件数量和运维成本
+ 支持后续扩展为更完整的 Agent、RAG 和评测体系
+ 对高风险操作保持可控，便于审计和回放

因此，本阶段优先考虑：

+ 少组件
+ 易开发
+ 易观测
+ 易演进

---

## 2. 总体结论

推荐的 MVP 技术栈如下：

+ 编程语言：`Python 3.11+`
+ Web 框架：`FastAPI`
+ Agent 编排：`LangGraph`
+ 数据模型校验：`Pydantic v2`
+ 主数据库：`PostgreSQL`
+ 向量检索：`pgvector`
+ 缓存/会话态：`Redis`
+ 大模型接入：`OpenAI SDK` 或兼容 OpenAI 协议的模型网关
+ 可观测与评测：`Langfuse` 或 `LangSmith`
+ 部署方式：`Docker Compose`

一句话概括：

> 第一阶段采用 `Python + FastAPI + LangGraph + PostgreSQL/pgvector + Redis` 的轻量方案，以最少基础设施完成客服 Agent MVP。

## 4. 为什么这样选

### 4.1 为什么选 Python

+ Agent、RAG、向量检索、评测工具链最完整
+ 原型到生产的迁移成本低
+ 团队后续无论接 LangChain、LangGraph、LlamaIndex 还是自定义工作流，Python 都最顺手

不优先选其他语言的原因：

+ `Java`、`Go` 适合高并发服务，但在 Agent 编排、RAG 生态和实验效率上通常不如 Python
+ MVP 阶段更看重迭代速度，不是极限性能

### 4.2 为什么选 FastAPI

+ 适合快速构建客服对话接口、后台 API、Webhook
+ 和 `Pydantic`、类型注解天然配合
+ 文档、自测、接口调试体验好

不优先选其他方案的原因：

+ `Flask` 更轻，但类型约束、工程化体验不如 FastAPI
+ `Django` 更重，本阶段不需要完整的大型 Web 框架

### 4.3 为什么选 LangGraph

+ 客服 Agent 本质上不是单轮问答，而是有状态工作流
+ 需要显式表达 `意图识别 -> 状态更新 -> 检索/查单 -> 澄清 -> 转人工`
+ 比“纯 prompt + function calling”更适合做可控流程和后续评测

不优先选其他方案的原因：

+ 直接手写流程当然可以，但随着节点变多，可维护性会迅速变差
+ 纯 `LangChain Agent` 更偏通用 agent 调度，对客服这种强流程场景不如 LangGraph 清晰

### 4.4 为什么选 PostgreSQL + pgvector

+ 客服 Agent 第一阶段不仅需要向量库，还需要业务主库
+ 需要存储会话记录、工单、日志、用户状态、规则、知识元数据
+ `PostgreSQL` 同时支持结构化数据、`JSONB`、复杂查询、全文检索
+ `pgvector` 能让向量检索直接落在 PostgreSQL 里，减少一套额外基础设施
+ `PostgreSQL FTS + pgvector` 可以做混合检索，适合客服 FAQ、规则问答、术语命中

不优先选 `MySQL` 的原因：

+ `MySQL` 也能做全文检索，但当前 AI/RAG 生态里 `PostgreSQL + pgvector` 更成熟、更常见
+ 如果主库选 `MySQL`，通常更早就要额外引入搜索引擎或独立向量库，系统复杂度更高

不优先一开始就选独立向量库的原因：

+ MVP 阶段数据规模通常还不大
+ 一开始就上独立向量库会增加双库同步、运维、权限、排障成本
+ 对当前 P0 场景来说，收益不一定大于成本

### 4.5 为什么选 Redis

+ 会话短期状态、验证码、限流、缓存天然适合放在 Redis
+ 响应速度快，集成简单
+
不把 Redis 当主存储的原因：

+ Redis 不适合承载审计、复杂查询和长期业务数据
+ 客服系统必须保留可追溯日志和结构化状态，主存储仍应是 PostgreSQL

### 4.6 为什么选 Langfuse / LangSmith

+ Agent 系统的难点不只是“能运行”，更在于“出了问题能不能看懂”
+ 需要看到 prompt、上下文、工具调用链、耗时、失败点、模型输出
+ 后续评测和回归测试都依赖这些观测能力

不建议先裸跑日志的原因：

+ 普通应用日志不足以还原 Agent 执行轨迹
+ 没有 trace 和回放，后续优化效率会很低

## 5. 为什么暂时不选这些方案

### 5.1 为什么不直接选 Qdrant

`Qdrant` 很适合作为独立向量检索层，但 MVP 阶段不作为首选，原因如下：

+ 它不能替代业务主库，最终仍然需要 PostgreSQL 或 MySQL 存业务数据
+ 一开始就上 `PostgreSQL + Qdrant`，会增加双库同步和系统复杂度
+ 当前阶段核心目标是先跑通客服流程，不是先把检索层独立出来

适合引入 Qdrant 的时机：

+ 检索数据量明显增长
+ 需要更强的原生 hybrid queries
+ 检索层准备独立扩容

参考URL：<https://qdrant.tech/documentation/search/hybrid-queries/>

### 5.2 为什么不直接选 Milvus

`Milvus` 更适合中大型向量检索平台，不适合作为当前 MVP 首选：

+ 体系更重，运维和学习成本更高
+ 更适合海量向量、高吞吐、独立检索层场景
+ 对当前 FAQ、订单、物流类客服场景来说，通常是过度设计

适合引入 Milvus 的时机：

+ 向量数据规模很大
+ 检索层要独立成专门基础设施
+ 后续要扩展多向量、多模态检索能力

参考URL：<https://milvus.io/docs/hybrid_search_with_milvus.md>

### 5.3 为什么不直接选 Elasticsearch

`Elasticsearch` 很强，但它更偏专业搜索引擎，不是本阶段首选：

+ 当前主要目标是构建客服 Agent，而不是先建设完整搜索平台
+ 第一阶段用 `PostgreSQL FTS + pgvector` 已能满足 FAQ 和规则问答的混合检索需求
+ 一开始就引入 Elasticsearch，会多出索引同步、映射管理、运维监控等复杂度

适合引入 Elasticsearch 的时机：

+ 搜索量大幅增长
+ 关键词检索、聚合、排序、同义词、召回调优需求显著增强
+ 知识库搜索成为一个独立能力中心

参考URL：<https://www.elastic.co/elasticsearch>

### 5.4 为什么不直接选 Chroma / FAISS

+ `Chroma` 适合原型和教学，但不适合长期承载生产客服检索链路
+ `FAISS` 是向量检索库，不是完整数据库；若要做混合检索，还要自己拼全文检索和结果融合

适合的场景：

+ 本地实验
+ 教学学习
+ 纯向量检索验证

## 6. MVP 推荐架构

### 6.1 核心链路

1. 用户请求进入 `FastAPI`
2. `LangGraph` 负责意图识别、状态跟踪、工具编排
3. FAQ/规则问题走 `PostgreSQL FTS + pgvector` 混合检索
4. 订单/物流问题调用内部业务 API
5. 高风险或低置信度场景转人工
6. 全链路日志、trace 和评测结果进入 `PostgreSQL + Langfuse/LangSmith`

### 6.2 MVP 架构特征

+ 单体服务优先，必要时再拆分
+ 主库统一存储，避免多套基础设施并行演进
+ 检索先做“够用”，再做“极致”
+ 流程先可控，再追求高度自治

---

## 7. 后续演进路线

### 7.1 第一阶段

+ `Python + FastAPI + LangGraph + PostgreSQL/pgvector + Redis`
+ 支持 FAQ、订单查询、物流查询、售后咨询引导、转人工

### 7.2 第二阶段

+ 引入更完善的评测体系
+ 优化混合检索、rerank、意图识别和状态跟踪
+ 根据数据规模考虑拆分独立检索层

### 7.3 第三阶段

+ 若检索层瓶颈明显，再评估 `Qdrant` / `Milvus` / `Elasticsearch`
+ 若多渠道和复杂流程增加，再拆分 Agent 服务、检索服务和业务集成层

---

## 8. 最终建议

MVP 阶段建议正式定版如下：

+ `Python 3.11+`
+ `FastAPI`
+ `LangGraph`
+ `Pydantic v2`
+ `PostgreSQL`
+ `pgvector`
+ `Redis`
+ `OpenAI SDK / 兼容网关`
+ `Langfuse` 或 `LangSmith`
+ `Docker Compose`

该方案的核心优点是：

+ 架构简洁
+ 开发效率高
+ 足够支撑客服 Agent 第一阶段落地
+ 后续可平滑演进为更复杂的检索和 Agent 系统
