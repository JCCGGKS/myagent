# 客服 Agent MVP 意图识别设计

## 1. 文档目标

本文档用于描述客服 Agent MVP 的意图识别设计，重点覆盖：

+ 分层路由
+ 主意图/子意图设计
+ `slot schema`
+ 情绪识别
+ 对话状态跟踪（DST）
+ 状态更新、补全与冻结

---

## 2. 设计目标

意图识别是客服 Agent 的路由入口，职责包括：

+ 判断用户当前在做什么
+ 提取推进任务所需的关键参数
+ 判断是否需要追问
+ 判断是否发生话题切换
+ 判断是否属于高风险场景
+ 判断用户当前情绪以及是否出现情绪升级

统一输出结构建议如下：

```json
{
  "intent": "logistics",
  "slots": {
    "order_id": "12345"
  },
  "confidence": 0.91,
  "needs_clarification": false,
  "risk_level": "low",
  "emotion": {
    "primary": "neutral",
    "confidence": 0.76,
    "trend": "stable"
  },
  "is_intent_shift": false,
  "candidate_intents": [
    "logistics",
    "aftersale_refund"
  ]
}
```

字段说明：

+ `intent`：当前主意图
+ `slots`：当前轮提取出的关键参数
+ `confidence`：识别置信度
+ `needs_clarification`：是否需要追问
+ `risk_level`：风险等级
+ `emotion`：当前识别到的主要情绪及变化趋势
+ `is_intent_shift`：是否发生主意图切换
+ `candidate_intents`：候选意图

说明：

+ `candidate_intents` 是当前最可能的几个意图候选
+ 其作用主要是支持意图澄清、路由兜底和日志分析
+ 它不是并行执行列表，MVP 阶段只执行最终确定的 `intent`

## 3. 主意图与子意图

MVP 阶段建议使用可执行动作命名（与 `config/intent_schemas.yml` 对齐）：

+ `order_query`
+ `logistics`
+ `aftersale_refund`
+ `complaint`
+ `handoff_service`
+ `chitchat`
+ `unrecognize`
+ `unsupported_biz`

> 注：`aftersale_refund` 覆盖退款/退货/换货咨询，`chitchat` 覆盖问候，
> 不作进一步子意图拆分；详细子意图见 `config/intent_rules.yml`。

设计规则：

+ `main_intent` 决定主链路
+ `sub_intent` 决定链路内当前分支或当前轮补槽动作
+ 业务对象不是主意图本身，例如“订单”通常是 `slot` 或上下文对象，而不是 `intent`

示例：

+ "订单12345怎么还没到" -> `main_intent=logistics`
+ "那我不要了，想退货" -> `main_intent=aftersale_refund`，并从上下文继承 `order_id`

## 4. 分层路由设计

本项目现阶段采用“规则 + LLM”的两层路由方案：

### 4.1 规则层

处理高频、刚性、可强匹配的请求：

+ 转人工
+ 明确查订单
+ 明确查物流
+ 投诉升级
+ 禁用操作

### 4.2 LLM 路由层

处理复杂场景：

+ 多轮上下文依赖
+ 用户改口
+ 歧义
+ 多意图
+ 风险判断
+ 是否追问
+ 情绪识别与情绪升级判断

执行顺序：

`规则层 -> LLM 路由层`

失败处理：

+ `confidence` 低于阈值时不直接执行
+ 缺少必要槽位时优先追问
+ 连续多轮无法判断时转人工

### 4.3 何时进行意图澄清

意图澄清只在“无法安全确定当前主链路”时触发。

触发条件：

+ `candidate_intents` 之间差距过小，无法稳定确定 `main_intent`
+ 当前输入可能映射到多个主链路
+ 当前输入强依赖上下文，但现有上下文无法唯一解释
+ 高风险链路下仍存在目标歧义

不触发意图澄清的情况：

+ `main_intent` 已明确，只是缺少 `slot`
+ 当前只是链路内分支推进
+ 当前只是工具执行前参数不全

判断边界：

+ 不知道“用户现在想做什么” -> **意图澄清**
+ 知道“想做什么”，但不知道“参数是什么” -> **slot 澄清**
+ 知道目标和参数，只差规则依据 -> **RAG**
+ 知道目标和参数，只差实时业务数据 -> **工具调用**

## 5. Slot Schema 设计

本项目采用 `intent-driven slot schema` 作为主方案。

核心规则：

+ `main_intent` 决定链路级关键参数
+ `sub_intent` 决定当前轮补哪个参数或走哪个分支
+ `slot filling` 面向任务参数抽取，而不是通用实体识别

> **Slot Schema 来源**：MVP 阶段 slot schema 从 `config/intent_schemas.yml` 加载，
> 由 `IntentSchemaRegistry` 解析并缓存，不直接依赖数据库表 `intent_definitions`。
> `intent_definitions` 表后续可用于运营后台管理，但运行期以 YAML 为权威来源。

每个 `main_intent` 绑定一份 `slot schema`，至少包含：

+ `required_slots`
+ `optional_slots`
+ `slot_types`
+ `inheritable`
+ `overwritable`
+ `clarification_order`
+ `completion_rule`

示例：

```json
{
  "intent_code": "aftersale_refund",
  "required_slots": ["order_id", "refund_reason"],
  "optional_slots": ["product_name", "evidence"],
  "slot_types": {
    "order_id": "string",
    "refund_reason": "string",
    "product_name": "string",
    "evidence": "list[string]"
  },
  "inheritable": ["order_id", "product_name"],
  "overwritable": ["order_id", "refund_reason"],
  "clarification_order": ["order_id", "refund_reason", "evidence"],
  "completion_rule": "required_slots 全部齐全后进入下一阶段"
}
```

定义原则：

+ 只定义链路级关键参数
+ 不把所有可见实体都纳入主 schema
+ `tool schema` 仅用于执行前校验，不作为主 slot 定义来源

## 6. DST 与状态对象

建议维护如下状态对象：

```json
{
  "current_intent": "aftersale_refund",
  "stage": "collecting_info",
  "slots": {
    "order_id": "12345"
  },
  "missing_slots": [
    "refund_reason"
  ],
  "confirmed_slots": [
    "order_id"
  ],
  "topic_changed": false,
  "emotion": {
    "primary": "anxious",
    "confidence": 0.82,
    "trend": "escalating"
  },
  "summary": "用户正在咨询订单12345的退款流程"
}
```

字段说明：

+ `current_intent`：当前主任务
+ `stage`：当前阶段
+ `slots`：当前已知参数
+ `missing_slots`：仍缺失的必填参数
+ `confirmed_slots`：已确认参数
+ `topic_changed`：是否切话题
+ `emotion`：当前会话情绪状态，用于调整回复策略和转人工判断
+ `summary`：当前任务摘要

### 6.1 情绪识别设计

情绪识别的目标不是单独做情感分析，而是为客服链路提供策略信号，判断是否需要调整语气、减少追问、优先安抚或提前转人工。

MVP 阶段建议采用少量高价值标签：

+ `neutral` 中立
+ `confused` 困惑
+ `anxious` 焦虑
+ `angry` 生气
+ `urgent` 紧急
+ `happy` 快乐

识别输入建议同时参考三类信号：

+ 词汇信号：如“太慢了”“没人处理”“我要投诉”
+ 语气信号：重复问号、感叹号、强烈否定、连续催促
+ 上下文信号：多轮未解决、工具失败后情绪升级、反复要求处理结果

建议采用“规则优先 + LLM 补充”的识别方式：

+ 规则层优先识别明显投诉、催促、不满表达
+ LLM 层处理复杂语境下的焦虑、困惑、情绪升级判断
+ 最终只保留当前主情绪、置信度和趋势，避免把情绪识别做成一个过重的分类系统

情绪识别结果主要用于：

+ 调整回复语气和解释密度
+ 控制澄清次数，避免对高压情绪用户反复追问
+ 作为转人工和投诉升级的辅助信号

## 7. 状态更新与状态补全

状态更新采用固定流程：

1. 识别 `main_intent`
2. 读取该 `main_intent` 的 `slot schema`
3. 只抽取 schema 中定义的 slot
4. 将本轮 `new_slots` 合并进 `state.slots`
5. 重新计算 `missing_slots`
6. 若缺失则按 `clarification_order` 追问
7. 若必填槽位齐全，则推进到下一阶段

更新规则：

+ 同链路补信息：合并 slot
+ 用户改口：若 slot 可覆盖，则覆盖旧值
+ 可继承 slot：若上下文已有可信值，则直接补全
+ 链路切换：冻结旧任务状态，初始化新链路状态

运行示例：

+ Turn1: "我想退货" -> `aftersale_refund`，缺 `order_id/refund_reason`
+ Turn2: "订单12345" -> 补 `order_id`
+ Turn3: "因为尺码不合适" -> 补 `refund_reason`
+ 结果：`missing_slots=[]`，状态推进到下一阶段

### 7.1 自动人机切换

自动人机切换采用“规则优先 + 状态驱动 + 风险兜底”。

核心原则：

+ 能安全推进则继续由机器人处理
+ 无法安全推进、推进成本过高或风险过高时，自动转人工

建议触发条件：

+ 用户主动要求人工
+ 连续 `2~3` 次意图澄清仍未收敛
+ 连续 `2~3` 次 `slot` 澄清仍未补齐必要参数
+ 高风险或超权限场景
+ 工具调用连续失败
+ 情绪升级、投诉或明显不满

不建议在以下情况下立即转人工：

+ `main_intent` 已明确，只是一次普通 `slot` 缺失
+ 当前链路仍能通过一次澄清继续推进
+ FAQ / 规则问题仍可通过知识检索解决

建议在 `policy layer` 统一输出：

+ `continue_bot`
+ `ask_intent_clarification`
+ `ask_slot_clarification`
+ `handoff_human`

转人工时应附带：

+ 当前 `main_intent`
+ 当前 `stage`
+ 已收集 `slots`
+ `missing_slots`
+ 当前 `emotion`
+ 当前摘要 `summary`
+ 最近失败原因
+ 转人工原因 `handoff_reason`

> **`handoff_reason` 标准枚举**（MVP 阶段）：
> + `user_request` — 用户主动要求人工
> + `low_confidence` — 意图识别连续低置信度
> + `slot_clarification_failed` — 连续多次 slot 澄清仍未补齐
> + `emotion_escalation` — 情绪升级/投诉
> + `tool_failure` — 业务工具连续调用失败
> + `risk_high` — 高风险操作不允许自动执行
> + `unrecognize` — 无法识别的用户意图
> + `unsupported_biz` — 超出业务范围的意图

## 8. 意图切换与状态冻结

当 `main_intent` 发生切换时，应将旧意图对应的任务状态冻结并存档，再初始化新意图状态。

目标：

+ 防止旧任务上下文丢失
+ 支持后续切回旧任务
+ 支持转人工时传递完整任务轨迹
+ 便于审计与问题回放

冻结的对象不是整段对话，而是任务状态快照，建议包含：

+ 旧意图
+ 当时阶段
+ 已收集的 `slots`
+ `missing_slots`
+ 状态摘要
+ 冻结原因

示例：

```json
{
  "intent": "logistics",
  "stage": "done",
  "slots": {
    "order_id": "12345"
  },
  "summary": "用户已查询订单12345物流",
  "archived_reason": "intent_shift_to_aftersale_refund"
}
```

切换处理流程：

1. 判断发生 `main_intent shift`
2. 冻结旧状态
3. 初始化新意图状态
4. 从旧状态中继承允许继承的 slot
5. 重新计算新链路的 `missing_slots`

说明：

+ `sub_intent` 变化通常不需要冻结旧状态
+ 只有 `main_intent` 切换才触发状态冻结

---

## 9. 与工具调用的职责边界

本项目中两者分工明确：

+ `slot schema`：负责任务推进，决定当前主链路需要抽取哪些参数
+ `tool schema`：负责执行前校验，确保工具调用参数完整且类型正确

主流程采用：

`intent -> slot schema -> state update -> policy decision -> tool call(if needed)`

不采用“先判断工具，再根据工具参数反推 slot”作为主方案，以减少额外推理和工具解析复杂度。

---

## 10. 最终建议

本项目的意图识别建议正式定版如下：

+ 采用 `main_intent + sub_intent + DST`
+ 路由采用 `规则 + LLM` 分层策略
+ `slot schema` 由 `main_intent` 驱动
+ 情绪识别作为路由和策略层辅助信号进入状态对象
+ 状态更新采用“抽取 -> 合并 -> 补全 -> 推进”固定流程
+ `main_intent` 切换时冻结旧状态，并按规则继承可复用 slot

该方案适合客服 Agent 的多轮对话、链路切换和后续审计需求。
