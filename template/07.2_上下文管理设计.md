# 客服 Agent MVP 上下文管理设计
合理使用状态变量，避免不必要的大量历史数据
## 1. 文档目标

本文档用于细化客服 Agent MVP 的上下文管理设计，重点覆盖：

+ 上下文组成
+ 状态对象
+ 执行态上下文
+ 状态更新与状态补全
+ 上下文控制策略
+ 摘要窗口机制

---

## 2. 设计目标

上下文管理的目标是向模型提供“当前决策所需的最小充分信息”。

---

## 3. 上下文组成

建议将上下文拆成四层：

+ 最近几轮原始对话
+ 当前结构化状态
+ 执行轨迹与最近动作结果
+ 当前任务相关知识片段
+ 必要的长期记忆

---

## 4. 推荐上下文输入结构

建议提供给 LLM 的上下文包括：

+ 最近 3 到 6 轮用户和系统对话
+ 当前会话状态对象
+ 当前执行态上下文（最近动作、动作历史、当前表单状态）
+ 当前检索结果或工具结果
+ 必要的业务规则

不建议直接提供：

+ 全量长对话历史
+ 无筛选的原始工具结果
+ 与当前任务无关的历史记忆

这篇 CoTA 的 DST 文档给出的关键启发是：上下文管理不应只保存“业务状态”，还要保存“执行状态”。也就是说，不仅要知道用户当前要做什么，还要知道系统刚刚执行了什么、输出了什么、当前由哪个动作接管。

---

## 5. 会话状态对象

建议维护如下状态对象：

```json
{
  "current_intent": "refund_consult",
  "stage": "collecting_info",
  "slots": {
    "order_id": "12345"
  },
  "missing_slots": [
    "refund_reason"
  ],
  "confirmed_slots": [
    "order_id"
  ],
  "current_action": "RefundForm",
  "latest_action_result": null,
  "topic_changed": false,
  "summary": "用户正在咨询订单12345的退款流程"
}
```

字段说明：

+ `current_intent`：当前主任务
+ `stage`：当前任务阶段
+ `slots`：当前已知参数
+ `missing_slots`：继续推进任务所需但尚缺失的信息
+ `confirmed_slots`：已确认无歧义的信息
+ `current_action`：当前负责推进任务的动作或节点
+ `latest_action_result`：最近一次动作执行结果的摘要
+ `topic_changed`：是否发生话题切换
+ `summary`：对当前任务的简要摘要

### 5.1 执行态上下文

借鉴 CoTA 的 DST，建议在业务状态之外补一层“执行态上下文”，用于记录 Agent 自己的执行轨迹，而不仅是用户业务语义。

建议至少维护以下字段：

+ `history_messages`：用户与系统最近若干轮原始对话
+ `action_history`：动作执行历史，记录名称、结果摘要、时间戳
+ `current_action_name`：当前正在执行的动作
+ `latest_action_name`：最近一次执行的动作
+ `latest_action_result`：最近动作的输出摘要
+ `current_form_name`：当前正在处理的表单或任务节点
+ `current_form_slot_states`：当前表单槽位状态
+ `knowledge`：与当前任务相关的知识检索结果

在客服场景中，这些字段分别回答不同问题：

+ 业务状态回答“用户现在想做什么”
+ 执行态上下文回答“系统刚才做了什么、接下来哪个节点该继续”

例如：

+ 用户问“订单12345到哪了” -> 业务状态进入 `query_logistics`
+ 系统执行物流查询工具 -> 执行态记录 `latest_action_name=query_logistics_tool`
+ 用户接着说“那我要退货” -> 业务状态切到 `refund_consult`，执行态也需要记录当前接管节点已经从物流查询切到退款表单

### 5.2 状态变量化注入

CoTA 的另一个重要做法是：不要把所有状态打包成一个超大 JSON 直接喂给模型，而是拆成可按需注入的状态变量。

建议在本项目中也采用类似方式，按链路选择性注入：

+ 回复生成节点：`summary + history_messages + latest_action_result + knowledge`
+ 路由节点：`action_descriptions + action_history + summary + latest_user_query`
+ 表单更新节点：`current_form_slot_states + current_form_name + latest_user_query`
+ 工具执行后回复节点：`latest_action_result + current_intent + slots + summary`

这样做的好处是：

+ 减少无关上下文进入 prompt
+ 降低 token 消耗
+ 提高不同节点的上下文可解释性
+ 更方便排查“为什么模型在这一轮做了这个决定”

---

## 6. 状态更新与状态补全

状态更新采用“读 schema -> 抽 slot -> 合并状态 -> 计算缺失 -> 决定下一步”的固定流程：

1. 识别 `main_intent`
2. 读取该 `main_intent` 对应的 `slot schema`
3. 只抽取 schema 中定义的 slot
4. 将本轮 `new_slots` 合并进 `state.slots`
5. 重新计算 `missing_slots`
6. 若缺失则按 `clarification_order` 追问
7. 若必填槽位齐全，则推进到下一阶段

更新规则：

+ 同链路补信息：合并 slot
+ 用户改口：若 slot 可覆盖，则覆盖旧值
+ 可继承 slot：若上下文已有可信值，则直接补全
+ 链路切换：冻结旧任务状态，初始化新链路状态
+ 动作执行后：同步更新 `latest_action_name / latest_action_result / action_history`
+ 表单推进后：同步更新 `current_form_name / current_form_slot_states`

运行示例：

+ Turn1: “我想退货” -> `refund_consult`，缺 `order_id/refund_reason`
+ Turn2: “订单12345” -> 补 `order_id`
+ Turn3: “因为尺码不合适” -> 补 `refund_reason`
+ 结果：`missing_slots=[]`，状态推进到下一阶段

### 6.1 状态生命周期

借鉴 CoTA 的状态生命周期，建议按以下顺序更新上下文：

1. 会话开始：初始化状态对象，必要时加载历史摘要和长期记忆
2. 用户输入：更新 `latest_user_query`、最近消息窗口
3. 路由决策：更新 `current_action_name`、候选动作信息
4. 动作执行：写入 `latest_action_result`，并将结果摘要追加到 `action_history`
5. 表单处理：更新 `current_form_slot_states` 和任务阶段
6. 会话结束或轮次结束：必要时生成摘要并持久化状态

这意味着“上下文管理”不是单次拼 prompt，而是一套贯穿整轮对话的状态更新机制。

---

## 7. 上下文控制策略

建议结合以下机制：

+ 滑动窗口：保留最近若干轮原始对话
+ 摘要压缩：对较长历史生成任务摘要
+ 状态显式化：优先读状态对象，而不是让模型自己总结
+ 上下文隔离：工具原始响应与用户可见内容分离
+ 状态变量按需注入：不同节点只读取自己需要的状态

### 7.1 执行轨迹与原始结果分层

CoTA 的 DST 强调保留执行轨迹，但在客服 Agent 中不建议把完整工具原始输出直接放进主上下文。

建议分三层保存：

+ `raw_result`：工具原始返回，仅进入日志、审计和排障链路
+ `latest_action_result`：面向主推理链路的结果摘要
+ `action_history`：记录动作名称、结果摘要、时间戳，供路由和回复节点回溯

这样既保留执行过程，又避免原始结果污染上下文窗口。

---

## 8. 摘要窗口策略

客服 Agent 建议采用“摘要窗口”管理长对话上下文。

摘要窗口的基本形式为：

+ `running_summary`：对较早历史对话的增量摘要
+ `recent_messages`：最近若干轮保留原文
+ `state`：当前任务的结构化状态对象

---

## 9. 何时触发摘要压缩

建议采用“硬阈值 + 软阈值”组合触发：

### 9.1 软触发条件

+ 历史上下文 token 占可用预算约 `60%`
+ 历史轮次超过 `8` 轮
+ 当前任务状态较稳定
+ 早期消息中已经出现大量重复确认、寒暄或低价值内容

### 9.2 硬触发条件

+ 历史上下文 token 占可用预算约 `80%`
+ 历史轮次超过 `12` 轮

### 9.3 暂缓压缩条件

+ 刚发生 `intent shift`
+ 最近刚补齐关键槽位
+ 最近刚发生高风险确认
+ 最近刚返回关键工具结果，且尚未完成状态更新

---

## 10. 摘要窗口的推荐默认值

MVP 阶段建议默认如下：

+ 最近原文窗口保留：`6` 轮
+ 软触发轮数：`8` 轮
+ 硬触发轮数：`12` 轮
+ 历史上下文软触发：可用上下文预算的 `60%`
+ 历史上下文硬触发：可用上下文预算的 `80%`

如果最近发生了以下事件，可临时扩大原文窗口到 `8` 轮：

+ 意图切换
+ 关键槽位改写
+ 高风险确认
+ 关键工具调用结果刚返回

---

## 11. 摘要更新方式

建议采用“增量摘要”，而不是每次重写全量历史摘要。

推荐方式：

1. 永远保留最近 `N` 轮原始消息
2. 将“即将移出窗口的旧消息”合并进 `running_summary`
3. 新摘要替换旧摘要
4. 后续上下文使用：`running_summary + recent_messages + state`

不建议：

+ 每轮都重新总结全部历史
+ 每次固定删除一半消息
+ 摘要后只保留 1 到 2 轮原文

---

## 12. 摘要内容要求

摘要应保留当前任务推进所需的关键事实：

+ 历史主意图和当前主意图
+ 已确认的关键槽位
+ 用户明确表达的诉求
+ 工具调用得到的重要结果
+ 关键动作执行轨迹和最近动作结果
+ 已给出的规则说明或承诺
+ 当前仍待补充的信息
+ 转人工原因或风险标记

推荐摘要示例：

+ “用户先查询订单12345物流，后切换为退款咨询；订单号已确认，退款原因为尺码不合适；已告知退货规则，当前仍缺少图片凭证”

---

## 13. 参考与落地结论

参考框架：

+ [CoTA DST 文档](https://cotaai.github.io/cota/concepts/dst.html)

从 CoTA 的 DST 设计中，当前文档建议吸收三点：

+ 上下文管理不仅要存业务状态，也要存执行状态
+ 状态应拆成可按需注入的变量，而不是总是传递完整大对象
+ 状态需要有明确生命周期，支持跨轮更新、摘要压缩和持久化恢复

因此，本项目中的上下文管理最终应理解为：

+ `业务状态`：当前意图、阶段、槽位、摘要
+ `执行状态`：动作历史、最近动作结果、当前表单状态
+ `注入规则`：不同节点按需取用不同状态变量
