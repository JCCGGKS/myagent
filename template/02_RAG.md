# RAG

## 是什么
检索增强生成：
+ 检索。从外部知识库检索和query最相关的片段
+ 增强。注入用户提示词作为上下文的一部分
+ 生成。基于增强后的上下文生成答案

## 解决的问题
+ 知识过期。喂给llm的训练数据不是实时的
+ 知识空白。喂给llm的训练数据不包含私有数据
+ 模型幻觉。查询真实数据避免llm胡编乱造

## 大致流程
+ 存储
    + 文档加载
    + 文档切块(上下文窗口有限)
    + embedding向量化
    + 存入向量库
+ 检索
    + query改写。基于上下文修改query，比如指代消解
    + 检索
    + 增强提示词
    + 生成答案

## rag && 微调
微调是让llm继续学习从而修改模型参数，成本高
rag通过外接知识手册扩充知识面

## 如何动态持续更新
先删除旧文档对应的所有chunk，然后对新文档重新切分-向量化-入库
+ 如何知道文档发生了变化。内容hash和文档ID\chunk ID列表一起保存；记录文档的最后修改时间，发生变化就计算hash值和原hash值比较
+ 文档ID和chunk ID的设计。让chunk ID带上文档ID作为前缀，比如`product_manual_v3_chunk_001`，这样可以按照前缀批量查找和删除对应的所有chunk；使用`metadata`
+ 变更感知方式。定时轮询；事件驱动


## 落地较难的地方
+ 文档预处理
+ 检索评估和调优
+ 生成评估

## 图数据库
当业务问题设计多个实体之间的关联推理时，可以考虑引入图数据库增强
向量检索的局限在于只能做单跳检索，找和问题直接相关的文档，没办法沿着实体之间的关系链做推理

## 如何避免幻觉
产生幻觉的原因
+ 检索没有召回到相关内容(检索层)
+ 召回到相关内容但是llm没有遵循(生成层)

如何避免幻觉
+ 检索层。rerank分数低于阈值直接拒绝回答
+ 生成层。加强prompt约束，明确告知llm只能根据检索到的资料进行回答，不能胡编乱造

# 文本分块
分块的目的

+ 满足模型上下文限制。嵌入模型有严格的输入长度上限。llm也有严格的上下文窗口限制
+ 块并非越大越好。大多数嵌入模型都基于`Transform`编码器。文本块越长，包含的语义点越多，单一向量承载的信息越稀释，导致关键细节被模糊，降低检索精度
    + 分词(`Tokenization`)。将输入的文本块分解成一个一个token
    + 向量化(`Vectorization`)。 `Transformer`为每个`Token`生成一个高维向量表示
    + 池化(`Pooling`)。 通过某种方法，将所有`Token`的向量压缩成一个单一的向量，代表整个文本块的语义
+ `Lost in Middle`。 研究表明，当llm处理非常长的、充满大量信息的上下文时，更倾向于记住开头和结尾的信息，从而忽略中间的部分
+ 主题稀释导致检索失败。好的文本块应该聚焦于明确单一的主题。如果一个块包含太多不相关的主题，语义会被稀释，导致检索时无法精确匹配

基础分块策略(`LangChain`)
+ 固定大小分块。先按照段落分割，然后将分割后的段落依次合并，超过`chunk_size`时形成新块，并通过重叠机制(`chunk_overlap`)保持上下文连续性。（并非严格的固定大小分块）
    + 优先保持段落完整性。
    + 处理超长段落。如果单个段落超过`chunk_size`，系统会发出警告但仍将其作为完整块保留
    + 应用重叠机制。通过`chunk_overlap`保证块之间的内容连贯性
    + 实现简单，处理速度快且开销小。可能会在语义边界处切断文本，影响内容完整性和连贯性
+ 递归字符分块。采用一组有层次结构的分隔符(如段落、句子、单词)进行递归分割，平衡语义完整性和块大小控制
    + 首先尝试使用最高优先级的分隔符切分文本，块仍然过大会继续对剩下的块应用下一级分隔符
+ 语义分块。在语义主题发生显著变化的地方切分
    + 
+ 基于文档结构的分块。对于具有明确结构标记的文档格式(如Markdown、HTML、LaTex)，可以利用这些标记实现更智能年呢个、更符合逻辑的分割
    + Markdown文档：先按照标题分组，再按需细分
    + 将文档按照标题分割成若干个大的、带有元数据的逻辑块
    + 对较大的逻辑块应用其他分割方案，比如递归字符分割

# 向量化
语义检索的基础，决定检索质量的关键
一个优秀的Embedding模型能够精确捕捉问题和文档之间的深层语义联系，
## 向量嵌入-Embedding
向量嵌入将高维的数据对象(文本、图像、音频、视频)转换为低维、稠密的连续数值向量，并嵌入原始数据的所有关键信息
embedding产生的向量不是随机数值的堆砌，而是语义的数学编码

### 关键度量
+ 余弦相似度(Cosine Similarity)。 计算两个向量夹角的余弦值。值越接近1,方向越一致，语义越相似。
+ 点积(Dot Product)。两个向量的乘积和。向量归一化之后，等价于余弦相似度。
+ 欧式距离(Euclidean Distance)。 计算两个向量在空间中的直线距离。

### 技术发展
+ 静态词嵌入：上下文无关的表示。
    + 代表模型：`Word2Vec`,`GloVe`
    + 主要原理：为词汇表中的每个单词生成一个固定的、与上下文无关的向量。
    + 局限性：无法处理一词多以的问题。
+ 动态上下文嵌入。 
    + BERT模型利用Transformer的编码器，通过掩码语言模型MLM等自监督任务进行预训练，生成深度上下文相关的嵌入。
+ 满足RAG。 
    + 领域自适应能力。通过微调或使用指令适应特定领域的术语和语义
    + 多粒度与多模态支持。能够处理不同长度和类型的输入数据
    + 检索效率与混合检索。支持混合检索的嵌入模型`BGE-M3`

### 嵌入模型训练原理
主要训练任务

+ 掩码语言模型。
    + 过程：随机将输入句子中15%的词元(TOKEN)替换为一个特殊的标记，让模型预测被掩盖的原始词元
    + 目标：通过这个任务，模型被迫学习每个词元及其上下文之间的关系
+ 下一句预测。
    + 过程：构造训练样本，每个样本包含两个句子A和B；其中50%的样本，B是A真实的下一句，另外50%的样本，B是从语料库中随机抽取的句子；让模型判断B是否是A的下一句
    + 目标：让模型学习句子之间的逻辑关系

效果增强策略-引入更针对性的训练策略
+ 度量学习
+ 对比学习

### 嵌入模型选型

从MTEB排行榜开始
[MTEB (Massive Text Embedding Benchmark)](https://huggingface.co/spaces/mteb/leaderboard)
选择开源嵌入式模型时需要权衡的四个核心维度
+ 横轴-模型参数量(Number of Parameters)。 通常，参数量越大的模型，潜在能力越强，对计算资源要求也越高
+ 纵轴-平均任务得分(Mean Task Score)。 这个分数是模型在分类、聚类、检索等一系列标准NLP任务上的表现，分数高，模型的通用语义理解能力越强
+ 气泡大小-嵌入维度(Embedding Size)。 气泡越大，维度越高，理论上能编码更多丰富的语义细节，也会占用更多存储和计算资源
+ 气泡颜色-最大处理长度(Max Tokens)。 颜色越深，表示模型能处理的Token数量越多，对长文本适应性越好

关键评估维度
查看榜单时，除了分数，还需要关注以下几个维度
+ 任务(Task)。 对于RAG应用需要关注模型在Retrieval检索任务下的排名
+ 语言(Language)。 模型是否支持业务数据所使用的语言
+ 模型大小(Size)。 模型越大，性能越好，对硬件性能要求也越高，推理速度越慢。根据部署环境和性能权衡
+ 维度(Dimensions)。 维度越高，能编码的信息越丰富，会占用更多存储空间和计算资源
+ 最大Token数(Max Tokens)。 决定模型能处理的文本长度上限。设计文本分块策略时必须考虑的依据，块的大小不能超过该限制
+ 得分与机构(Score & Publisher)。 知名机构发布的模型通常质量更有保障
+ 成本(Cost)。 API服务的模型，需要考虑调用成本； 自部署开源模型，需要评估对硬件资源的消耗


迭代测试与优化
不要只依赖公开榜单做最终决定
+ 确定基线，根据上述维度，选择几个符合要求的模型作为初始基准模型
+ 构建私有评测集，根据真实业务数据，手动创建一批高质量的评测样本，每个样本包含一个典型用户问题和标准答案
+ 迭代优化，使用基线模型在私有评测集上运行，评估其召回的准确率和相关性。

### 多模态嵌入
目的是将不同类型的数据(如图像和文本)映射到同一个共享的向量空间
实现这一目标的关键，在于解决跨模态对齐的挑战

CLIP模型，在图文多模态领域，OpenAI的CLIP(Contrastive Language-Image Pre-training)是一个很有影响力的模型，为多模态嵌入定义了一个有效的范式。
采用双编码器架构，包含一个图像编码器和一个文本编码器，分别将图像和文本映射到同一个共享的向量空间
常用多模态嵌入模型(`bge-visualized-m3`)

## 向量数据库

### 向量数据库的作用
向量数据库的核心价值在于高效处理海量高维向量的能力

+ 高效的相似性搜索。利用专门的索引技术(如HNSW、IVF)，能够在数十亿级别的向量中实现毫秒级的ANN查询，快速找到与给定查询最相似的数据
+ 高维数据存储与管理。 专为存储高维向量(通常维度成百上千)而优化，支持对向量数据进行增删改查等基本操作
+ 丰富的查询能力。除了基本的相似性搜索，还支持按标量字段过滤查询
+ 可扩展与高可用。现代向量数据库通常采用分布式架构，具备良好的水平扩展能力和容错性
+ 数据与模型生态集成。与主流的AI框架和机器学习工作流无缝集成，简化应用开发流程

向量数据库 VS 传统数据库

| 维度         | 向量数据库                     | 传统数据库 (RDBMS)                                   |
| ------------ | ------------------------------ | ---------------------------------------------------- |
| 核心数据类型 | 高维向量 (Embeddings)          | 结构化数据 (文本、数字、日期)                        |
| 查询方式     | 相似性搜索 (ANN)               | 精确匹配                                             |
| 索引机制     | HNSW, IVF, LSH 等 ANN 索引     | B-Tree, Hash Index                                   |
| 主要应用场景 | AI 应用、RAG、推荐系统、图像/语音识别 | 业务系统 (ERP, CRM)、金融交易、数据报表              |
| 数据规模     | 轻松应对千亿级向量             | 通常在千万到亿级行数据，更大规模需复杂分库分表       |
| 性能特点     | 高维数据检索性能极高，计算密集型 | 结构化数据查询快，高维数据查询性能呈指数级下降       |
| 一致性       | 通常为最终一致性               | 强一致性 (ACID 事务)                                 |


### 工作原理
在RAG系统中，向量一般通过嵌入模型将原始数据转换为高维向量表示

向量数据库通常采用四层架构
+ 存储层。负责存储向量数据和元数据，优化存储效率并支持分布式存储
+ 索引层。维护索引算法(HNSW、LSH、PQ等)，负责索引的创建与优化，并支持索引调整
    + 基于树的方法。
    + 基于哈希的方法。
    + 基于图的方法。
    + 基于量化的方法。
+ 查询层。处理查询请求，支持混合查询并实现查询优化
+ 服务层。管理客户端连接，提供监控和日志能力，实现安全管理

### 主流向量数据库介绍

区分“支持向量检索”和“原生支持混合检索”很重要。混合检索通常指将`dense vector`语义检索与`BM25/稀疏向量`关键词检索进行融合，再通过`RRF`、加权融合或`rerank`得到最终结果。客服、企业知识库、规则问答这类场景通常更适合混合检索，而不是纯向量召回。

| 产品 | 开源/托管 | 核心架构 | 是否原生支持混合检索 | 核心特性 | 适用场景 | 参考URL |
| ---- | ---- | ---- | ---- | ---- | ---- | ---- |
| PostgreSQL + pgvector | 开源 | 关系型数据库 + 向量扩展 | 支持，但依赖 PostgreSQL `full-text search` 与 `pgvector` 组合实现 | 一套库同时承载业务数据、元数据、全文检索、向量检索；适合低运维成本方案 | 中小规模 RAG、客服系统、需要事务与结构化数据共存的业务系统 | `pgvector`：<https://github.com/pgvector/pgvector>；PostgreSQL FTS：<https://www.postgresql.org/docs/current/textsearch.html> |
| Pinecone | 完全托管云服务(Serverless) | 存储计算分离、自动扩缩容、负载均衡 | 支持 | 托管化运维、支持 dense+sparse 混合检索、企业级 SLA | 企业生产环境、高并发、大规模云端部署 | <https://docs.pinecone.io/guides/search/hybrid-search> |
| Milvus | 开源分布式 | 分布式架构，支持 GPU 加速 | 支持 | 多类索引算法、dense/sparse/multi-vector 检索、完善开源生态 | 大规模集群部署、高性能检索、自定义二次开发 | <https://milvus.io/docs/hybrid_search_with_milvus.md> |
| Qdrant | 开源 | Rust 底层开发 | 支持 | 支持 dense + sparse + fusion，多阶段查询能力较强 | 性能敏感业务、高并发、中小规模集群部署 | <https://qdrant.tech/documentation/search/hybrid-queries/> |
| Weaviate | 开源 | GraphQL API 驱动 | 支持 | 内建 hybrid search，支持 BM25 与向量融合，多模态能力较强 | AI 应用开发、多模态数据处理、快速落地 RAG 业务 | <https://docs.weaviate.io/weaviate/search/hybrid> |

| FAISS | 开源库 | 本地 ANN 检索库 | 不支持 | 专注高性能向量相似度搜索，本身不负责 BM25、稀疏索引或融合排序 | 自定义检索系统底座、实验验证、纯向量检索场景 | <https://faiss.ai/> |

补充说明：
+ `PostgreSQL + pgvector` 的混合检索不是“一个黑盒 API 自动完成”，而是工程上非常实用的组合方案：用 PostgreSQL 的全文检索做关键词召回，用 `pgvector` 做语义召回，再通过 SQL 层做 `RRF`、加权分数融合或交给 reranker 重排。优点是轻量、统一、好维护。
+ `Chroma` 更适合本地原型和教学，不应默认理解为具备与 `Qdrant`、`Weaviate` 同等级别的原生 hybrid search 能力。
+ `FAISS` 是向量检索引擎，不是完整的向量数据库。若要做混合检索，通常需要额外接入 Elasticsearch、PostgreSQL FTS 或 BM25 实现，再自行融合结果。

场景选型指南
1. 新手入门、本地原型、小型项目：ChromaDB / FAISS
优势：与 LangChain/LlamaIndex 集成方便，部署简单，适合快速验证纯向量检索链路。
注意：如果业务明显依赖关键词精确命中，不建议长期停留在这类方案。

2. 轻量级生产、希望一套库搞定业务数据和检索：PostgreSQL + pgvector
优势：学习和运维成本低；适合客服、FAQ、工单、订单等结构化数据和检索能力并存的场景；天然支持事务、过滤、全文检索和向量检索组合。

3. 线上生产、百万级以上向量、高并发实时更新、复杂过滤与原生混合检索：Qdrant / Milvus / Weaviate / Pinecone(云托管)
优势：向量检索能力更完整，适合独立检索层或中大型生产环境。

4. 客服 Agent / 企业知识库的推荐顺序
+ 已有 PostgreSQL 技术栈：优先考虑 `PostgreSQL + pgvector`
+ 想独立部署轻量向量库：优先考虑 `Qdrant`
+ 数据规模更大、后续要扩展多模态或多向量：考虑 `Milvus` 或 `Weaviate`

## Milvus

核心组件
+ Collection(集合)
+ 索引(Index)

### Collection
+ Collection(集合)。 是所有数据的顶层容器，一个Collection可以包含多个Partition，每个Partition可以包含多个Entity
+ Partition(分区)。 将数据物理隔离，检索更高效
    + 每个Collection在创建时都会有一个名为_default的默认分区，可以根据业务需求创建更多的分区，将数据按特定规则存入不同分区
    + 目的：提升查询性能，数据管理
    + 一个Collection最多有1024个分区
+ Schema(模式)。 定义了数据都有哪些字段
    + 创建Collection之前，需先定义Schema，通常包含以下几类字段
    + 主键字段(Primary Key Field)。 每个Collection有且只有一个主键字段用于唯一标识实体
    + 向量字段(vector Field)。 存储核心的向量数据
    + 标量字段(Scalar Field)。 存储向量之外的元数据
+ Entity(实体)。 数据本身
+ Alias(别名)。 指向某个具体的Collection，实现数据更新时的无缝切换(只需要将Alias指向新的集合即可)

### 索引
索引通用组成模块
| 模块 | 是否可选 | 功能说明 |
| ---- | ---- | ---- |
| 数据结构 | 必选 | 索引骨架，定义向量组织形式，例：HNSW多层图结构 |
| 量化 | 可选 | 向量压缩手段，降低精度换取更小内存、更快计算 |
| 结果精炼 | 可选 | 粗筛候选集后二次精确计算，优化检索精度 |

字段索引分类
| 索引类型 | 作用 | 常用索引 | 说明 |
| ---- | ---- | ---- | ---- |
| 标量字段索引 | 加速元数据过滤 | INVERTED、BITMAP | 直接选用推荐类型即可 |
| 向量字段索引 | 核心向量检索 | FLAT/IVF/HNSW/DiskANN | 需要权衡查询速度、召回率、内存占用 |

主流向量索引算法详情
| 索引名称 | 核心原理 | 优点 | 缺点 | 适用场景 |
| ---- | ---- | ---- | ---- | ---- |
| FLAT（精确暴力检索） | 遍历全量向量，逐一枚举计算距离 | 召回率100%，结果绝对精准 | 查询速度慢、内存开销高 | 百万级以内小数据集、极致精度需求场景 |
| IVF系列（倒排索引，含IVF_FLAT/IVF_SQ8/IVF_PQ） | 聚类分桶；检索仅扫描少量相似桶，变体差异为是否开启向量量化压缩 | 大幅缩小检索范围，速度均衡，通用性强 | 召回无法达到100%，相似向量可能落在未检索分桶 | 通用大规模数据集、高并发吞吐量业务 |
| HNSW（多层近邻图索引） | 搭建多层邻近图，顶层稀疏图快速定位，底层稠密图精细检索 | 查询速度极快、召回表现优秀，适配高维向量、低延迟需求 | 内存占用极高，索引构建耗时久 | 实时推荐、在线搜索等强低延迟线上业务 |
| DiskANN（磁盘优化图索引） | 面向SSD高速磁盘优化的图结构索引 | 支持十亿级超大规模数据，无需全量载入内存，延迟可控 | 检索延迟高于纯内存索引 | 数据总量超出服务器内存上限的海量存储场景 |


如何选择索引
| 场景 | 推荐索引 | 备注 |
| ---- | ---- | ---- |
| 数据可完全载入内存，追求低延迟 | HNSW | 内存占用较大，但查询性能和召回率都很优秀。 |
| 数据可完全载入内存，追求高吞吐 | IVF_FLAT / IVF_SQ8 | 性能和资源消耗的平衡之选。 |
| 数据量巨大，无法载入内存 | DiskANN | 在 SSD 上性能优异，专为海量数据设计。 |
| 追求 100% 准确率，数据量不大 | FLAT | 暴力搜索，确保结果最精确。 |



### 检索
基础向量检索(ANN Search)
ANN利用提前构建好的索引，极速的从海量数据中找到与查询向量最相似的Top-K个结果，是一种在速度和精度之间取得极致平衡的策略


增强检索
+ 过滤检索(Filtered Search)。 先根据提供的过滤表达公(Filter)筛选出符合条件的实体，然后仅在这个子集内执行ANN检索，提高查询和精准度
+ 范围检索(Range Search)。 范围检索允许定义一个距离(或相似度)的阈值范围。Milvus会返回所有与查询向量的距离落在这个范围内的实体
+ 多向量混合检索(Hybrid Search)。针对不同的向量字段(一个用于文本语义的密集向量，一个用于关键词匹配的稀疏向量，一个用于图像内容的多模态向量)分别发起ANN检索请求
+ 分组检索(Grouping Search)。 分组检索允许指定一个字段对结果进行分组。Milvus会在检索后，确保返回的结果中每个组只出现一次(或指定的次数)，且返回的是该组内与查询最相似的那个实体

## 索引优化
### 上下文扩展

<font color=‘red’>句子窗口检索</font>

在RAG系统常常面临一个权衡问题：使用小块文本检索可以获得更高的精确度，但缺乏足够的上下文，可能导致LLM无法生成高质量的答案；大块文本虽然上下文丰富，容易引入噪音，降低检索相关性。
`LlamaIndex`提出了一种实用的索引策略---句子窗口检索(Sentence Window Retrieval)，巧妙结合两种方法的优点：在检索时聚焦高精度精确的单个句子，送入LLM生成答案前，智能的将上下文扩展回一个更宽的窗口，从而同时保证检索的准确性和生成的质量，主要思路如下
+ 索引阶段。在构建索引时，文档被分割成单个句子。每个句子作为单独的节点存入向量数据库并在元数据存储上下文窗口(句子原文中的前n个句子和后n个句子)
+ 检索阶段。用户发起查询时，对单一句子节点执行相似度搜索。精准定位与用户提问相关的句子
+ 后处理阶段。读取检索到的句子的元数据，并用元数据中存储的完整上下文窗口替换原来单一句子内容
+ 生成阶段。包含丰富上下文的节点传递给LLM




### 结构化索引
在索引文本块的同时，为其附加结构化的元数据，元数据可以是任何有助于筛选和定位信息的标签
+ 文件名
+ 文档创建日期
+ 章节标题
+ 作者
+ 任何自定义的分类标签

先元数据过滤，后在小范围内进行相似度搜索

基于多表格的递归检索
+ 创建两个独立的向量索引。
    + 摘要索引：为每个excel工作表(例如，1994年电影数据)创建一个非常简短的摘要性Document，例如此文档包含1994年的电影信息。所有摘要文档构建一个轻量级的向量索引。目的是为了充当路由器
    + 内容索引：将每个工作表的实际数据(例如，整个表格)转换为一个大的文本，并为其附加一个关键的元数据标签如{"sheet_name": "年份_1994"}。 然后为包含真实内容的文档构建一个向量索引
+ 执行两步查询。
    + 路由。摘要检索定位到sheet_name
    + 检索。元数据过滤sheet_name，只在正确的，经过筛选的数据范围内找到答案


这种先路由，后用元数据过滤并检索的方式，实现了跨多个数据源的查询能力

## 查询重构与分发
### query rewrite 
用户的问题往往口语化、模糊、带缩写，直接检索会降低召回率，查询重构将原始问题转化为更易于检索的问题

+ 直接改写。llm把口语化的表述转成更精准的表述
+ 多查询分解。如果问题比较复杂，直接用整个问题执行检索可能效果不好。分解技术的核心思想是将复杂问题拆解成多个简单的、更具体的子问题。然后针对每个子问题进行检索，最后将结果合并、去重。
+ HyDE(Hypothetical Document Embedings)。 解决的问题：用户的查询通常简短、关键词有限，数据库中存储的文档则内容丰富详实，两者在语义向量空间中可能存在鸿沟，导致检索效果不好。
    + 生成。llm根据查询生成一个详细的、理想的假设性答案文档。
    + 编码。对生成的假设文档转换为向量
    + 检索。使用假设文档向量，进行语义相似性检索
+ step-back Prompting。 具体问题向上抽象一层，检索更通用的背景知识
    + 抽象化。引导llm从用户的原始具体问题中，生成一个更高层次、更概括退步问题(探寻问题背后的原理或核心概念)
    + 推理。先获取退步问题的答案，将通用原理作为上下文，推理生成最终答案


### 查询路由
当系统接入多个不同的数据源或具备多种处理能力时，需要一个智能调度中心分析用户的查询，并动态选择最合适的处理路径
本质： 替代硬编码规则，通过语义理解，将查询分发到最合适的数据源或检索器。优化复杂RAG系统的关键技术

应用场景
+ 数据源路由。根据查询意图，将其路由到不同的知识库。比如，"最新的iphone有什么功能" -> 产品文档向量数据库
+ 组件路由。根据问题的复杂性，将其分配给不同的处理组件。比如，简单FAQ->向量检索 复杂操作->调用agent执行
+ 提示模板路由。为不同类型的任务选用不同的提示词模板。比如数学问题->分步思考

实现方法
+ 基于llm的意图识别。 
    + 定义清晰的路由选项
    + llm分析查询并输出决策标签
    + 代码根据标签调用相应的检索器或工具
+ 嵌入相似路由。不依赖llm进行分类。通过计算用户查询与预设的路由示例语句之间的向量嵌入相似度作出决策
    + 定义路由并描述向量化。 为每个路由创建一个详细的文本描述，并使用嵌入模型将其转换为向量，供后续相似度计算使用
    + 定义目标链。创建路由最终要分发到的目标处理链，并将路由名称和路由链的映射存储到dict
    + 定义路由函数。接收用户问题，计算与各路由描述的相似度，选择最相似的路由并调用
## 混合检索
### 稀疏向量
+ 稀疏向量/词法向量。基于词频统计的传统信息检索方法的数学表示，通常是一个维度极高(与词汇表大小相当)但绝大多数元素为零的向量。向量的权重计算方法是`TF-IDF`。 在信息检索领域，`BM25`是基于稀疏表示的排序算法之一
+ 可解释性强(每个维度表示一个确切的词语)，能够实现关键词的精确匹配，对于专业术语和特定名词的检索效果好。无法理解语义
+ 只记录非零维度，用字典存储

```json
// {索引:权重}
{
    "88": 1.2,
    "666": 0.8
}
// (维度,[索引列表],[值列表])
(50000,[88,666],[1.2,0.8])
```

+ [qdrant-bm25](https://ai.gitcode.com/hf_mirrors/Qdrant/bm25)
+ [wiki-bm25](https://en.wikipedia.org/wiki/Okapi_BM25)

### 密集向量
+ 密集向量/语义向量。通过深度学习模型学习到的数据的低维度、稠密的浮点数表示，旨在将原始数据映射到一个连续的、充满意义的语义空间
+ `Word2Vec` 、 `GloVe`、 基于`Transformer`的模型生成的嵌入
+ 每个维度都有数值，用数组存储
```json
[0.89,-0.12,0.77,...,-0.45]
```

`BGE-M3`作为向量生成器，能够同时生成稀疏向量和密集向量
### 混合检索
结合多种搜索算法，(常见的是稀疏:精确性和密集:范化性)，提升搜索结果相关性和召回率，解决单一检索技术的局限性

融合方法
+ 倒数排序融合(RRF)。 重点关注每个文档在各自结果集中的排名，一个文档在不同检索体系中的排名越靠前，最终得分就越高
+ 加权融合。先将不同检索体系的得分归一化处理，通过权重参数进行线性组合

## 元数据过滤器
构建向量索引时，会为文档块(`chunks`)附加元数据(`metadata`)，例如文档来源等

自查询检索器(`Self-Query Retriever`)是`LangChain`实现这一功能的核心组件
+ 定义元数据结构：向llm清晰的描述文档内容和每个元数据字段的含义及类型
+ 查询解析： 当用户输入一个自然语言查询时，自查询检索器会调用llm，将查询分为两部分
    + 查询字符串(Query String)： 用于进行语义搜索的部分
    + 元数据过滤(Metadata Filter)：从查询中提取出结构化的过滤条件
+ 执行查询：同时执行语义搜索和元数据过滤
## 检索进阶技术

### 重排序
+ RRF。 如果只考虑排名信息，会忽略原始的相似度分数，可能会丢失部分有用信息
+ RankLLM/LLM-based Reranker。 代表一类直接利用LLM本身进行重排的方法(让LLM判断哪些内容最相关)
    + 通过精心的提示词实现。提示词包含用户的查询和一系列候选文档(通常是文档的摘要和关键部分)，然后要求LLM以特定格式比如JSON，输出一个排序后的文档列表，并给出每个文档的相关性分数
    + [文本排序模型-阿里云](https://help.aliyun.com/zh/model-studio/text-rerank-api)
+ Cross-Encoder重排。交叉编码器能提供出色的重排精度。工作原理是将查询(Query)和每个候选文档拼接成一个单一的输入，然后将这个整体输入到一个预训练的Transformer模型，最后输出一个单一的分数，代表query和文档的相关性
+ ColBERT重排。在cross-encoder的高精度和双编码器的高效率之间取得平衡。采用了后期交互机制

重排方法对比

| 特性 | RRF | RankLLM | Cross-Encoder | ColBERT |
|------|-----|---------|---------------|---------|
| 核心机制 | 融合多个排名 | LLM 推理，生成排序列表 | 联合编码查询与文档，计算单一相关分 | 独立编码，后期交互 |
| 计算成本 | 低（简单数学计算） | 中 (API 费用与延迟) | 高（N次模型推理） | 中（向量点积计算） |
| 交互粒度 | 无（仅排名） | 概念/语义级 | 句子级（Query-Doc Pair） | Token 级 |
| 适用场景 | 多路召回结果融合 | 高价值语义理解场景 | Top-K 精排 | Top-K 重排 |

# RAG评估
+ 对于开发者。如何量化追踪、迭代并提升RAG应用的性能？当系统出现幻觉或答非所问时，如何快速定位问题根源

## 评估三元组


| 维度名称 | 评估对象 | 核心评判问题 | 核心作用/重要性说明 |
| ---- | ---- | ---- | ---- |
| 上下文相关性<br>(Context Relevance) | 检索器 Retriever | 检索到的上下文内容，是否与用户的查询（Query）高度相关？ | 检索是RAG流程在响应用户查询时的第一步。如果检索回来的上下文充满了噪声或无关信息，那么无论后续的生成模型多么强大，都没法做出正确答案。 |
| 忠实度 / 可信度<br>(Faithfulness / Groundedness) | 生成器的可靠性 | 生成的答案是否完全基于所提供的上下文信息？ | 这个维度主要在于量化LLM的“幻觉”程度。一个高忠实度的回答意味着模型严格遵守了上下文，没有捏造或歪曲事实。如果忠实度得分低，说明LLM在回答时“自由发挥”过度，引入了外部知识或不实信息。 |
| 答案相关性<br>(Answer Relevance) | 系统的端到端（End-to-End）表现 | 最终生成的答案是否直接、完整且有效地回答了用户的原始问题？ | 这是用户最直观的感受。一个答案可能完全基于上下文（高忠实度），但如果它答非所问，或者只回答了问题的一部分，那么这个答案的相关性就很低。例如，当用户问“法国在哪里，首都是哪里？”，如果答案只是“法国在西欧”，那么虽然忠实度高，但答案相关性很低。 |


## 检索评估
检索评估聚焦于三元组的上下文相关性，本质上是一次白盒测试。
此阶段的评估需要一个标注数据集，其中包含一系列查询以及每个查询对应的真实相关文档
这项评估借鉴了信息检索领域的多个经典指标
| 指标 | 定义 | 公式 | 作用 |
| ---- | ---- | ---- | ---- |
| 上下文精确率(Context Precision) | 衡量检索结果的准确性。在前`k`个检索结果中，相关文档所占的比例。`k`是预设的评估范围，例如`k=3`或`k=5` | `Precision@k = 前k个结果中的相关文档数 / k` | 高精确率意味着检索结果噪声更少，返回结果更准。 |
| 上下文召回率(Context Recall) | 衡量检索结果的完整性。在前`k`个检索结果中，找到的相关文档占所有真实相关文档总数的比例 | `Recall@k = 前k个结果中的相关文档数 / 数据集中所有相关文档总数` | 高召回率意味着系统能够找回更多关键信息。 |
| F1分数(F1-Score) | 精确率和召回率的调和平均数，用于平衡这两个指标 | `F1 = 2 * Precision * Recall / (Precision + Recall)` | 当精确率和召回率都较高时，F1分数也会较高。 |
| 平均倒数排名(MRR, Mean Reciprocal Rank) | 评估系统将第一个相关文档排在靠前位置的能力。对于单个查询，倒数排名等于第一个相关文档排名的倒数；整体MRR是所有查询倒数排名的平均值 | `MRR =(1 / \|Q\|) * Σ(1 / rank_q)` | 适合用户通常只关心第一个正确答案的场景。`\|Q\|`表示查询总数，`rank_q`表示第`q`个查询中第一个相关文档的排名。 |
| 平均准确率均值(MAP, Mean Average Precision) | 综合评估检索结果的精确率和相关文档排序质量。先计算每个查询的平均精确率`AP(q)`，再对所有查询取平均 | `MAP = (1 / \|Q\|) * Σ AP(q)` | 同时考察是否检索到相关文档，以及它们在结果列表中的排序位置。 |


计算上述指标，前提是拥有一个高质量的标注数据集，包括查询和每个查询对应的真实相关文档

## 生成评估/响应评估
响应评估覆盖RAG三元组中的忠诚度和答案相关性
此环节通常采用端到端的评估范式，因为它直接衡量用户感知的最终输出质量。

### 核心评估维度
+ 忠诚度/可信度。衡量生成的答案在多大程度上可以由给定的上下文所证实。
+ 答案相关性。 衡量生成的答案与用户原始查询的对齐程度。一个高相关性的答案必须是直接的、切题的，并且不包含与问题无关的冗余信息

### 主要评估方法
基于LLM的评估。
能够提供更深度的语义评估，正逐步成为主流选择。利用一个高性能、中立的llm作为评估者，对上述维度进行深度的语义理解和打分

+ 忠诚度评估。首先将生成的答案分解成一系列独立的声明或断言。对每一个断言，在提供的上下文中进行验证，判断其真伪。最终的忠诚度分数是所有被上下文证实的断言所占的比例
+ 答案相关性评估。评估者需要同时分析用户查询和生成的答案。评分时惩罚那些答非所问、信息不完整的包含过多无关细节的答案

基于词汇重叠的经典指标
该指标需要在数据集中包含一个或多个标准答案，通过计算生成答案与标准答案之间n-gram(连续的n个词)的重叠程度来评估质量
+ ROUGE
全称：Recall-Oriented Understudy for Gisting Evaluation
1. 核心侧重点：以召回率为核心，衡量标准答案中的词汇、语义信息被生成文本覆盖的程度，主打评估生成内容**完整性**。
2. 常用变体
    - ROUGE-N：基于n元词组匹配计算；
    - ROUGE-L：基于最长公共子序列匹配计算。
3. 计算公式
$$ROUGE-N = \frac{匹配的\ n-gram\ 数量}{参考答案中\ n-gram\ 的总数}$$

+ BLEU
全称：Bilingual Evaluation Understudy
1. 核心侧重点：以精确率为核心，统计生成文本内有效、存在于标准答案中的词汇占比；内置长度惩罚BP，约束模型输出过短文本，侧重评估生成答案**流畅度与精准度**。
2. 计算公式
$$BLEU = BP \times \exp\left(\sum_{n=1}^{N}w_n\log p_n\right)$$
    - $BP$：长度惩罚因子；
    - $p_n$：修正后的n-gram精确率；
    - $w_n$：各阶n-gram权重。

+ METEOR
全称：Metric for Evaluation of Translation with Explicit ORdering
1. 定位：BLEU优化改进指标，同时融合精确率、召回率调和平均；支持词干还原、同义词匹配，语义贴合度更高，评估结果和人工打分相关性更强。
2. 分段计算公式
$$F_{mean} = \frac{P \times R}{\alpha P + (1-\alpha)R}$$
$$METEOR = F_{mean} \times (1-Penalty)$$
    - $P$：精确率，$R$：召回率；
    - $\alpha$：权重系数；
    - $Penalty$：语序错乱惩罚项。



基于LLM的评估更注重语义和逻辑，评估质量高，但成本也更高且存在评估者偏见。
基于词汇重叠的指标客观、计算快、成本低，但无法理解语义，可能误判同义词或释义。
在实践中，可以将两者结合，使用经典指标进行快速、大规模的初步筛选，再利用LLM进行更精细的评估

# RAG范式
RAG的演进过程
+ Naive RAG。 用户提问 -> 向量检索 -> 增强prompt -> 答案生成
+ Advanced RAG。 query改写 -> 向量检索 -> rerank -> 增强prompt -> 答案生成
+ Modular RAG。 把RAG的各个环节拆成可以独立替换的模块

高级范式
+ Self-RAG。 基于特定数据集训练llm，训练后的llm会自主决定要不要检索
+ CRAG。 检索到的内容质量高，正常走RAG流程；检索到的内容质量低，自动切换到网络搜索
+ GraphRAG。 知识图谱增强全局理解，
+ Agentic RAG。 llm自行决定要不要检索，检索几次。不需要重新训练模型
