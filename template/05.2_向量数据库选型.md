# 05.2 向量数据库选型

## 背景

客服 Agent 的 RAG 工具化改造需要引入向量检索能力，支持以下检索策略：
- 关键词检索（BM25）
- 语义向量检索（dense vector）
- 混合检索（两者融合排序，支持 RRF 和加权融合两种策略）

Embedding 模型已选定：**text-embedding-v4**（阿里云 DashScope API）。

---

## 关键词检索方案对比：BM25 vs 稀疏向量

### 背景

混合检索需要同时支持**关键词精确匹配**和**语义相似检索**。关键词部分有两种主流实现方式：
1. **BM25**：基于统计的关键词排序算法（TF-IDF 的改进版）
2. **稀疏向量（Sparse Vector）**：用深度学习模型（如 BGE-M3、SPLADE）将文本编码成高维稀疏向量，再通过向量相似度召回

---

### BM25

**原理**：
- 基于词频（TF）和逆文档频率（IDF）计算相关性分数
- 引入文档长度归一化，避免长文档优势
- 不依赖模型，纯统计方法

**优点**：
- 零训练成本，开箱即用
- 可解释性强（每个分数对应具体词匹配）
- 对专业术语、产品名、订单号等**精确关键词命中**效果极佳
- PostgreSQL / Elasticsearch / Qdrant 均原生支持

**缺点**：
- 无法理解语义（"手机"匹配不了"移动电话"）
- 依赖分词质量，对错别字、口语化表达鲁棒性差

---

### 稀疏向量（Sparse Vector）

**原理**：
- 用深度学习模型（如 BGE-M3、SPLADE）将文本编码成高维稀疏向量
- 向量中非零维度对应"隐含关键词"，模型可以自动扩展到同义词、相关词
- 既保留关键词匹配能力，又具备一定语义理解

**优点**：
- 相比 BM25 有语义泛化能力（能匹配同义词）
- 和 dense vector 同属向量空间，融合更自然
- Qdrant / Milvus 等专用向量库原生支持

**缺点**：
- 需要额外的 embedding 模型（推理成本和延迟）
- 对**精确关键词命中**（如订单号、产品型号）不一定比 BM25 强
- 可解释性弱于 BM25

---

### 对比总表

| 维度 | BM25 | 稀疏向量 |
| --- | --- | --- |
| 实现方式 | 统计算法（无模型） | 深度学习模型编码 |
| 精确关键词命中 | **强** | 中 |
| 语义泛化能力 | 无 | **有** |
| 推理成本 | 零 | 有（需要 embedding） |
| 可解释性 | **强** | 弱 |
| 错别字/口语化鲁棒性 | 弱 | 中 |
| 原生支持（PG FTS） | **有** | 无 |
| 适合场景 | 专业术语、产品名、订单号 | 通用文本、需要语义泛化 |

---

### 为何选择 BM25

**MVP 阶段选择 BM25，理由如下**：

1. **技术栈对齐**：PostgreSQL 原生全文检索（FTS）基于 BM25，不需要引入额外模型或服务
2. **零推理成本**：BM25 是纯统计计算，不涉及 embedding 推理，延迟低、无 API 调用成本
3. **客服场景适配**：客服 FAQ、规则、售后条款等内容的检索，用户问题通常包含**明确关键词**（如"退货"、"物流"、"订单号"），BM25 命中效果好
4. **可解释性强**：BM25 分数可以拆解到具体词匹配，便于调试和 Bad Case 分析
5. **后续可升级**：若后续发现 BM25 语义泛化能力不足，可以再引入稀疏向量作为补充，BM25 仍保留作为精确匹配层

**结论**：MVP 阶段用 BM25 实现关键词检索，后续若有语义泛化需求再评估稀疏向量。

## 候选方案对比

### 方案 A：PostgreSQL + pgvector

**架构**：关系型数据库 + 向量扩展插件

**混合检索实现方式**：
- 关键词部分：PostgreSQL 原生全文检索（FTS，`to_tsvector` / `plainto_tsquery`）
- 语义部分：`pgvector` 扩展（`<=>` 余弦距离算子）
- 融合：应用层分别召回后，用 RRF 或加权分数在 Python 层合并

**优点**：
- 一套库同时承载业务数据、会话日志、JSON 状态、全文检索、向量检索，事务一致
- 如果已有 PostgreSQL 实例，零新增运维成本
- pgvector 成熟稳定，LangChain / LlamaIndex 均有集成
- MVP 阶段组件数量最少，符合"先跑通流程"的原则

**缺点**：
- 混合检索需要在应用层自己拼装，不是原生的单一 API
- 大规模向量（亿级）性能不如专用向量库
- BM25 分数和向量距离的量纲不同，融合时需要仔细调权重或用 RRF

**适用场景**：MVP 阶段、已有 PG 技术栈、数据规模中等、希望低运维成本

---

### 方案 B：Qdrant

**架构**：Rust 编写的专用向量数据库，支持 Docker / 云服务部署

**混合检索实现方式**：
- 原生支持 `dense + sparse` 混合查询（Sparse Vector 用 BM25 或 fastembed 生成）
- 内置 `fusion` 策略（`rrf` 或 `dls_fusion`），可在单次查询中完成融合
- API 简洁，Python SDK 完善

**优点**：
- 原生混合检索，不需要应用层拼装，开发效率高
- Rust 实现，性能强，资源占用相对低
- Docker 一行启动，运维简单
- 支持多阶段查询（prefetch + main query），适合复杂 RAG 场景
- 社区活跃，LangChain / LlamaIndex 均有集成

**缺点**：
- 需要独立部署（或引入云服务），增加运维组件
- 和 PG 的业务数据不在同一库，跨库查询需要额外设计

**适用场景**：专门做向量检索层、需要原生混合检索、中等至大规模数据

---

### 方案 C：Milvus / Milvus Lite

**架构**：分布式专用向量数据库，Milvus Lite 可嵌入 Python 进程

**混合检索实现方式**：
- 原生支持 `dense + sparse` 混合搜索（`Hybrid Search` API）
- 支持 RRF 和加权融合
- `template/rag/04_multi_milvus.py` 有参考实现

**优点**：
- 原生混合检索，功能最完整
- Milvus Lite 可零部署嵌入进程，适合 MVP
- 大规模集群部署能力强，后续扩展无忧
- 参考实现现成

**缺点**：
- Milvus Server 版运维复杂度较高
- Milvus Lite 不适合生产级并发
- 社区活跃度略低于 Qdrant

**适用场景**：大规模向量、需要 GPU 加速、后续有多模态扩展计划

---

### 方案 D：Elasticsearch

**优点**：成熟的全文检索 + 向量检索一体化方案

**缺点**：
- 当前目标是构建客服 Agent，不是先建设完整搜索平台
- `PostgreSQL FTS + pgvector` 已能满足第一阶段混合检索需求
- 一开始引入 Elasticsearch 会增加索引同步和运维复杂度

**结论**：MVP 阶段不选，后续若有复杂搜索需求可再评估。

---

## 对比总表

| 维度 | PostgreSQL + pgvector | Qdrant | Milvus (Lite) | Elasticsearch |
| --- | --- | --- | --- | --- |
| 混合检索 | 应用层拼装 | **原生 API** | **原生 API** | 原生 API |
| 部署复杂度 | 最低（已有 PG 则零新增） | 低（Docker 一行） | Lite 零部署 / Server 较高 | 高（需要 JVM / 调优） |
| 运维成本 | 低 | 中 | Lite 低 / Server 高 | 高 |
| 大规模性能 | 中 | 高 | 最高 | 高 |
| 与业务数据同库 | 是 | 否 | 否 | 否（但可存业务数据） |
| 参考实现现成 | 无 | 无 | 有（`template/rag/`） | 无 |
| MVP 适合度 | **最高** | 高 | 中 | 低 |

---

## 决策结论

**MVP 阶段选定：MySQL + Qdrant**

理由：
1. **方便后续扩展**：Qdrant 是专用向量数据库，原生支持混合检索（dense + sparse + RRF），后续扩展能力强
2. **开发效率高**：Qdrant 内置 fusion 策略，混合检索一步完成，不需要应用层拼装
3. **部署简单**：Qdrant Docker 一行启动，运维成本低
4. **性能强**：Rust 实现，资源占用低，适合中小规模向量检索
5. **BM25 由 Qdrant 原生支持**：不需要额外引入全文检索组件

**关键词检索选择：BM25**
- MVP 阶段用 BM25（Qdrant 原生支持），零推理成本，对客服场景的关键词命中效果好
- 后续若有语义泛化需求，可再引入稀疏向量作为补充

**后续演进**：
- 第二阶段：优化混合检索、rerank、意图识别
- 第三阶段：若数据规模增长，Qdrant 可平滑升级到集群模式
