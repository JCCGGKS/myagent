# 01 响应时间调优

客服 Agent 单次对话响应时间从约 20s 优化到约 5~8s，并消除冗余 LLM 调用。

## 问题现象

- 单次 `/chat` 响应约 20s，远超可接受范围。
- 主链路每轮对话多次调用 LLM，时延累加明显。

## 根因

1. **思维链（thinking）未关闭**：本地使用 `qwen3.7-plus`（Qwen3 推理模型），主路径 LLM 调用默认开启 `enable_thinking`，单次推理即可耗时 10~20s。此前仅意图兜底（`llm_fallback.py`）硬编码关闭，主路径 `agent_node` / `response_generator` / `clarification` 均未关闭。
2. **冗余第三轮 LLM 调用**：`agent_node` 生成答案后不写 `state.reply`，下游 `response_generator` 再用一次 LLM 重新措辞，属于重复调用。
3. **生成参数不可控**：`temperature` / `max_tokens` / `top_p` 等无法配置，缺乏统一开关。

## 优化措施

### 1. 关闭思维链（配置化，默认 off）
- `app/config/llm.py`：`LLMConfig` 新增 `enable_thinking: bool = False`、`temperature` / `max_tokens` / `top_p`（可选），并提供 `generation_kwargs()` 统一构造 `chat.completions.create` 额外参数（`enable_thinking` 经 `extra_body` 下发）。
- `app/utils/llm.py`：`call_llm` / `call_llm_async` 透传 `generation_kwargs`。
- `agent_node.py` / `response.py` / `clarification.py` / `graph.py`（摘要折叠器）均接入 `llm_config`，主路径统一关闭思维链。
- `llm_fallback.py` 的意图兜底原本硬编码 `extra_body={"enable_thinking": False}`，改为统一走 `config.generation_kwargs()`，**一条开关控制全部路径**。

### 2. 去冗余第三轮 LLM 调用
- `agent_node.py`：`run()` 在 LLM 直接给出结论（无 `tool_calls`）时，把 `content` 写入 `state.reply`。
- `response.py`：已有早返回 `if state.reply: return`，命中后不再调用 LLM。
- 效果：每轮 LLM 调用数从约 3 次降到约 2 次（仅在需要工具 / 超限兜底时才走 `response_generator`）。

### 3. 配置文件同步
- `config/llm_config.local.yml`（生效的 Qwen3 配置）：显式 `enable_thinking: false` + `temperature: 0.3` + `max_tokens: 1024`。
- `config/llm_config.test.yml` / `llm_config.prod.yml` / `llm_config.local.example.yml`：加注释版块，`enable_thinking: false` 显式开启，采样参数留注释占位。

## 涉及文件

- `app/config/llm.py` — `LLMConfig` 新增生成参数 + `generation_kwargs()`
- `app/utils/llm.py` — `call_llm` / `call_llm_async` 透传 `generation_kwargs`
- `app/business/agent/agent_node.py` — 接入 config + 写 `state.reply`（去冗余）
- `app/business/dialog/response.py` — 接入 config（早返回不变）
- `app/business/dialog/clarification.py` — 接入 config
- `app/business/agent/graph.py` — 摘要折叠器接入 `enable_thinking`
- `app/business/intent/llm_fallback.py` — 意图兜底统一走 `config.generation_kwargs()`
- `app/api/chat.py` — 透传 `llm_config`
- `config/llm_config.*.yml` — 同步生成参数

## 验证

- `pytest tests/` 75 passed（含 `test_dialog_services.py` 修复——原测试类 `DialogServicesTestCase` 不匹配 pytest 默认 `Test*` 收集规则，文件从未被执行，已重命名为 `TestDialogServices` 并新增 4 个用例覆盖参数透传、默认关闭 thinking、agent 写 reply、generator 早返回）。
- 实际加载验证：`local → {'extra_body': {'enable_thinking': False}, 'temperature': 0.3, 'max_tokens': 1024}`；`test` / `prod → {'extra_body': {'enable_thinking': False}}`。

## 预期收益

- 主路径思维链关闭：单次对话约 20s → 5~8s。
- 去冗余：每轮少一次 LLM 调用（约 1~2s + 一次网络往返）。
- 配置化：后续可按环境/模型自由调节 `enable_thinking` / 采样参数，无需改代码。
