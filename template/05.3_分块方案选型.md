## 1. 总原则：切块要 format / structure-aware

- **一刀切的递归字符切分只适合同质纯文本**；有结构的文档应先解析出"结构"再沿自然边界切。
- 不同格式先各自**解析**出结构（标题树 / 表格 / 代码块 / 版式坐标），再切块，最后统一吐成 `Chunk`。
- **结构性文档 = 结构划分优先 + 递归字符兜底**：先沿标题 / DOM 边界切出逻辑块（继承 `heading_path`，**块间不重叠**）；若某块超过 `chunk_size` 先按 `\n\n→\n→。→；→，` 分隔符逐级切，**当某段仍超长且分隔符都切不开时，兜底用 `chunk_size` + `chunk_overlap` 的滑动窗口硬切**——目的是让相邻切块在边界处重叠 `chunk_overlap` 字符、**保证语义连贯**。即"结构为主、字符为补"，而非一上来就按字符切。
- **本项目地基已就绪，无需重写检索**：
  - `Chunk`（`app/business/rag/chunker.py`）已带 `heading_path` / `doc_type` / `metadata` 维度 → 天然支持按类型带元数据切块；
  - 检索已是 **dense + bm25(sparse) 双路 + Hybrid(RRF) 融合**（`retrieval_strategy.py`），新增向量模态也只是多加个 named vector；
  - 已有 `DOC_TYPE_CREDIBILITY` 按 `doc_type` 调权（`rerank` 关时生效）、`rag_retrieve` 走 `response` 模板分支渲染 → 表格 / 条款块可走不同渲染；
  - **现状缺口**：上传只收 `.md`/`.json`（`AGENTS.md`），`chunker.py` 只做 Markdown 标题切块 + 递归字符兜底。**最大短板在 parser 层，不在检索层**。

## 2. 场景背景：客服场景常见文档类型（选型输入）

> 选型不是凭空选，先看知识库里到底有哪些文档。客服场景的文档基本分**对外（用户可见的政策/产品/流程）** 和 **对内（话术/培训/工单）** 两大类；结合本项目意图（订单/物流/退款/投诉/转人工），常见类型如下。

### 2.1 常见文档类型矩阵

| 类型 | 典型内容 | 常见格式（解析方式） | 推荐切块 | 对应意图/用途 |
|---|---|---|---|---|
| **政策/规则** | 退换货政策、七天无理由、保修条款、发票规则 | Markdown / Word（标题层级） | **结构切块**（标题边界） | 退款咨询、投诉 |
| **FAQ 问答对** | "怎么开发票？""多久发货？" | JSON / Markdown Q&A | **每对一块**（已是原子单元） | 通用咨询、降澄清 |
| **产品知识** | 商品参数、SKU 规格、尺码表 | Excel/CSV 表、PDF/PPT 产品册 | **行级 + 表头上下文**（列名=值，规格表）+ 表概要块互补 | 订单/售前 |
| **运费/资费表** | 各地区运费、加急费、保价费 | Excel / PDF 表格 | **行级 + 表头上下文** + 表概要块（查"某区多少钱"） | 退款/物流 |
| **订单/物流 SOP** | 发货时效、签收规则、改地址流程 | Markdown / Word | 结构切块 | 订单/物流 |
| **标准话术/应答模板** | 投诉安抚话术、拒赔话术、开场白 | Markdown | 结构切块（按场景分块） | 投诉/转人工 |
| **合同/用户协议** | 服务条款、隐私政策、会员协议 | Word / PDF（长文档） | **条款级 + 条款号元数据** | 投诉/合规 |
| **排查/帮助手册** | 常见问题排查步骤、APP 操作指引 | Markdown（步骤）/ PPT | 结构切块（按问题分块） | 通用自助 |
| **内部培训/工单样本** | 新员工培训、历史会话、工单记录 | PPT / PDF / JSON | 结构切块 / 每样本一块（给 few-shot / eval 用） | 内部、不直接给用户 |

### 2.2 落到本项目的现实缺口

- **现有只收 `.md` / `.json`**（`AGENTS.md`）：能覆盖"政策/FAQ/话术/帮助手册"四类，但**表格（运费/SKU）、合同（PDF）、产品册（PPT）** 全进不来（图片类多模态见 §8 后续演进）——而这几类恰恰是客服高频查表需求。
- **抽象已就绪，只差 parser**：`Chunk` 带 `doc_type` / `heading_path` / `metadata` → 表格加 `chunk_type=table`、条款加 `chunk_type=clause` 即可复用；双路向量（dense + bm25）+ rerank、`DOC_TYPE_CREDIBILITY` 调权 → 新类型进来不改动检索；前端 `OrderDetailCard` / `StatsPanel` 已支持结构化渲染 → 表格 / 条款块直接走卡片。

### 2.3 一句话

> 客服知识库 = **政策/FAQ（结构+Q&A） + 表格（运费/SKU，行级） + 合同（PDF，条款级） + 产品册（PPT，版式） + 话术/手册（结构）**；本项目已能吞前两类，扩 parser 即可补齐后四类（图片类多模态见 §8 后续演进）。

## 3. 分类型推荐切分方式（按文档格式的决策矩阵，非多模态）

> 行 = 客服场景需支持的非多模态文档格式；列 = 该格式如何解析、怎样切块、处于哪个落地阶段。多模态（图片类）见 §8 后续演进。

| 格式 | 解析方式 | 推荐切块 | `chunk_type` | 跨页处理 | 阶段 / 落地 |
|---|---|---|---|---|---|
| **Markdown** | 抽标题树 | 结构切块（标题边界），块间不重叠 | `text` | 无 | 当前（`chunker.py` 已做） |
| **Word** | 抽标题树 / DOM 树 | 结构切块（标题边界） | `text` | 无 | 近期（复用 Markdown 逻辑） |
| **JSON** | 已是原子对（按 key 读） | 每对一块（QA / 记录级） | `text` | 无 | 当前（`ingest_json_records`） |
| **Excel / CSV** | 读单元格（openpyxl / pandas，无损、无跨页） | 行级 + 重建表头上下文（列名=值，单行语义自洽）+ 表概要块互补（完整表头+caption 负责"找表"）；每块带 `table_id`/`caption`/`header`/`row_index`；聚合需求（均价等）另存结构化表走精确匹配 | `table` | 无（一个 sheet = 一个逻辑表，sheet 名当 `table_id`） | 近期（新增 `chunk_type=table`） |
| **PDF** | 版式解析（unstructured / MinerU）+ 表格 / 条款识别；含表格时跨页先缝合 | 含表格 → 行级 + 表头上下文（同 Excel，切分同构）；长文档 → 版式区块 / 条款级（带条款号元数据） | `table` / `clause` | 有：parser 先按"缺表头 / 列一致 / caption 续 / 阅读序紧接"缝合为带 `page_range` 的逻辑 Table，再交给 Chunker 行级切 | 近期（新增 pdf parser → 产出 `Chunk`） |
| **PPT** | 版式解析 + 阅读序 | 版式区块切块（按版面分栏/段落） | `text` / `clause` | 有：版式块跨页靠阅读序缝合（同 PDF 思路） | 近期（新增 ppt parser → 产出 `Chunk`） |

**关键**：所有 parser 最终都吐统一的 `Chunk`（已有 schema），把 `doc_type` / `heading_path` / `page` / `section` 写进 `metadata`。切块策略变了，下游检索 / 重排 / 可信度加权（`DOC_TYPE_CREDIBILITY`）完全不用动——**扩展点只在 parser，不动检索**。

**补两句话（本轮修正）**：
- **overlap**：用途是**保证语义连贯**（`chunk_size` + `chunk_overlap` 滑动窗口），当前仅在 `_hard_split` 兜底层生效，结构块 / 正常标点切分不产生重叠。
- **跨页**：跨页是 **parser 缝合问题，不是 Chunker 问题**——Chunker 只吃已缝合好的逻辑 Table（带 `page_range` / `table_id`），Excel 无跨页、PDF/PPT 必须先在 parser 层缝合。

> 阶段说明：**当前**=已有能力；**近期**=扩 parser 即可补齐；多模态见 §8。

## 4. 切块方式"全家福"（按适用场景选）

| 方式 | 适用 | 本项目现状 |
|---|---|---|
| **定长递归切**（char / token） | 同质纯文本，无结构 | ✅ 现有兜底 |
| **结构切块**（标题 / DOM / AST） | 有层级结构的文档 | ✅ Markdown 已做 |
| **命题切块**（proposition：拆成原子事实） | QA 精度要求高、去噪 | ❌ 可加 |
| **表格感知**（行级 / 语义表块） | 表格密集 | ❌ 需加（近期） |
| **版式切块**（坐标 + 阅读序） | PDF / PPT | ❌ 需加（近期） |
| **LLM / Agentic 切块** | 复杂长文档，让模型定边界 + 写摘要 | ❌ 贵但高质量 |
| **滑动窗口 overlap** | **保证相邻切块边界的语义连贯性**：当文本段超 `chunk_size` 且分隔符切不开时，用 `chunk_size` + `chunk_overlap` 滑动窗口硬切，后块头部重复前块尾部 `chunk_overlap` 字符 | ⚠️ 当前仅在 `_hard_split` 兜底层生效，正常标点切分不产生重叠（见 §1） |

> 注：结构切块 与 定长递归切 在本项目是配合使用（见 §1），"主 + 补"而非二选一；overlap 的用途是**保证语义连贯**（`chunk_size` + `chunk_overlap` 滑动窗口），当前实现仅在 `_hard_split` 兜底层应用——结构块之间不重叠、正常标点切分也不产生重叠。

## 5. 表格切块专项

### 5.1 朴素切块为什么失效

把表格展平成散文再按字符切，会丢掉三样东西：

1. **行列结构** → 召回一个单元格，却拿不到它属于哪一行 / 哪一列；
2. **可过滤性** → "A100 的价格是多少" 这类查表需求直接废掉；
3. **跨页连续性** → 表头在第 1 页、数据在第 3–5 页，被切成互不相关的几段。

→ 表格必须**结构化解析优先，切块在后**。

### 5.2 表格切块四法 + 选择

| 方式 | 做法 | 适用 | 坑 |
|---|---|---|---|
| **整表一块** | 整张表（含表头+caption）作为一个 chunk | 小表（<~30 行）、"总结这张表" | 大表撑爆上下文；检索精度被稀释 |
| **行级切块** ✅ | 每行一块，**重建表头上下文**：`型号=A100, 价格=999, 库存=10` | 查表 / 明细 QA、大表 | 最稳，推荐默认 |
| **单元格级** | 每格一块 | 极少单独用 | 太碎，语义不全 |
| **语义 / 表格模型** | TAPAS / TaCube 之类把表当结构化查 | 需要聚合（"所有行均价"） | 非纯向量可解，需额外引擎 |

**结论**：查表类 RAG 默认**行级 + 表头上下文**；小表可整表；聚合类不走向量、走结构化引擎。

### 5.3 行级切块带什么（易错点）

- 带的是**列名（字段名）**，不是把第一行数据原样复制。每个行级块重建为 `列名=值`（如 `型号=A100｜价格=999｜库存=10`），让它被单独召回时**语义自洽**。
- **为什么必须带**：行级切块的意义就是"一行一块"；但被向量召回时这一行是孤立出现的，不带列名就成了一串无语义的值。
- **代价与缓解**（表头在 M 行里重复 M 次 → token / 存储放大）：
  1. **列名压缩**：用短别名（`型号`/`价`/`库`）而非长字段，正文紧凑；
  2. **表概要块互补**：另存一个块带**完整表头 + caption**（`表格X：列出 N 个型号规格…见 p3–5`）。行块用紧凑列键。检索时——概要块负责"找这张表"，行块负责"找某一行"，两路各管一段，避免全量冗余；
  3. **精确查表另存**：像"所有行均价""A100 库存"这类聚合 / 精确需求，不靠向量，把**结构化表**（列+行）另存一份走精确匹配 / 表格模型；向量只管召回。
- **跨页归属**："表头"里还应包含 `table_id + caption`（跨页拼接后的表标题），这样跨页的行块知道自己归属哪张表，召回多行后能用 `table_id` **归组**，交给表格模型 / 卡片渲染。

> 一句记：**不是"每行都挂一大段表头"，而是带最小可用的列名上下文（紧凑、可压缩）+ `table_id`/`caption` 归属信息——在自洽可检索与冗余成本之间取平衡。**

### 5.4 跨页表格：关键是 parser 先缝合

跨页本身不是切块问题，是**解析阶段没把碎片拼回一张逻辑表**。顺序必须是：

```
PDF/PPT 原始
  → 版式解析（unstructured / MinerU / PP-Structure）抽表格块 + 阅读序
  → 跨页连续性检测 → 拼成 1 个逻辑 Table 对象（表头/列/行/caption/页码跨度）
  → 再切块（行级 + 表头）
```

**连续性检测信号**（判断"这页的表是上一页的续表，不是新表"）：

- 本页表格**缺表头 / 表头与上页重复** → 续表；
- caption 带 `（续）` / `Table 3 (continued)`；
- **列 schema 一致**（列数、列名相同）；
- 阅读序上紧接上页表格块。

工具侧：`MinerU` 和 `unstructured` 的 TablePartitioner 对跨页表处理较好；`PP-Structure`（PaddleOCR 版式+表格识别）适合扫描件；`pdfplumber` / `camelot` 拿阅读序与坐标。

**缝合后切块**示例——同一张跨页表（p3–5）产出的 chunk：

- 1 个**表概要块**：`表格X：列出 N 个型号规格（价格/库存/交期），见 p3–5`；
- N 个**行级块**，每块都带完整表头：`[表X p3–5] 型号=A100｜价格=999｜库存=10`。

### 5.5 Excel vs PDF/PPT 表格（提取异构，切分同构）

| | Excel（xlsx/csv） | PDF / PPT |
|---|---|---|
| 表格本质 | **原生结构化数据**（格子模型） | **渲染出来的图/版式**（像素或坐标） |
| 提取方式 | 读单元格（`openpyxl`/`pandas`），**无损** | 版式识别 + 表格识别（`PP-Structure`/`MinerU`/`camelot`），**有损/歧义** |
| 跨页问题 | **无**（一个 sheet 就是一个逻辑表，多 sheet 也显式） | **有**（表头 p3、数据 p4–5 被切开） |
| 难点在哪 | 只在**粒度 + 元数据**（表名/表头行/类型） | 在**解析 + 跨页缝合** |
| parser 类型 | 单元格读取器 | 版式 / OCR + 表格重建器 |

- **Excel 特有**：多 sheet → 每个 sheet 一个 `table_id`（用 **sheet 名**当表标识，天然去重）；表头行是**显式**的（通常第 1 行），不用猜；**公式 / 类型**要处理（公式求值 vs 保留公式串；日期/数字类型影响"所有行均价"聚合）；跨 sheet 关联（VLOOKUP/键）通常**忽略**，各表独立切。
- **PDF/PPT 特有**：扫描件=图片表格 → 必须先 OCR + 表格识别；跨页续表信号（缺表头/列一致/caption 续）；有些根本不是"真表格对象"，只是文字摆成网格 → 靠版式判定归为表。

## 6. 落到本项目（最小演进路径）

### 6.1 分阶段演进（非多模态）

1. **扩 parser 层**：上传入口放开 `pdf/docx/xlsx/html`，各自解析成统一 `Chunk`。表格类两个 parser 同构产出 `Chunk`：`ExcelParser`（读 sheet → 每 sheet 一个 `table_id`（sheet 名）→ 行级块 + 表概要块）；`PdfTableParser`（版式识别 → 跨页缝合 → 行级块，带 `page_range`）。
2. **扩 Chunk 维度**：`Chunk` 加 `chunk_type`（`text`/`table`/`clause`），其余 `heading_path`/`doc_type`/`metadata` 沿用。
3. **稳检索**：dense + bm25 双路不变，表格 / 条款块复用现有检索与 `DOC_TYPE_CREDIBILITY` 调权。
4. **检索后处理**：`table` / `clause` chunk 在 `response` 阶段走不同渲染（前端卡片已支持结构化）。
5. **重排复用**：DashScope rerank 对 `table` / `clause` 同样适用。

> 多模态（image embedding / 图片块渲染）属后续演进，见 §8。

### 6.2 演进方向（多模态见 §8）

> 多模态（图片类文档的切块与检索）本次不考虑，统一收归 §8 后续演进，不影响上面非多模态方案的落地。

## 7. 一句话决策树

```
有结构(标题/代码/AST)?     → 结构切块(沿自然边界)
  ├ 是表格?              → 行级 + 表头上下文 + 表概要块
  │    ├ 跨页?          → parser 先按"缺表头/列一致/caption续"拼回一张逻辑表,再切
  │    └ Excel?         → 读单元格(无损),sheet名当 table_id;PDF/PPT→版式识别(有损)
  ├ 是图片(多模态)?    → [后续演进·§8] OCR + caption + 统一多模态向量(BGE-M3)
  └ 纯文本/日志?        → 递归字符切块
未知/复杂长文档?         → LLM/Agentic 切块(贵但高质量)
```

> 核心：切块策略（行级那一层）跨类型**可复用**；差异全在**上游 parser 如何把文档变成结构化 `Chunk`**。多模态（图片类）见 §8 后续演进，是这一体系的**演进延伸**（加一路向量），而非另起炉灶。

## 8. 后续演进：多模态（本次不考虑）

> 本章为 deferred 内容：上面 §1–§7 的切块方案只覆盖非多模态文档（Markdown / Word / JSON / Excel / CSV / PDF / PPT）。图片类文档的切块与检索演进单独放在此处，不影响上面的方案。

### 8.1 图片类（多模态）切块
- **图块 + OCR 文本 + caption**，统一多模态向量空间；chunk 含 `image_path + ocr_text + caption`，并加 `chunk_type=image`。

### 8.2 演进方向：图文同向量空间
多模态不是与"表格切块"平级的当前方法，而是本项目的**下一步演进方向**——当知识库要纳入图片类文档（商品图、尺码对照图、流程示意图、发票样张、APP 截图）时才需要。
- **两种架构**：
  - **统一多模态向量空间**（CLIP / BGE-M3 / multilingual-e5-multimodal）：图文同空间，文本 query 能直接召回图片。最省心，融合简单。
  - **模态分治 + 晚融合**：文本→dense、图→CLIP、表→table-embedding，查询时各空间分别召回再 rerank 融合。可控但工程重。
  - → 本项目双路（dense + bm25）已是"分治"雏形，**加 image embedding 作为第三个 named vector 即可，Hybrid 融合逻辑复用**——这是把多模态接进现有检索的最小代价路径。
- **具体落地设想**：
  - **图片**：OCR（PaddleOCR/Azure）抽文本 → 与 image embedding（CLIP/BGE-M3）一起存；检索时文本 query 命中 OCR 文本和/或 image embedding。
  - **表格图片**：先用表格识别（TableMaster/PP-Structure）转结构化，再走表格切块。
  - **统一向量空间**：BGE-M3 这类支持 text+image 同空间，简化融合。

> 定位：多模态依赖 §6.1 第 2 步（`chunk_type=image`）与扩向量（image embedding named vector）落地，属于"演进"阶段而非当前可用能力。
