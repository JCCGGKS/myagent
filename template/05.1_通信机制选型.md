# 客服 Agent MVP 通信机制选型

## 1. 选型目标

为客服 Agent MVP 选定前端与后端的通信机制，需同时满足：

1. **流式输出**：Agent 执行过程中向前端推送中间状态（意图识别结果、工具调用、最终回复）
2. **会话持续性**：多轮对话上下文不丢失，刷新页面可恢复
3. **实现简洁**：MVP 阶段不过度设计，控制组件数量和复杂度
4. **部署友好**：能穿透企业反向代理，不依赖特殊网络配置

---

## 2. 场景约束（选型的出发点）

### 2.1 当前 Agent 的执行特征

```text
用户发消息
  → 意图路由（~50ms）
  → 状态更新（~10ms）
  → 策略判断（~5ms）
  → 工具调用 / 知识检索（~200-500ms）
  → 生成回复（~500-1000ms）
  → 返回
```

**关键观察**：
- 每轮是**独立的请求-响应**，不是长时双向交错通信
- 用户"打断"的窗口期只有 LLM 生成回复的 ~500ms，很短
- 没有服务端主动推送需求（转人工由用户触发，非后台主动通知）

### 2.2 打断需求的真实分析

**看起来需要 WebSocket（全双工）的场景**：

> 用户在 Agent 生成回复过程中，突然发一条新消息（如"算了直接转人工"），需要立即中断当前回复、处理新消息。

**实际上 SSE 也能满足**：

| 方案 | 打断实现方式 | 复杂度 |
|------|-------------|--------|
| WebSocket | 同一连接内服务端检测到新消息，`task.cancel()` 中断当前执行 | 高：需管理连接生命周期、心跳、重连 |
| SSE + 独立 HTTP | 前端 `AbortController.abort()` 取消当前 SSE，立即发新 HTTP 请求 | 低：纯 HTTP，前端逻辑简单 |

**结论**：MVP 阶段不需要为了"打断"引入 WebSocket。

---

## 3. 方案对比

### 3.1 三种方案

| 维度 | 纯 HTTP `POST /chat` | SSE `POST /chat/stream` | WebSocket `WS /ws/chat` |
|------|------------------------|--------------------------|---------------------------|
| 流式输出 | ❌ 等完整回复才能返回 | ✅ 原生支持，`text/event-stream` | ✅ 支持，但需手动分包 |
| 打断 | ❌ | ✅ `AbortController` | ✅ 同一连接内直接收新消息 |
| 部署穿透性 | 最好，纯 HTTP | 较好，HTTP 兼容 | 较差，部分代理/负载均衡器默认不转发 `Upgrade` |
| 前端对接 | 最简单，`fetch` 即可 | 中等，需处理 `ReadableStream` | 最复杂，需处理重连、心跳 |
| 服务端状态 | 无连接概念，天然无状态 | 无连接概念，天然无状态 | 需管理连接池、会话-连接映射 |
| 目前主流产品做法 | — | ✅ 豆包、Kimi 均采用 SSE | ✅ ChatGPT（需双向语音场景） |

### 3.2 与当前项目特征的匹配度

| 场景特征 | 最适合的方案 | 原因 |
|---------|-------------|------|
| 每轮独立请求-响应 | HTTP 或 SSE | WebSocket 的长连接优势发挥不出来 |
| 需要中间状态推送（意图、工具调用） | SSE 或 WebSocket | 纯 HTTP 无法满足 |
| 企业内网部署（可能有 Nginx） | HTTP 或 SSE | WebSocket 需额外配置 `proxy_http_version 1.1; proxy_set_header Upgrade ...` |
| MVP 快速落地 | SSE | 实现简单，与现有 `chat_events()` 事件结构完全兼容 |

## 4. 选型结论

> **MVP 阶段采用 `HTTP + SSE`，不引入 WebSocket。**

接口形态：

```text
POST /chat                # 同步，用于测试、脚本、非 Web 客户端
POST /chat/init          # 初始化会话，返回 session_id（前端"新对话"按钮调用）
POST /chat/stream        # SSE，事件流，前端首选（唯一流式接口）
```

WebSocket **不实现**。若后续有如下需求再引入：
- 服务端需要主动推送与当前用户请求无关的消息（如：订单状态变更通知、管理员介入提示）
- 需要真正的双向低延迟通信（如：语音对话、实时协同）

---

## 5. 为什么不选其他方案

### 5.1 为什么不完全依赖纯 HTTP（不用 SSE）

纯 HTTP `POST /chat` 必须等 Agent 完整处理完才能返回，用户等待期间前端无任何进度反馈，体验较差。

客服场景的消息长度通常较短（几十字到二百字），逐字流式输出对体验提升有限，但**中间状态（意图识别结果、工具调用）的实时推送**对前端很有价值，纯 HTTP 无法支持。

### 5.2 为什么不选 WebSocket 作为 MVP 首选

**1. 实现复杂度高，与 MVP 原则相悖**

WebSocket 需要额外管理：
- 连接建立 / 断开的生命周期
- 心跳保活（防止代理超时断开）
- 重连机制（前端需持久化 `session_id`）
- 连接与会话的绑定（`session_id` → WebSocket 映射）

SSE 是纯 HTTP，这些问题都不存在。

**2. 当前存在两套执行路径（`chat()` vs `chat_events()`），行为可能不一致**

`chat()` 走 LangGraph 引擎，`chat_events()` 手动串联节点、手动推事件。若引入 WebSocket，需要保证三套路径行为一致，维护成本陡增。

**3. 企业部署环境兼容性差**

部分反向代理（如 Nginx 默认配置）不转发 `Upgrade` 头，需要额外配置：

```nginx
location /ws/chat {
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "Upgrade";
}
```

SSE 是普通 HTTP 响应，穿透性更好。

**4. 当前 MVP 没有服务端主动推送场景**

WebSocket 的真正优势是"服务端主动推消息给客户端"，例如：
- 后台管理员操作了订单，实时通知对话中的用户
- 转人工后，客服接入的通知

当前阶段没有这些需求，WebSocket 的优势无法发挥。

## 6. SSE 事件格式设计

复用当前 `chat_events()` 已定义的事件结构，每行一个事件：

```text
event: status
data: {"type":"status","value":"processing"}

event: intent
data: {"type":"intent","main_intent":"order_service","confidence":0.91}

event: state
data: {"type":"state","current_main_intent":"order_service","stage":"executing"}

event: trace
data: {"type":"trace","node":"business_tool_executor","status":"completed","duration_ms":320}

event: tool_result
data: {"type":"tool_result","summary":"订单 A1001 已发货，物流中"}

event: final
data: {"type":"final","reply":"您的订单 A1001 已发货，预计明日送达。"}

event: handoff
data: {"type":"handoff","handoff_reason":"user_request","handoff_summary":"用户要求转人工"}
```

> **注意**：`session_init` 事件不再需要通过 SSE 返回，因为 session_id 已在 `POST /chat/init` 的 HTTP 响应中返回。

前端使用 `EventSource` 或 `fetch + ReadableStream` 解析，与当前消息面板的展示逻辑完全兼容。

## 7. 会话管理与 session_id 设计

### 7.1 session_id 的生成与传递

采用**后端生成 + 前端持久化**方案：

```text
首次进入客服页面：
  前端：从 localStorage 读出 session_id
  → 有 session_id：恢复上次的对话记录，输入框可用
  → 无 session_id：显示"新对话"按钮，用户点击后获取

用户点"新对话"：
  → POST /chat/init（body 里无 session_id）
  → 后端生成 session_id 并返回
  → 前端收到后存入 localStorage
  → 初始化聊天界面（清空消息列表，输入框可用）

刷新页面：
  前端：从 localStorage 读出 session_id
  → 后续请求 body 里带上 session_id
  → 后端从 SessionStore 恢复 state，上下文不丢
```

**为什么不是前端生成 session_id**：

前端生成 uuid 方案要求前端负责持久化，且刷新页面后若 uuid 丢失则后端无法恢复上下文，体验更差。

### 7.2 session_id 的传递方式

SSE 连接是 POST 请求，session_id 放在请求体里：

```json
POST /chat/stream
{
  "session_id": "xxx",     // 已建立会话时传
  "user_id": "user-001",
  "message": "查订单 A1001",
  "channel": "web"
}
```

`session_id` 一定存在（由"新对话"或恢复上次会话时获取），后端不需要处理空 session_id 的边界情况。

## 8. 中断（打断）的实现方案

SSE 方案下，打断通过前端 `AbortController` 实现：

```typescript
// 前端
const controller = new AbortController();

// 发送消息（SSE 流式）
fetch("/chat/stream", {
  method: "POST",
  body: JSON.stringify(payload),
  signal: controller.signal,   // 绑定 AbortController
});

// 用户点"中断"按钮
controller.abort();

// 然后立即发新消息（新请求，新 AbortController）
fetch("/chat/stream", { ... });
```

后端收到新请求时，从 SessionStore 取出上轮的 state 继续处理，上下文不丢。

**不需要 WebSocket**。

## 9. 部署注意事项

### 9.1 Nginx 配置（SSE 穿透）

```nginx
location /chat/stream {
    proxy_pass http://127.0.0.1:8000;
    proxy_set_header Connection "";
    proxy_http_version 1.1;
    chunked_transfer_encoding off;
    proxy_buffering off;      # 关键：关闭代理缓冲，事件实时推送
    proxy_cache off;
}
```

### 9.2 前端的 SSE 消费方式

**推荐 `fetch + ReadableStream`**（而非 `EventSource`），原因：
- 支持 POST（EventSource 只支持 GET）
- 支持 `AbortController`（打断）
- 兼容性好（所有现代浏览器）

## 10. 后续演进

| 阶段 | 通信机制 | 触发条件 |
|------|-------------|----------|
| MVP（当前） | SSE（`/chat/stream`） | — |
| 第二阶段 | SSE 为主，可选增加 WebSocket 推送通道 | 需要服务端主动推送（转人工通知、订单变更） |
| 第三阶段 | 若有多模态输入（语音、图片） | 评估 WebSocket 或 WebRTC |
