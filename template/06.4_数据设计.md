# 客服 Agent MVP 数据设计

## 1. 文档目标

本文档用于将客服 Agent MVP 的核心能力映射到可落地的数据结构，支撑以下需求：

+ 多轮对话状态跟踪
+ 轻量意图识别样本管理
+ FAQ / 规则知识检索
+ 工具调用日志与审计
+ 转人工与问题回放
+ 后续评测与持续优化

MVP 阶段目标不是覆盖全部业务数据，而是定义最小可用、便于扩展的数据模型。

---

## 2. 设计原则

+ 主事实来源以 `MySQL` 为主
+ 向量和全文检索由 `Qdrant` 承担，不放在 MySQL
+ 短期热状态可放 `Redis`，但不替代长期存储
+ 状态对象优先存结构化字段，而不是大段自然语言
+ 工具原始结果、清洗结果、用户可见结果分层存储
+ 意图识别样本、FAQ 知识、规则知识逻辑分表
+ 所有高风险节点保留审计痕迹

> **MVP 实现范围说明**：本文是该数据模型的规划稿。当前 MVP 实际落库的会话持久化通道仅采用 `users` / `sessions` / `messages` 三张表（见 `template/sqls/` 下的 DDL 与 `app/model/` 下的 ORM）。早期设计中的 `conversation_states`（对应 ORM `StateSnapshot`）、`tool_calls`、`handoff_records` 三张审计表**未在本通道采用，已从 ORM 模型与建表脚本中移除**，`SessionStore` 的 `record_tool_call` / `record_handoff` 接口保留但为 no-op。下文 §5.1、§8 的相关章节保留为后续扩展参考，不代表当前已落库结构。

---

## 3. 核心数据域

MVP 阶段建议将数据拆成以下几类：

+ 会话与消息
+ 状态与记忆
+ 意图定义与意图样本
+ 知识库与检索片段
+ 工具调用与审计
+ 转人工与人工接手
+ 评测与问题分析

## 4. 会话与消息设计

### 4.1 `sessions`

用于记录一次客服会话的全局信息。

建议字段：

| 字段 | 类型 | 说明 |
| ---- | ---- | ---- |
| `id` | UUID | 会话主键 |
| `session_no` | VARCHAR | 可读会话编号 |
| `user_id` | INT | 用户标识（取自 `users.id`，不建外键） |
| `channel` | VARCHAR | 渠道，如 web/app/im |
| `status` | VARCHAR | `active / handoff / closed` |
| `current_intent` | VARCHAR | 当前主意图 |
| `current_stage` | VARCHAR | 当前任务阶段 |
| `risk_level` | VARCHAR | 当前风险等级 |
| `summary` | TEXT | 会话摘要窗口中的运行摘要 |
| `handoff_required` | BOOLEAN | 是否需转人工 |
| `created_at` | TIMESTAMP | 创建时间 |
| `updated_at` | TIMESTAMP | 更新时间 |
| `deleted_at` | TIMESTAMP NULL | 软删除时间（NULL 表示未删除，删除仅标记不物理移除） |

说明：

+ `summary` 用于存放摘要窗口中的 `running_summary`
+ `current_intent` 和 `current_stage` 便于快速查询当前会话位置

### 4.2 `messages`

用于记录每一轮用户消息、系统消息和工具消息。

建议字段：

| 字段 | 类型 | 说明 |
| ---- | ---- | ---- |
| `id` | UUID | 主键 |
| `session_id` | UUID | 关联 `sessions.id` |
| `role` | VARCHAR | `user / assistant / system / tool` |
| `message_type` | VARCHAR | `text / summary / tool_result / clarification` |
| `content` | TEXT | 原始消息文本 |
| `sanitized_content` | TEXT NULL | 清洗后的可用内容 |
| `tool_call_id` | UUID NULL | 若由工具产生则关联工具调用 |
| `sequence_no` | BIGINT | 会话内顺序号 |
| `token_count` | INT NULL | 可选，记录 token 数量 |
| `created_at` | TIMESTAMP | 创建时间 |

说明：

+ 近场原始上下文主要来自此表
+ 摘要消息也可作为特殊 `message_type=summary` 保存

## 5. 状态与记忆设计

### 5.1 `conversation_states`（设计参考，MVP 未落库）

> 该表对应早期 ORM 类 `StateSnapshot`，已在会话持久化通道中移除。当前 `ConversationState` 仅在内存（`SessionStore._states`）中维护，不落库。下表保留为后续扩展参考。

用于持久化每个会话在不同轮次的结构化状态快照。

建议字段：

| 字段 | 类型 | 说明 |
| ---- | ---- | ---- |
| `id` | UUID | 主键 |
| `session_id` | UUID | 关联会话 |
| `current_intent` | VARCHAR | 当前主意图 |
| `stage` | VARCHAR | 当前阶段，如 `collecting_info` |
| `slots` | JSONB | 当前已知槽位 |
| `missing_slots` | JSONB | 仍缺失的槽位列表 |
| `confirmed_slots` | JSONB | 已确认槽位列表 |
| `candidate_intents` | JSONB | 候选意图列表 |
| `needs_clarification` | BOOLEAN | 是否需要追问 |
| `topic_changed` | BOOLEAN | 是否切话题 |
| `risk_level` | VARCHAR | 当前风险等级 |
| `state_summary` | TEXT | 当前状态摘要 |
| `is_active` | BOOLEAN | 是否为当前活跃状态 |
| `archived_reason` | VARCHAR NULL | 归档原因，如 `intent_shift_to_refund_service` |
| `parent_state_id` | UUID NULL | 被切换前状态链路的父状态 |
| `created_at` | TIMESTAMP | 创建时间 |

说明：

+ 这张表是“状态对象”的落地形式
+ 主推理链路暴露给模型的信息应优先来自这里，而不是直接读取原始消息或工具结果
+ `is_active=true` 表示当前会话正在使用的状态快照
+ 当 `main_intent` 切换时，旧状态快照应标记为非活跃并写入 `archived_reason`
+ 状态顺序通过 `created_at` 判断，直接前驱关系通过 `parent_state_id` 表达

### 5.2 状态冻结落表方案

MVP 阶段建议采用“单表状态快照 + 活跃标记”的方式处理状态冻结，不单独引入 `archived_task_states` 表。

建议规则：

+ 当前链路状态写入 `conversation_states`，并标记 `is_active=true`
+ 当发生 `main_intent shift` 时，将旧状态快照冻结为 `is_active=false`
+ 同时写入 `archived_reason`
+ 新链路初始化后写入一条新的 `is_active=true` 状态记录
+ 新状态通过 `parent_state_id` 指向被切换前的旧状态
+ 若新链路继承了旧链路中的可复用 slot，可在新状态中直接落地继承后的 `slots`

这种做法可以满足：

+ 审计与问题回放
+ 转人工时查看任务切换轨迹
+ 后续切回旧任务时读取最近一条已冻结状态

### 5.3 `user_memories`

用于记录少量跨会话长期记忆。

建议字段：

| 字段 | 类型 | 说明 |
| ---- | ---- | ---- |
| `id` | UUID | 主键 |
| `user_id` | INT | 用户标识（取自 `users.id`，不建外键） |
| `memory_type` | VARCHAR | `profile / preference / verified_fact` |
| `memory_key` | VARCHAR | 记忆键 |
| `memory_value` | JSONB | 记忆值 |
| `confidence` | NUMERIC(5,4) | 可信度 |
| `source` | VARCHAR | 来源 |
| `created_at` | TIMESTAMP | 创建时间 |
| `updated_at` | TIMESTAMP | 更新时间 |

MVP 说明：

+ 这张表不是第一优先级
+ 仅保存少量稳定、跨会话有价值的信息

## 10. Redis 中建议缓存的数据

Redis 只建议缓存短期热数据，不作为事实主库。

建议缓存：

+ 活跃会话最近状态
+ 最近几轮消息窗口
+ 热门 intent 示例向量或索引结果
+ 热门 FAQ 召回结果
+ 幂等 key
+ 限流计数器

不建议放 Redis 的核心事实：

+ 审计日志
+ 工具原始结果长期存档
+ 长期知识库文档
+ 评测结果主记录

## 11. 数据流与表关系建议

一个典型流程如下：

1. 用户发起消息，写入 `sessions` 和 `messages`
2. 意图识别读取 `intent_examples`、`intent_definitions`
3. 路由结果在内存 `ConversationState` 中维护（MVP 不落 `conversation_states` 表）
4. FAQ/规则检索读取 `knowledge_chunks`
5. 工具调用结果仅用于当轮推理，MVP 不写 `tool_calls` 表（接口保留为 no-op）
6. 若需转人工，转人工信息随会话状态在内存维护，MVP 不写 `handoff_records` 表
7. 失败样本和回归评测进入 `failure_cases`、`evaluation_records`

## 12. MVP 最小建表建议

如果第一阶段只建最小集，建议至少落以下表：

+ `sessions`
+ `messages`
+ `intent_definitions`
+ `intent_examples`
+ `knowledge_documents`
+ `knowledge_chunks`

> 当前 MVP 实际落库的会话持久化通道仅 `users` / `sessions` / `messages`（`conversation_states` / `tool_calls` / `handoff_records` 未采用，详见文首实现范围说明）。

第二阶段再补：

+ `user_memories`
+ `faq_examples`
+ `evaluation_records`
+ `failure_cases`

## 13. 最终建议

MVP 阶段的数据设计应优先服务以下目标：

+ 状态清晰可追踪
+ 检索链路可落地
+ 工具调用可审计
+ 转人工上下文不丢失
+ 后续评测和优化有数据基础

因此建议采用：

+ `MySQL` 作为主存储
+ `Qdrant` 支撑意图样本和知识检索（向量 + BM25 混合检索）
+ `Redis` 仅缓存短期热状态
+ 状态对象、工具结果、用户可见结果分层存储
